## 5.2 실험 A: PoW 체인 무결성

**목적.** PoW 해시 체인이 에이전트 실행 증적의 위변조를 탐지할 수 있는지 검증한다 (RQ1).

**방법.** 74개의 기존 PoW 블록에 대해 (1) 정상 상태 baseline 검증, (2a) `block_hash` 변조, (2b) `prev_hash` 변조, (2c) `nonce` 변조를 순차 수행한 후 `verify_chain`을 호출하고, (3) 원본 복원 후 재검증하였다.

**결과.**

| 시나리오 | 변조 대상 | 탐지 결과 | 탐지 메커니즘 |
|---------|----------|----------|-------------|
| Baseline | 없음 | valid=True | — |
| 2a | block_hash | **탐지** (valid=False) | hash 재계산 불일치 |
| 2b | prev_hash | orphan 분류 | linked-list 체인 분기 |
| 2c | nonce | **탐지** (valid=False) | hash 재계산 불일치 |
| 복원 | 없음 | valid=True | — |

`block_hash` 및 `nonce` 변조는 해시 재계산에 의해 100% 탐지되었다. `prev_hash` 변조는 M27의 linked-list 재구성 알고리즘에 의해 해당 블록이 주 체인에서 분리(orphan)되어 간접적으로 감지된다. 위양성은 0건이었다.

**RQ1 답변.** PoW 해시 체인은 `block_hash`/`nonce`의 부분 변조를 100% 탐지하며, `prev_hash` 변조는 체인 분기(orphan)로 감지한다. 위양성률 0%. 단, DB 관리자가 전체 체인을 재작성하는 경우는 탐지하지 못하며(분산 합의 부재), 이는 6장에서 한계로 논의한다.

## 5.3 실험 H: 시간대 불변성

**목적.** 데이터베이스 세션의 시간대(timezone) 설정이 PoW 해시 검증에 영향을 미치는지 확인한다 (RQ1 보충).

**방법.** 최근 10개 PoW 블록에 대해 5개 시간대(UTC, Asia/Seoul, America/Los_Angeles, Europe/London, Asia/Tokyo)에서 `ts_raw` 기반 해시 재계산을 수행하고 저장된 `block_hash`와 비교하였다.

**결과.**

| 시간대 | 오프셋 | 해시 일치 | 일치율 |
|--------|--------|----------|--------|
| UTC | +0 | 10/10 | 100% |
| Asia/Seoul | +9 | 10/10 | 100% |
| America/Los_Angeles | -7 | 10/10 | 100% |
| Europe/London | 0/+1 | 10/10 | 100% |
| Asia/Tokyo | +9 | 10/10 | 100% |
| **합계** | | **50/50** | **100%** |

M27에서 도입한 `ts_raw` 필드(원본 ISO 8601 UTC 문자열)가 timezone 변환을 우회하여, 어떤 시간대에서도 해시 검증이 일관되게 성공한다.

**RQ1 보충 답변.** PoW 해시 검증은 시간대에 완전히 불변(invariant)하다.

## 5.4 실험 B: RL 수렴 및 탐색

**목적.** Q-learning 정책이 수렴하는지, UCB1 탐색 전략이 미방문 상태-행동 쌍을 효과적으로 탐색하는지 검증한다 (RQ2).

**방법.** 약 100건의 에피소드(task_reward)에 대해 20 에포크 학습을 수행하고, 에포크별 Q-value 변화량(delta)을 측정하였다. 이후 greedy, UCB1, ε-greedy 세 전략의 추천 결과를 비교하였다.

**결과.**

**표 5. Q-learning 수렴 곡선**

| Epoch | Q_mean | Delta | Coverage | Visited/192 |
|-------|--------|-------|----------|-------------|
| 1 | 0.0184 | — | 4.2% | 8 |
| 5 | 0.0243 | 0.0008 | 4.2% | 8 |
| 10 | 0.0272 | 0.0004 | 4.2% | 8 |
| 15 | 0.0288 | 0.0002 | 4.2% | 8 |
| 20 | 0.0298 | 0.0001 | 4.2% | 8 |

Epoch 5에서 delta < 0.001에 도달하여 빠른 수렴을 확인하였다. 다만 커버리지는 4.2%(8/192 셀)에 머물렀는데, 이는 단일 에이전트 환경에서 success_rate와 task_order 버킷의 다양성이 제한되기 때문이다.

**표 6. 탐색 전략 비교**

| 상태 (risk) | Greedy | UCB1 | ε-greedy |
|------------|--------|------|----------|
| low | low (방문 1,374회) | **medium** (미방문) | low |
| medium | medium (방문 412회) | **low** (미방문) | low |
| high | high (방문 204회) | **low** (미방문) | high |

UCB1은 모든 상태에서 미방문 행동을 정확히 추천하여, greedy 전략의 exploitation 편향을 보완한다.

**RQ2 답변.** Q-learning은 5 에포크 내에 수렴하며(delta < 0.001), UCB1은 미방문 행동을 정확히 추천하여 정책 다양성을 확보한다. 커버리지 확대에는 다중 에이전트 환경이 필요하다.

## 5.5 실험 G: Playbook 재현성

**목적.** Playbook 기반 실행이 결정론적 재현을 보장하는지 검증한다 (RQ3).

**방법.** 3개 태스크(hostname, uname -a, df -h)로 구성된 Playbook을 10회 반복 실행하고, 각 실행의 stdout을 baseline(1회차)과 비교하여 의미적 재현율을 측정하였다.

**결과.**

| 실행 | vs Baseline | 재현율 | 실행 결과 |
|------|-----------|--------|---------|
| Run 1 | (baseline) | — | success |
| Run 2~10 | Run 1 | **100.0%** | success (×9) |

10회 전체에서 의미적 완전 일치(100%)를 달성하였다. LLM adhoc 모드(Playbook 없이 매번 LLM이 명령을 생성)에서는 비결정론으로 인해 동일한 재현이 보장되지 않는다.

**RQ3 답변.** Playbook 기반 결정론적 빌더는 LLM의 비결정론을 완전히 제거하여 100% 재현율을 달성한다.

## 5.6 실험 E: 상태 머신 유효성

**목적.** 상태 머신이 무효 전이를 차단하고, replan을 통한 실패 복구가 가능한지 검증한다 (RQ4).

**방법.** (1) 유효 전이 3건(intake→plan, plan→execute, execute→success)과 무효 전이 5건(plan→validate, plan→close, plan→report, execute→plan 직접, execute→close)을 시도하고, (2) 의도적 실패 후 replan→재실행 복구를 검증하였다.

**결과.**

| 전이 유형 | 시도 | 결과 | HTTP 코드 |
|----------|------|------|----------|
| 유효 전이 | 3건 | 3건 성공 (100%) | 200 |
| 무효 전이 | 5건 | 5건 차단 (100%) | 400 |
| Replan 복구 | 1건 | 성공 (failed→plan→execute→success) | 200 |

**RQ4 부분 답변.** 상태 머신은 무효 전이를 100% 차단하며, replan을 통한 실패 복구가 정상 동작한다.

## 5.7 실험 C: 증거 완전성

**목적.** 모든 태스크(성공, 실패, 오류)에 대해 evidence, PoW 블록, reward가 빠짐없이 기록되는지 검증한다 (RQ4).

**방법.** 5개 태스크(성공 3건, 실패 1건, 오류 1건)를 execute-plan으로 실행한 후, evidence, proof_of_work, task_reward 테이블의 레코드 수를 태스크 수와 비교하고, replay API로 타임라인 재구성을 검증하였다.

**결과.**

**표 7. 태스크별 증거 기록**

| # | 태스크 | exit_code | reward | evidence | PoW 블록 |
|---|--------|-----------|--------|----------|---------|
| 1 | success-echo | 0 | +1.3 | ✓ | ✓ |
| 2 | success-hostname | 0 | +1.3 | ✓ | ✓ |
| 3 | success-date | 0 | +1.3 | ✓ | ✓ |
| 4 | fail-exit1 | 1 | -1.1 | ✓ | ✓ |
| 5 | error-badcmd | 127 | -1.0 | ✓ | ✓ |

| 지표 | 결과 |
|------|------|
| Evidence 완전율 | 100% (5/5) |
| PoW 블록 완전율 | 100% (5/5) |
| Task Reward 완전율 | 100% (5/5) |
| 타임라인 재구성 | 성공 (5/5 단계 매핑) |
| 체인 무결성 | valid=True (74블록) |

성공, 실패, 오류 모든 유형에서 evidence, PoW 블록, reward가 100% 기록되었으며, replay API를 통한 타임라인 재구성도 100% 성공하였다.

**RQ4 부분 답변.** OpsClaw는 모든 태스크 유형에 대해 증거를 100% 기록하며, 사후 타임라인 재구성이 완전히 가능하다.

## 5.8 실험 F: 병렬 스케일링

**목적.** 병렬 Multi-Agent Dispatch의 확장성을 측정한다 (RQ4).

**방법.** 동일 태스크(`sleep 2 && echo done`, 각 ~2초)를 N=1, 2, 3, 5개로 순차(sequential) 및 병렬(parallel) 실행하고, wall-clock 시간과 speedup을 비교하였다.

**결과.**

**표 8. 병렬 스케일링 결과**

| N | Sequential (ms) | Parallel (ms) | Speedup | 효율 (%) |
|---|----------------|--------------|---------|---------|
| 1 | 2,173 | 2,236 | 0.97× | 97.0 |
| 2 | 4,359 | 2,407 | 1.81× | 90.5 |
| 3 | 6,504 | 2,444 | 2.66× | 88.7 |
| 5 | 10,823 | 2,709 | **4.00×** | 80.0 |

```
그림 5. 병렬 스케일링 그래프

Speedup
  5 ┤                              ╱ 이론 최대
    │                           ╱
  4 ┤                      ●──╱── 4.00x (N=5)
    │                   ╱
  3 ┤              ●──╱──── 2.66x (N=3)
    │           ╱
  2 ┤      ●──╱────── 1.81x (N=2)
    │    ╱
  1 ┤ ●╱──────── 0.97x (N=1)
    │
  0 ┼──┬──┬──┬──┬──
    0  1  2  3  4  5  N (태스크 수)
```

N=5에서 4.0배 가속을 달성하였다. N=1에서의 미세한 성능 저하(0.97×)는 ThreadPoolExecutor 생성 오버헤드에 기인한다. 효율이 80%인 이유는 단일 SubAgent의 동시 요청 경합과 PoW 블록의 순차적 `prev_hash` 의존 때문이며, 서버별 독립 SubAgent 운용 시 이론치에 근접할 것으로 예상된다.

**RQ4 부분 답변.** 병렬 dispatch는 N=5에서 4.0배 가속을 달성하며, 단일 API 호출로 다수 서버에 동시 작업을 위임할 수 있다.

## 5.9 실험 D: 경험 재활용

**목적.** 4-Layer 메모리의 자동 경험 승급과 RAG 기반 경험 재활용의 효과를 검증한다 (RQ5).

**방법.** (1) 20건의 고보상 태스크에 대한 `auto_promote_high_reward`의 자동 승급을 확인하고, (2) A/B 테스트로 프로젝트 컨텍스트 주입 유무에 따른 LLM 응답 품질을 비교하였다.

**결과.**

**Phase 1: 자동 경험 승급**
- 20건 전량 `[Auto]` 접두어로 자동 승급 (avg_reward ≥ 1.1 조건 충족)
- 카테고리: 전부 "operations" (LLM 분류 미연결 시 기본값)

**Phase 2: A/B 비교**

| 지표 | A (컨텍스트 없음) | B (컨텍스트 주입) | 차이 |
|------|-----------------|-----------------|------|
| 응답 길이 | 1,348자 | 2,040자 | +51% |
| 명령어 구체성 | 3개 (df, du, rm) | 6개 (df, du, rm, find, ncdu, sudo) | +100% |
| RAG 검색 적중 | 0건 | 0건 | — |

프로젝트 컨텍스트 직접 주입 시 응답 길이 51% 증가, 명령어 다양성 100% 증가를 확인하였다. 다만 RAG 검색은 한국어 FTS 미지원으로 인해 적중 0건이었으며, 이는 6장에서 한계점으로 논의한다.

**RQ5 답변.** 자동 경험 승급이 정상 동작하며(20건 전량), 프로젝트 컨텍스트 주입이 LLM 응답 품질을 유의미하게 개선한다. RAG 기반 검색은 한국어 지원 확장이 필요하다.

## 5.10 비교 실험: OpsClaw vs Claude Code 단독

**목적.** 동일 태스크 벤치마크에서 오케스트레이션 하네스(OpsClaw)와 단독 에이전트(Claude Code)의 성능을 다차원으로 비교한다.

**방법.** 5개 다중 서버 운영 점검 태스크를 OpsClaw(execute-plan, parallel=true)와 Claude Code(SSH 직접, 병렬 허용)로 각각 실행하고, 7개 차원에서 비교하였다. **양쪽 모두 병렬 실행을 허용**하여 공정성을 보장하였다.

**결과.**

**표 9. OpsClaw vs Claude Code 공정 비교 (양쪽 병렬)**

| # | 차원 | OpsClaw | Claude Code | 분석 |
|---|------|---------|-------------|------|
| D1 | 실행 시간 (병렬) | 570ms | **414ms** | CC가 27% 빠름 (하네스 오버헤드) |
| D2 | 증적 자동 기록 | **10건** (evidence 5 + PoW 5) | 0건 | 하네스 고유 가치 |
| D3 | 무결성 검증 | **해시 체인 149블록** | 불가 | 부분 변조 탐지 |
| D4 | 재현성 | **1 API call** 재실행 | 히스토리 복사 | 구조적 차이 |
| D5 | 프로젝트 추적 | **replay 타임라인** | 없음 | 감사 추적 |
| D6 | RL 정책 학습 | **UCB1 추천** | 없음 | 자율 개선 |
| D7 | 완료 보고 | **자동 생성** | 없음 | 운영 편의 |

**분석.** 순수 실행 속도에서는 Claude Code 직접 SSH(414ms)가 OpsClaw(570ms)보다 27% 빠르다. OpsClaw의 156ms 오버헤드는 API 처리, evidence 5건 기록, PoW 5블록 생성, reward 5건 산출에 기인한다. 그러나 D2~D7의 6개 차원에서 하네스가 우위를 보이며, 이는 모두 명령 실행 "후"에 발생하는 가치이다. 하네스의 진짜 가치는 속도가 아닌 **증적·추적·학습의 자동화**에 있다.
