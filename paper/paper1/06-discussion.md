# 6. 논의 (Discussion)

## 6.1 주요 발견 요약

본 논문의 실험 결과는 OpsClaw의 5가지 연구 질문에 대해 다음과 같이 답한다.

**RQ1 (PoW 무결성).** SHA-256 PoW 해시 체인은 `block_hash` 및 `nonce` 변조를 100% 탐지하고(실험 A), `prev_hash` 변조는 orphan 분류로 간접 감지한다. M27의 `ts_raw` 기반 해싱은 5개 시간대에서 완전한 불변성을 보장한다(실험 H). 이는 합의 프로토콜 없이도 단일 조직 환경에서 충분한 감사 무결성을 달성할 수 있음을 입증한다.

**RQ2 (RL 수렴).** Q-learning은 5 에포크 만에 delta < 0.001로 수렴하여(실험 B), 소규모 상태 공간에서 빠른 정책 학습이 가능함을 보였다. UCB1 탐색 전략은 미방문 행동을 정확히 추천하여, greedy 전략의 exploitation 편향을 효과적으로 보완한다.

**RQ3 (재현성).** Playbook 기반 결정론적 빌더는 10회 반복에서 100% 재현율을 달성하여(실험 G), LLM adhoc 실행(~30-50%)과 극명한 대비를 보인다. 파라미터 바인딩이 LLM 비결정론 제거의 핵심 메커니즘임을 확인하였다.

**RQ4 (확장성 및 안전성).** 상태 머신이 무효 전이를 100% 차단하고(실험 E), 모든 태스크에 대해 증거가 100% 기록되며(실험 C), 병렬 dispatch가 N=5에서 4.0배 가속을 달성한다(실험 F). 3계층 위임 아키텍처가 안전성과 확장성을 동시에 보장함을 입증하였다.

**RQ5 (경험 재활용).** 자동 경험 승급(20건)이 정상 동작하며, 프로젝트 컨텍스트 주입이 LLM 응답 품질을 51% 개선한다(실험 D). 다만 RAG 기반 검색은 한국어 FTS 미지원으로 제한적이다.

## 6.2 실운영 적용 경험

OpsClaw는 M0(설계 기준선)부터 M25(인프라 구축)까지 25개 마일스톤에 걸쳐 점진적으로 개발되었다. 이 과정에서 얻은 주요 교훈은 다음과 같다.

**점진적 무결성 강화.** 초기(M18)에는 PoW 블록이 타임스탬프 기반 순서에 의존하였으나, 실운영에서 시간대 차이로 인한 해시 불일치가 발견되었다(B-02 버그). M27에서 `ts_raw` 필드 추가와 linked-list 재구성으로 근본적으로 해결하였다. 이는 블록체인 설계에서 타임스탬프에 의존하지 않는 체인 구성의 중요성을 보여준다.

**위임 준수의 구조적 강제.** Mode B에서 외부 AI(Claude Code)가 SubAgent에 직접 접근하는 것을 API 수준에서 차단하는 것만으로는 불충분하며, 위임 위반 감지 메커니즘(delegation audit)이 필수적임을 확인하였다. 실험 과정에서 Claude Code의 직접 SSH 실행을 실시간 감시하는 프로세스를 운영하였다.

**인증 미들웨어의 중요성.** Purple Team 실험에서 OpsClaw API에 인증이 없는 상태에서 원격 코드 실행(RCE)이 가능함이 발견되었고, M28에서 X-API-Key 미들웨어를 긴급 추가하였다. 이는 control-plane 보안이 보호 대상 시스템의 보안만큼 중요함을 보여주는 실운영 교훈이다.

## 6.3 하네스 프리미엄의 실체

비교 실험(5.10절)에서 Claude Code 직접 병렬 SSH(414ms)가 OpsClaw(570ms)보다 27% 빠른 것으로 나타났다. 이는 하네스의 가치가 실행 속도가 아님을 명확히 보여준다. 하네스의 진짜 가치는 **"실행 후" 자동화**에 있다: 증적 10건 자동 기록, 해시 체인 무결성 검증, 보상 산출과 RL 정책 학습, 프로젝트 타임라인 재구성, 완료 보고 자동 생성. 이러한 가치는 1회성 명령 실행에서는 오버헤드지만, 감사가 필요한 환경, 반복 운영, 다중 에이전트 성과 관리에서 정당화된다.

## 6.4 해시 체인의 정직한 평가

PoW 해시 체인은 **부분 변조 탐지**에 효과적이나(실험 A, 100% 탐지), 분산 합의가 없으므로 DB 관리자가 전체 체인을 재작성하면 탐지할 수 없다. 이는 "위변조 방지 블록체인"이 아니라 **"검증 가능한 작업 기록(Verifiable Work Record)"**으로 보는 것이 정확하다. VWR의 실용적 가치는 (1) 부분 변조/실수의 탐지, (2) 에이전트 작업의 구조적 기록, (3) 보상 기반 성과 추적에 있으며, 에이전트 성과 평가, 비용 정산, 컴플라이언스 증빙, SLA 모니터링 등으로 확장 가능하다.

## 6.5 한계점

본 연구의 한계점은 다음과 같다.

**L1: RL 커버리지 4.2%.** Q-table의 48개 상태 중 8개만 학습되었다. 이는 단일 에이전트, 균일한 task_order, 고정된 success_rate 환경에서의 구조적 한계이다. 다중 에이전트 환경에서 다양한 시나리오를 실행하면 커버리지가 자연스럽게 확대될 것으로 예상되나, 본 실험에서는 검증하지 못하였다.

**L2: 단일 SubAgent 병목.** 병렬 실행 시 로컬 SubAgent가 동시 요청을 순차 처리하는 경향이 있어, 이론 대비 80% 효율에 머물렀다(실험 F). 서버별 독립 SubAgent 운용 시 해소될 것으로 예상되나, 본 실험에서는 단일 SubAgent만 사용하였다.

**L3: 한국어 FTS 미지원.** RAG 검색이 영문 기반 PostgreSQL FTS에 의존하여, 한국어 쿼리의 적중률이 0%이다(실험 D). 한국어 환경에서의 실효적 활용을 위해 `pg_trgm` 확장 또는 별도 한국어 형태소 분석기 연동이 필요하다.

**L4: prev_hash 변조의 간접 탐지.** `prev_hash` 변조 시 해당 블록이 tampered가 아닌 orphan으로 분류되어, 명시적 변조 경고가 발생하지 않는다(실험 A). 이는 블록체인 설계상 체인 분기(fork)와 의도적 변조를 구분하기 어려운 본질적 한계이며, orphan 수 임계값 경보로 보완할 수 있다.

**L5: N=1 태스크 오버헤드.** 단일 태스크에서 OpsClaw는 ThreadPoolExecutor 생성 비용으로 인해 직접 실행보다 약간 느리다(0.97×, 실험 F). 오케스트레이션 하네스의 이점은 N ≥ 2에서 발현된다.

**L6: 단일 실행 (N=1).** 본 실험의 대부분은 단일 실행으로 수행되어 통계적 유의성이 제한적이다. 향후 다중 회차 반복과 통계 검정이 필요하다.

## 6.6 향후 연구

**Deep RL로의 확장.** Q-table의 48개 이산 상태는 현재 시나리오에서 충분하나, 에이전트 수와 태스크 유형이 증가하면 상태 공간이 기하급수적으로 확대된다. DQN(Deep Q-Network) 또는 PPO(Proximal Policy Optimization)로의 전환을 통해 연속적 상태 공간과 대규모 행동 공간을 처리할 수 있다.

**다중 SubAgent 분산 아키텍처.** 현재 단일 SubAgent에 집중되는 병목을 해소하기 위해, 서버별 독립 SubAgent에 대한 로드 밸런싱과 장애 격리를 구현할 수 있다. 이는 병렬 효율을 이론치에 근접시킬 것으로 예상된다.

**Multi-LLM 오케스트레이션.** Mode A/B의 이중 모드 설계를 확장하여, 복수의 LLM(Claude, GPT-4, 오픈소스 모델)이 Master 역할을 교대하거나 협업하는 구조를 탐구할 수 있다. Manager와 SubAgent 계층이 Master에 무관하게 설계되어 있어, 인터페이스 변경 없이 Master 교체가 가능하다.

**한국어 trigram FTS.** PostgreSQL의 `pg_trgm` 확장과 한국어 형태소 분석기를 연동하여, RAG 검색의 한국어 적중률을 개선할 수 있다. 이는 비영어권 환경에서의 경험 재활용 효과를 크게 향상시킬 것이다.

**Codex CLI 비교.** OpenAI Codex CLI와의 3자 비교를 통해 하네스 프리미엄의 일반성을 검증할 필요가 있다. 본 연구에서는 환경 미구축으로 인해 2자 비교에 그쳤다.
