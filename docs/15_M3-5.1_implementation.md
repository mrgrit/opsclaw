# OpsClaw M3-5.1 완료 보고서 (LLM Registry + Role Binding + yncai Provider)

작성일: 2026-03-05
범위: **멀티 LLM 연결(Registry) + 역할 바인딩(Role Binding) + 학교 LLM(yncai) Provider 연동 + API 테스트 성공**

---

## 1) 목표(Goal)

M3-5.1의 목표는 다음을 “실제 동작”하게 만드는 것이다.

1. **LLM Connection Registry**를 만들어 다양한 LLM 연결을 등록/조회/삭제할 수 있다.
2. **역할별(Role) LLM 선택**이 가능하다. (Master/Manager/SubAgent 기본값)
3. 학교 제공 LLM 게이트웨이(Provider=`yncai`)를 연결하고 **실제 호출 테스트가 성공**한다.

---

## 2) 구현 결과 요약(What’s Done)

### ✅ A. LLM Connection Registry API

다음 API가 동작한다.

* `GET /llm/connections` : 등록된 LLM 연결 목록 조회
* `GET /llm/connections/{conn_id}` : 단건 조회
* `POST /llm/connections` : upsert(등록/갱신)
* `DELETE /llm/connections/{conn_id}` : 삭제

LLM Connection 필드(최소):

* `id`: 연결 ID
* `name`: 표시명
* `provider`: `"ollama" | "openai" | "anthropic" | "yncai"`
* `base_url`: provider별 API base url
* `api_key`: (옵션) 인증 토큰/키
* `model`: (옵션) 모델명(게이트웨이가 반환하는 모델명을 그대로 수용 가능)
* `timeout_s`: 요청 타임아웃
* `headers`: (옵션) 커스텀 헤더

저장 위치:

* `/data/state/_llm_conns.json`

보안:

* API 응답에서 `api_key`는 `***`로 마스킹 처리

---

### ✅ B. Role Binding (Master/Manager/SubAgent)

역할별 기본 LLM 연결을 설정/조회한다.

* `GET /llm/roles`
* `POST /llm/roles`

저장 위치:

* `/data/state/_llm_roles.json`

역할:

* `master_conn_id`
* `manager_conn_id`
* `subagent_default_conn_id`

---

### ✅ C. yncai Provider 연동(학교 LLM)

학교 제공 게이트웨이를 provider=`yncai`로 등록하고 호출 성공.

* yncai base_url:

  * `https://factchat-cloud.mindlogic.ai/v1/gateway`
* 테스트 API:

  * `POST /llm/test/{conn_id}?prompt=...`

결과:

* `ok:true`
* `provider:"yncai"`
* `text:"OK"`
* (gateway가 반환하는) `model` 값도 함께 반환됨

  * 예: `"claude-sonnet-4-6"`

## 7) 다음 단계 계획: M3-5.2 (LLM Probe Loop v0 + run_auto 결합)

### 7.1 목표(Goal)

“LLM이 계획/결정만 하는 게 아니라 **확인(Probe) 명령을 스스로 생성하고 실행 결과를 근거로 판단**”하는 루프를 최소 구현한다.

* 입력이 부족하거나(unknown/missing) 모호하면(ambiguous)

  * LLM이 **확인용 probe**(shell/API)를 생성
  * SubAgent 실행 결과를 받아
  * 결정을 내리거나 최소 질문만 human에게 요청
* 성공/실패를 evidence/audit/state로 남겨 **재현 가능**하게 만든다.

---

### 7.2 기능 범위(Scope)

#### A) Probe Loop 엔진(v0)

* 트리거:

  * `needs_clarification` (missing_inputs / choices)
  * validate fail (pass=false)
* 동작:

  1. 상황 요약(state + plan + missing + choices + last errors)
  2. LLM에게 “probe/actions 생성” 요청
  3. SubAgent로 실행
  4. 결과를 LLM에 다시 넣어 “결정/입력값/다음 실행” 생성
  5. 반복 상한(예: 3회) + timeout

#### B) Action IR v0 (최소)

LLM 출력은 아래 중 하나만 허용:

* `actions`: shell/http 실행 목록
* `resolved_inputs`: 입력값 확정
* `question`: human에게 묻는 최소 질문(choices 포함)

#### C) run_auto 연결(v0)

* `POST /projects/{id}/run_auto`

  * 내부적으로 plan→probe→execute 흐름을 수행
  * LLM 커넥션은 role=`master`로 선택(yncai 가능)

---

### 7.3 기술 설계(Tech)

* LLM 호출은 `llm_registry.resolve_llm_conn_for_role("master", target_id)`로 연결 선택
* 실행은 기존 SubAgent `POST /a2a/run_script` 재사용 (shell action은 script로 실행)
* 증빙:

  * 각 probe/action 실행 결과를 evidence_refs로 저장
  * audit 이벤트: PROBE_START/PROBE_DONE/DECIDE/EXEC_START/EXEC_DONE/REPLAN 등

---

### 7.4 테스트 케이스(Test Cases)

> **[MANAGER(OpsClaw 서버)]** 기준

1. **missing input 자동 해결**

* 예: iface_in/iface_out 같은 값이 없으면

  * LLM이 `ip -o link show` 같은 probe를 생성
  * 결과로 후보를 선택하거나 질문 1개로 축소

2. **모호한 플레이북 선택**

* “점검해줘” 같이 모호한 request에 대해

  * probe(시스템 상태/대상 확인) 후
  * playbook 후보를 줄이거나 질문 1개로 축소

3. **실패 후 재계획**

* 일부 step 실패를 유도
* LLM이 원인 파악용 probe를 생성하고
* 수정 action을 실행한 뒤 재시도

---

### 7.5 완료 조건(Exit Criteria)

1. `run_auto`가 LLM(master role)을 사용해 probe/actions를 생성한다.
2. 최소 2개 시나리오에서:

   * (a) 자동으로 missing을 줄이고 진행하거나
   * (b) 질문을 1~2개로 축소해 human에게 반환한다.
3. 각 실행이 `audit`/`evidence`에 남아 재현 가능하다.
4. 반복 상한/timeout이 적용되어 무한루프가 없다.
