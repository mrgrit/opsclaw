좋다. 이 사용 케이스들이 기준이다.
이제 M3-5.3을 **“네트워크 입력 자동판별”이 아니라, 저런 다양한 업무를 다 처리할 수 있는 범용 실행 골격을 만드는 단계**로 다시 못 박으면 된다.

문서 기준으로도 OpsClaw의 최종 비전은 이미 분명하다.
사용자는 자연어로 목표만 말하고, LLM이 필요한 확인(probe)을 설계·실행하고, 증거 기반으로 계획을 만들고, CLI/웹/OSS 도구를 찾아·설치·학습·실행·랩핑까지 하며, 검증·오류해결·재계획을 반복하는 시스템이다. 
또 Skills 문서도 구조를 정확히 나눠놨다. Dynamic layer는 Master가 **Probe → Decide → Act**를 설계하고, Skill layer는 반복적·고위험 작업을 재사용 가능한 controlled execution으로 수행한다. `sys.probe`, `net.probe`, `svc.probe`, `tool.discover`, `tool.acquire`, `tool.understand`, `tool.run`, `tool.wrap` 같은 분류가 이미 정의돼 있다. 

즉 네가 방금 준 12개 사용 케이스는 전부 하나의 공통 패턴으로 묶인다.

* 서비스 장애 복구
* 서버 온보딩
* 보안 점검
* 네트워크 구조 파악
* 로그 파이프라인 장애 추적
* 방화벽 정책 변경
* OSS 도구 도입
* 백업/복구 리허설
* IAM 권한 감사
* 성능 튜닝
* 보안 이벤트 대응
* 주간 운영 리포트
* 그리고 네가 추가로 든 **이메일 읽기/분류/삭제/답장**까지

이 모든 건 결국 같은 루프다.

**Goal → Unknowns → Probes → Decisions → Actions → Validate → Replan**

이 방향은 기존 문서에서도 이미 누적돼 있다.
M3-4.4의 미완성 핵심은 “YAML discovery 땜질”이 아니라 **LLM Probe Loop + IR 기반 실행**이 들어가야 비로소 범용 자연어 제어 비전이 충족된다고 정리돼 있다.
M3-5.1 계획도 동일하다. unknown/missing/ambiguous가 있으면 LLM이 probe를 만들고, 실행 결과로 판단하고, 필요한 경우만 최소 질문으로 멈추며, 전 과정을 audit/evidence/state에 남기는 게 목표라고 명시돼 있다.
그리고 장기 로드맵은 IR에 `goal`, `context`, `constraints/policy`, `unknowns`, `probes`, `evidence`, `decisions`, `plan`, `validate`, `errors`, `fixes`, `replans`, `iterations`를 넣는 형태로 정리돼 있다.

그래서 M3-5.3은 이렇게 다시 정의하는 게 맞다.

# M3-5.3의 진짜 정의

**M3-5.3 = 범용 Goal/Unknown/Probe/Decision 구조를 코드에 처음 정착시키는 단계**

아직 M4 수준의 완전한 IR 엔진 전체를 다 만드는 단계는 아니다.
하지만 최소한 현재 `run_auto + input_resolver + probe_loop` 흐름을 아래처럼 바꿔야 한다.

현재 v0:

* playbook 선택
* missing input 계산
* YAML discover.commands 시도
* 안 되면 probe_loop
* 실행

M3-5.3 목표:

* 목표와 현재 상태에서 **unknowns/facts** 추출
* 각 unknown에 대해

  * 이미 아는지
  * deterministic probe로 알 수 있는지
  * LLM probe가 필요한지
  * human 질문/승인이 필요한지
    판단
* probe 결과를 **evidence + rationale + decision**으로 저장
* 필요한 action만 실행
* validate 후 실패하면 다음 가설로 replan

이 구조여야 이메일 자동처리도 된다.
예를 들어 “메일 읽고 요약하고, 의미없는 메일 삭제, 안 중요한 메일 답장” 요청은 다음 unknowns로 분해된다.

* 메일 시스템 종류
* 접근 수단/권한
* 삭제 정책
* 자동 답장 허용 정책
* 중요도 기준
* 답장 tone/style
* 내부/외부 발신 구분
* 휴지통 이동 vs 영구 삭제 여부

이건 인터페이스 번호가 아니라 **facts/unknowns resolution** 문제다.

# 그래서 M3-5.3에서 먼저 만들어야 할 것

우선순위는 이 순서가 맞다.

## 1. 공통 Resolution Result 스키마

어떤 unknown을 다루든 같은 형태로 결과를 낼 수 있어야 한다.

예를 들면 이런 구조다.

```python
{
  "status": "resolved" | "needs_clarification" | "needs_approval" | "insufficient_evidence",
  "fact_key": "mail_access_mode",
  "value": "gmail_api",
  "choices": [],
  "question": None,
  "approval": None,
  "rationale": [
    "OAuth token exists",
    "gmail connector available"
  ],
  "evidence_refs": [...]
}
```

이게 있어야

* OS 판별
* package manager 판별
* service 상태 확인
* 네트워크 구조 파악
* 메일 접근 수단 판별
* 삭제 승인 필요 여부
  를 전부 같은 루프로 처리할 수 있다.

## 2. Unknown/Fact 개념 도입

지금 `input_resolver`는 “missing inputs 채우기”에 가깝다.
이걸 “이 작업을 수행하려면 어떤 사실(fact)이 필요한가”로 승격해야 한다.

예:

* `target_os`
* `pkg_manager`
* `service_name`
* `config_path`
* `iface_out`
* `mail_provider`
* `mail_delete_policy`
* `reply_style`

즉 M3-5.3부터는 입력도 전부 사실(fact)의 한 종류로 보는 게 맞다.

## 3. Probe Action 공통 포맷

문서의 Action IR v0도 같은 방향이다. LLM 출력은 `actions`, `resolved_inputs`, `question` 중 하나로 제한했는데, 다음 단계에서는 이를 `probe`, `decision`, `question/approval` 중심으로 더 일반화해야 한다.

적어도 내부적으로는 이런 구분이 있어야 한다.

* observation probe
* validation probe
* action
* high-risk action
* human question
* approval gate

## 4. Risk/Approval 분기

네 사용 케이스들 공통으로 중요한 건 이거다.

* 로그 읽기, 상태 확인, 메일 목록 읽기 → low risk
* 삭제, 외부 발송, 방화벽 적용, 서비스 재시작, 권한 변경 → approval 또는 policy gate 필요

문서에서도 human 개입은 모호하거나 위험한 결정만 최소로 한다고 잡혀 있다.
이걸 코드에 반영하지 않으면 이메일 자동 삭제/자동 답장 같은 건 바로 위험해진다.

## 5. Evidence / Rationale / Audit 표준화

M3-5.1 이후 문서들은 계속 모든 probe/action이 evidence/audit/state에 남아 재현 가능해야 한다고 못 박고 있다.
이건 이메일 처리에도 똑같다.

* 어떤 메일을 읽었는지
* 왜 “의미없음”으로 판단했는지
* 왜 삭제/답장을 제안했는지
* 실제로 삭제했는지, 휴지통 이동했는지, 초안만 만들었는지

이게 다 남아야 한다.

# 그러면 M3-5.3 구현 순서는 이렇게 바뀐다

이제 step by step도 이 기준으로 가야 한다.

## Step 1

`input_resolver.py`를 “missing input 채우기” 함수에서
**generic fact resolution coordinator**로 확장할 발판을 만든다.

이번 단계에서는 아직 큰 기능 추가 안 한다.
대신 반환 구조를 다음으로 확장할 준비를 한다.

* `resolved_facts`
* `questions`
* `approvals`
* `rationale`
* `evidence_refs`

## Step 2

공통 result helper를 만든다.
예:

* `resolved_result(...)`
* `clarification_result(...)`
* `approval_result(...)`
* `insufficient_result(...)`

이건 아직 어떤 도메인도 안 박고 골격만 만든다.

## Step 3

`probe_loop.py`와 맞물리는 공통 fact/probe schema를 만든다.

즉 LLM이 다루는 대상도 더 이상 “입력값 채우기”만이 아니라,

* 어떤 fact를 알아야 하는가
* 어떤 probe가 필요한가
* 어떤 risk gate가 필요한가
  를 명시하게 한다.

## Step 4

첫 번째 observation domain 샘플 2개만 붙인다.
`net.probe`보다 더 범용적인 것부터 가는 게 맞다.

* `sys.probe` : OS / distro / pkg manager / uname / service manager
* `svc.probe` : 서비스 상태 / 포트 / 프로세스 / health endpoint

이 둘이 있어야 서버 온보딩, 장애복구, 로그 파이프라인, 백업 복구, 성능 튜닝 대부분의 첫 probe가 가능하다. `net.probe`는 그 다음이다. Skills 문서도 이런 분류를 이미 제안하고 있다. 

## Step 5

그 다음에 `net.probe`를 붙인다.
여기서 `iface_in/out`은 하나의 세부 fact로 들어간다.
주연이 아니라 `net.probe`의 하위 결과일 뿐이다.

## Step 6

그 다음에 tool lifecycle과 higher-risk action gate를 붙인다.
그래야 “도구 없으면 찾아서 clone해서 써” 같은 사용 케이스가 진짜 된다. 이건 로드맵상 M4-2, M4-3 방향과도 일치한다. 

# 결론

네가 준 사용 케이스들을 기준으로 보면,
M3-5.3의 첫 구현 포인트는 **NIC가 아니라 공통 unknown/fact/probe/decision 구조**다.

그리고 이메일 예시까지 포함하면 더 분명하다.

OpsClaw는 앞으로 이런 식으로 가야 한다.

* 특정 작업용 하드코딩 자동화 도구가 아니라
* 자연어 목표를 받아
* 필요한 unknown을 식별하고
* probe를 설계/실행하고
* 위험도에 따라 질문/승인/자동실행을 나누고
* 근거와 증빙을 남기며
* 재계획까지 반복하는 범용 오케스트레이터

이제부터 구현도 그 비전에 맞춰서 간다.

다음 단계는 **`api/input_resolver.py`를 건드리기 전에, M3-5.3용 공통 Resolution Result 구조를 어떤 파일에 어떤 형태로 둘지 먼저 정하는 것**이다.
그걸 정하고 나면, 그 다음에 파일별로 아주 작게 패치해 나가면 된다.


좋아.
그럼 **M3-5.3 Observation Domain v0 설계안**을 먼저 고정하자.

이 설계의 목적은 딱 하나다.
지금처럼 작업별로 `iface resolver`, `os resolver`, `mail resolver`를 계속 늘리는 게 아니라, **범용 관측 도메인**을 먼저 만들고, 그 위에서 LLM이 Probe→Decide→Act를 돌게 하는 것이다. 이 방향은 기존 비전 문서의 “자연어 목표 → probe 설계/실행 → evidence 기반 결정 → 최소 human 질문”과 정확히 일치한다.
또 Skills 문서가 말한 Hybrid Model, 즉 Dynamic Layer는 Master가 Probe→Decide→Act를 설계하고, Skill Layer는 반복적·고위험·표준화가 필요한 것만 모듈화한다는 원칙과도 맞다.

## 1. M3-5.3에서 만들 것의 정체

M3-5.3은 “Suricata 인라인용 NIC 판별 기능 추가”가 아니다.
M3-5.3은 **Observation Domain v0**를 세워서, 현재 `run_auto + input_resolver + probe_loop`가 앞으로 어떤 작업이 들어와도 공통적으로 “무엇을 확인해야 하는가”를 같은 방식으로 다루게 만드는 단계다.

즉 앞으로 작업은 이렇게 나뉜다.

* 사용자의 목표를 해석한다.
* 필요한 unknowns를 뽑는다.
* unknowns를 Observation Domain에 배정한다.
* Domain probe 결과를 evidence로 저장한다.
* LLM 또는 decider가 결정을 내린다.
* 위험하거나 사용자만 아는 사항은 질문한다.
* 실행 후 validate / replan 한다.

이건 문서의 장기 IR 방향인 `unknowns`, `probes`, `evidence`, `decisions`, `plan`, `validate`, `errors`, `fixes`, `replans` 구조와도 연결된다.

## 2. Observation Domain v0 목록

M3-5.3 v0에서는 4개만 잡는다.

### `sys.probe`

대상 시스템의 기본 성격을 파악한다.
예를 들면 운영체제, 배포판 계열, 패키지 매니저, 서비스 매니저, 호스트명, 커널 같은 것들이다. Skills 문서의 Observation Skills 분류에서도 `sys.probe`는 OS/CPU/MEM/DISK/PROC 기본 수집 역할로 정의돼 있다.

### `svc.probe`

서비스/프로세스/포트/헬스 상태를 관측한다.
장애 복구, 로그 파이프라인 추적, 서비스 배포 검증, 성능 이상 원인 추적 같은 데 공통으로 쓰인다. Skills 문서에서도 `svc.probe`는 포트 리슨, 헬스 엔드포인트, 프로세스 상태를 담당한다.

### `net.probe`

네트워크 환경을 관측한다.
NIC 목록, 주소, 라우트, DNS, reachability, default route 같은 정보를 수집하고, 필요 시 `iface_in/out` 같은 결정의 근거를 제공한다. 중요한 점은 `ens33/ens34` 같은 이슈가 특정 제품 전용 코드가 아니라 `net.probe` 도메인으로 흡수돼야 한다는 점이다. 이 방향은 Skills 문서에 이미 명시돼 있다.

### `fs.probe`

파일/디렉토리/설정 파일의 존재, 권한, 크기, 수정 시각, 간단한 목록, 설정 파일 후보 등을 관측한다.
파일 정리, 설정 위치 탐색, 로그 존재 여부 확인, 백업 파일 유무 확인, 리포트 생성 같은 작업에 쓰인다.

이 4개면 지금 네가 준 사용 케이스들의 첫 probe 대부분을 감당할 수 있다. 서비스 장애 복구, 신규 서버 온보딩, 보안 점검, 네트워크 구조 파악, IDS/로그 파이프라인 장애, 백업/복구, 파일 정리, 주간 운영 리포트까지 공통의 진입점을 만들 수 있다. 이건 네가 준 USER_CASE 방향과도 맞다. 

## 3. 공통 출력 스키마 원칙

도메인은 달라도 상위 레이어가 읽는 모양은 최대한 같아야 한다.
M3-5.3 v0에서 각 domain probe는 최소한 이런 형태를 목표로 한다.

```python
{
  "status": "ok" | "partial" | "failed",
  "domain": "sys.probe",
  "facts": {...},
  "rationale": [...],
  "evidence_refs": [...],
  "unknowns": [...]
}
```

여기서 핵심은 5개다.

* `status`: probe 성공/부분성공/실패
* `domain`: 어떤 관측 도메인인지
* `facts`: 실제 관측 결과
* `rationale`: 사람이 읽을 수 있는 요약 근거
* `evidence_refs`: 원문 stdout/stderr, 파일, 아티팩트 참조
* `unknowns`: 여전히 남아 있는 불확실성

이 구조는 Skills 문서의 Skill Contract에서 말하는 입력/출력/정책/증빙 분리 원칙과 맞다. 출력에는 최소 `ok`, `data`, `logs`, `artifacts`, `evidence_refs`가 있어야 하고, 증빙은 실행 명령/응답 원문과 로그, 생성 파일, 버전 정보까지 남겨야 한다고 정리돼 있다.

## 4. 언제 probe하고, 언제 사용자에게 묻나

이건 네가 방금 못 박은 원칙대로 간다.

### 시스템이 직접 확인 가능한 것

probe로 푼다.

예를 들면:

* 운영체제, 배포판, 패키지 매니저
* 포트 리슨, 서비스 상태, 최근 오류 로그
* NIC/ADDR/ROUTE
* 파일 존재 여부, 권한, 크기
* 설치된 도구, 바이너리 존재 여부
* API endpoint 도달 가능 여부

### 사용자만 아는 것

질문한다.

예를 들면:

* 이 메일을 자동 삭제해도 되는가
* 외부 고객에게 자동 답장을 보내도 되는가
* 재시작/스케일 변경/차단 룰 적용을 승인하는가
* 두 NIC 후보 중 실제 운영상 in/out은 어느 쪽인가
* 어떤 파일이 “정리 대상”이고 무엇은 보존해야 하는가
* WAF 룰을 탐지 모드로 갈지 차단 모드로 갈지

즉 원칙은 간단하다.

**사실 확인은 시스템, 의도/정책/책임은 사용자.**

이건 기존 문서의 “사람은 모호하거나 위험한 결정만 최소로 승인/선택”과도 일치한다.

## 5. M3-5.3에서 Observation Domain이 차지하는 위치

지금 단계에서 Observation Domain은 최종 Skill 시스템의 완성형이 아니다.
역할은 두 가지다.

첫째, `probe_loop`가 뽑아낸 unknown을 어디서 확인해야 하는지 **도메인별로 정리된 실행점**을 제공한다.
둘째, 자주 반복되고 결과 형식이 표준화돼야 하는 관측은 나중에 Skill로 승격할 수 있게 **관측 결과 스키마와 증빙 구조를 먼저 고정**한다.

즉 지금은 “resolver 파일 난립”이 아니라, “나중에 skill로 승격 가능한 관측 도메인 v0”를 만든다고 보면 된다. Skills 문서도 반복적·고위험·표준화 필요성이 있을 때 Skill로 승격한다고 말한다.

## 6. M3-5.3 구현 우선순위

여기서부터는 실제 작업 순서다.

### 1단계: `sys.probe v0`

이미 일부 시작됐다.
`target_os`, `pkg_manager`를 지금처럼 개별 fact resolver로 두는 대신, 다음 단계에서 `sys.probe` 결과 안으로 묶을 준비를 한다.

최소 facts:

* `target_os`
* `pkg_manager`
* `service_manager`
* `hostname`
* `kernel`

### 2단계: `svc.probe v0`

최소 facts:

* 서비스 active 여부
* 프로세스 존재
* 리슨 포트
* health endpoint 결과
* 최근 오류 로그 요약

이게 만들어지면 서비스 지연/장애 자동 복구, IDS/로그 파이프라인 장애 추적, 배포 검증 같은 유즈케이스가 크게 열린다.

### 3단계: `net.probe v0`

최소 facts:

* interfaces
* addresses
* default_route
* route_summary
* DNS 또는 reachability 기본 결과
* 필요 시 `candidate_iface_in/out`

중요한 건 여기서도 목적이 NIC 맞추기 자체가 아니라, **네트워크 관측 결과를 구조화하는 것**이다.

### 4단계: `fs.probe v0`

최소 facts:

* exists
* type
* size
* modified_time
* perms
* directory listing summary
* config file candidates

이게 있으면 파일 정리, 로그 파일 확인, 백업 파일 확인, 설정 탐색, 리포트 생성에 공통으로 쓸 수 있다.

## 7. 당장 코드로 뭘 할지

지금 바로는 `net_fact_resolver.py` 만들지 않는다.
그건 보류한다.

다음 구현은 **`sys_fact_resolver`를 `sys.probe v0` 방향으로 정리하는 것**부터 가는 게 맞다.
이미 `target_os`, `pkg_manager`가 실제로 자동 채워지고 있으니, 이걸 “개별 팩트 해결기”가 아니라 “첫 Observation Domain의 축소판”으로 승격하는 쪽이 가장 자연스럽다.
이 방향은 M3-5.1 문서의 “missing input 자동 해결은 probe 결과로 resolved_inputs를 만들거나, 질문 1~2개로 축소해야 한다”는 완료 조건과도 맞다.

## 결론

정리하면 이렇다.

* M3-5.3은 특정 작업용 resolver 모음집이 아니라 **Observation Domain v0 정착 단계**다.
* 시작 도메인은 `sys.probe`, `svc.probe`, `net.probe`, `fs.probe` 4개다.
* 사용자만 아는 것과 크리티컬한 결정은 그냥 사용자에게 묻는다.
* 시스템이 확인 가능한 건 probe로 푼다.
* 지금 다음 구현은 `net`이 아니라, 이미 일부 성공한 `sys`를 먼저 **`sys.probe v0`로 정리**하는 게 맞다.

오케이면 다음 턴에서 바로 **Step 7-2: `sys_fact_resolver.py`를 `sys.probe v0` 형태로 바꾸기 위한 최소 리팩터링**으로 들어가자.
