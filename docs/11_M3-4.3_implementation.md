# OpsClaw M3-4.3 완료 보고서 (Multi-Target DAG Playbook + Failure Policy)

작성일: 2026-03-04
범위: **Playbook을 jobs/depends_on(DAG)로 확장 + job 단위 target_id 멀티 타겟 실행 + 실패정책(fail_fast/continue) + 결과 구조화(state/audit)**

---

## 1) 목표(Goal)

M3-4.3의 목표는 다음을 **실제로 동작**하게 만드는 것이다.

1. Playbook 스키마를 `jobs` + `depends_on` 기반 **DAG 형태**로 확장한다.
2. job 단위로 `target_id`를 지정하여 **멀티 타겟 분산 실행**을 지원한다.
3. 실패정책(`failure_policy`)을 최소 2개 지원한다.

   * `fail_fast`: 실패 시 즉시 중단
   * `continue`: 실패해도 계속 진행(단 depends_on 충족 실패 시 job은 skipped 처리)
4. 실행 결과를 job/step 단위로 구조화하여 **프로젝트 state에 누적 저장**한다.
5. 로컬(`local-agent-1`) 및 원격(`remote-1`)에서 DAG playbook 실행이 재현 가능해야 한다.

---

## 2) 구현 결과 요약(What’s Done)

### ✅ A. Playbook DAG 스키마 도입 (jobs/depends_on)

Playbook YAML에 아래 구조를 추가 지원:

* `defaults`:

  * `target_id`
  * `timeout_s`
  * `failure_policy` (`fail_fast` | `continue`)
* `jobs`:

  * 각 job은 `target_id`, `steps`를 가짐
  * `depends_on`으로 선행 job 의존성 지정 가능
* 기존 `steps` 기반 playbook은 **하위호환 유지**

---

### ✅ B. Runner 확장: jobs 모드(DAG) + 레거시 steps 모드 유지

`workflows/playbook_runner.py`에서 다음을 구현:

* playbook에 `jobs`가 존재하면 **jobs(DAG) 모드**
* `jobs`가 없으면 기존 `steps` 기반 실행 로직 그대로 실행

jobs 모드 핵심 로직:

* Kahn 알고리즘 기반 **토폴로지 정렬(toposort)**
* job 실행 시:

  * `target_id` → Manager의 `resolve_subagent_url(target_id)`로 subagent_url resolve
  * job 내부 steps를 순차 실행
  * step 결과(run_id/exit_code/stdout/stderr/evidence_refs)를 누적
* depends_on 규칙(v0):

  * 선행 job 중 하나라도 `ok`가 아니면 해당 job은 `skipped`
  * skipped job에는 `reason: dependency_not_ok` 저장

---

### ✅ C. 실패정책(failure_policy) 지원

* `fail_fast`

  * job 실행 중 step 실패(exit_code != 0) 발생 시 **즉시 Playbook 종료**
* `continue`

  * 실패해도 다음 job을 진행
  * 단, depends_on이 있는 job은 dependency_not_ok면 skipped 처리

---

### ✅ D. 결과 구조화 + state/audit 저장

* 응답 결과 및 state 저장 결과에 `mode: "jobs"` 포함
* `jobs[]` 배열에 job별 결과 구조화:

  * `job_id`, `target_id`, `status(ok/failed/skipped)`, `depends_on`, `reason`, `steps[]`
* 프로젝트 state에 `playbook_runs[]`로 누적 저장됨

## 4) 완료 조건(Exit Criteria)

아래 조건을 모두 만족하면 M3-4.3 완료로 판단한다.

1. DAG playbook(`jobs/depends_on`)이 정상 로드되고 실행된다.
2. job 단위 `target_id`로 로컬/원격 분산 실행이 가능하다.
3. `fail_fast` 정책이 실패 시 즉시 중단되는 것이 확인된다.
4. `continue` 정책이 실패를 허용하되, depends_on 불충족 job을 `skipped` 처리하는 것이 확인된다.
5. 결과가 `mode:"jobs"` 구조로 반환되고, 프로젝트 state의 `playbook_runs[]`에 누적 저장된다.
6. 원격 job 실행 시 원격 subagent 서버에 evidence 파일이 생성된다.

➡️ 위 조건 모두 충족됨.

## 6) 주요 이슈 / 트러블슈팅 (What Hurt)

### 이슈 1) playbook not found (continue 파일 누락)

**증상**

* `FileNotFoundError: playbook not found: ops-health-dag-continue`
* API는 500을 반환하고 jq가 파싱 실패

**원인**

* `ops-health-dag-continue.yaml` 파일이 레포/컨테이너에 존재하지 않음
* Dockerfile이 `COPY` 기반이라 파일 추가가 컨테이너에 자동 반영되지 않음

**대응**

* `api/playbooks/`에 파일 생성
* `docker compose build --no-cache api`로 재빌드 후 반영

---

### 이슈 2) COPY 기반 구성으로 인한 “YAML 추가 시 리빌드 필요”

**대응**

* playbook 파일 추가/수정 후:

  * `docker compose build --no-cache api`
  * `docker compose up -d`
  * `/playbooks` 목록에서 반영 확인
