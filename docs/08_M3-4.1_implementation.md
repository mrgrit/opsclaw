# OpsClaw M3-4.1 완료 보고서 (Targets + Remote SubAgent Routing)

작성일: 2026-03-03  
범위: **M3-4.1 Target Registry + target_id 기반 원격 SubAgent 라우팅/실행**

---

## 1) 목표(Goal)

M3-4.1의 목표는 다음 2가지를 **실제로 동작**하게 만드는 것이었다.

1. **Targets(원격 실행 대상) 등록/조회 API**를 제공한다.  
2. Manager API가 `target_id`를 받아 해당 target의 `base_url`로 **원격 SubAgent**를 호출하여 스크립트를 실행하고 결과를 수집한다.

---

## 2) 구현 결과 요약(What’s Done)

### ✅ A. Target Registry (Manager API)
- `/targets` 리스트 조회
- `/targets/{target_id}` 단건 조회
- `/targets` 등록/갱신(Upsert)
- `/targets/{target_id}` 삭제

Target 저장 필드(최소):
- `id`: 문자열 (예: `remote-1`)
- `base_url`: 원격 SubAgent 주소 (예: `http://192.168.208.138:55123`)
- `name`: 표시명
- `tags`: 분류 태그 배열 (예: `["suricata"]`)
- `notes`: 메모

Target 저장 위치:
- `/data/state/_targets.json` (JSON 기반)

---

### ✅ B. target_id 기반 원격 실행 라우팅
Manager의 프로젝트 실행 API에서 다음이 동작함:

- `POST /projects/{project_id}/run` 요청에 `target_id`를 받음
- `target_id`가 `local-agent-1`이면 기존 방식대로 로컬 subagent 사용
- `target_id`가 `remote-1`처럼 등록된 타겟이면:
  - Target Registry에서 `base_url` 조회
  - `base_url + /a2a/run_script`로 원격 SubAgent 호출
  - `exit_code/stdout/stderr/evidence_refs` 수집 후 프로젝트 state에 append

---

### ✅ C. Evidence 생성/수집 동작 확인
원격 SubAgent가 스크립트를 실행하면:
- 결과 stdout/stderr 로그 및 evidence 파일이 원격 SubAgent 호스트의 `/data/evidence/`에 생성됨
- Manager는 원격 SubAgent가 반환하는 `evidence_refs`를 기록함

---

## 3) 완료 조건(Exit Criteria)

아래 조건을 모두 만족하면 M3-4.1 완료로 판단.

1. `GET /targets`가 200으로 응답하고 등록된 target 목록을 반환한다.
2. `POST /targets`로 target 등록 후 `GET /targets/{id}`로 조회가 된다.
3. 등록한 target의 `base_url`이 실제 원격 SubAgent에 연결된다(health OK).
4. Manager의 `POST /projects/{project_id}/run` 호출 시 `target_id`를 주면:
   - 해당 target의 SubAgent에서 스크립트가 실행되고
   - 실행 결과(exit_code, stdout/stderr, evidence_refs)가 Manager 응답에 포함된다.
5. 원격 SubAgent 호스트에서 `/data/evidence`에 실행 결과 파일이 실제 생성된다.

➡️ 위 조건 모두 충족됨.

```bash

  -H "content-type: application/json" \
  -d '{"id":"remote-1","base_url":"http://192.168.208.138:55123","name":"lab1","tags":["suricata"],"notes":"test"}' | jq .

# 목록/단건 조회
curl -s http://localhost:8000/targets | jq .

```

### 4.5 프로젝트 생성 + 원격 실행 라우팅 검증 (Manager → Remote SubAgent)

```bash
# 프로젝트 생성
PID=$(curl -s -X POST http://localhost:8000/projects \
  -H "content-type: application/json" \
  -d '{"project_type":"generic","targets":[],"request_text":"routing test"}' | jq -r .project_id)

# 원격 타겟으로 실행
curl -s -X POST "http://localhost:8000/projects/$PID/run" \
  -H "content-type: application/json" \
  -d '{"command":"uname -a","timeout_s":30,"approval_required":false,"target_id":"remote-1"}' | jq .
```

- `evidence_refs`에 `/data/evidence/...` 경로들이 포함

## 5) 주요 이슈 / 트러블슈팅 (What Hurt)

### 이슈 1) 파이썬 문법/들여쓰기 오류로 API가 반복적으로 기동 실패

**증상**

- `IndentationError`
- `SyntaxError: 'return' outside function`
- `uvicorn` 시작 실패

**대응**

- `python3 -m py_compile api/main.py`로 컴파일 오류 위치를 빠르게 확인
- 작은 단위로 수정 후 컨테이너 재빌드/재기동

### 이슈 2) Targets API 404 / Not Found

**원인**

- 라우팅 핸들러가 실제 `main.py`에 반영되지 않았거나 import/정의 누락

**대응**

- `openapi.json`에서 `/targets` 존재 여부 확인
- 실제 핸들러/의존 함수 누락 제거

### 이슈 3) NameError 연쇄 (`Request`, `TargetReg`, `authz`, `list_targets` 등)

**원인**

- 패치 과정에서 의존 함수/모델 이름이 기존 코드와 불일치

**대응**

- 현재 프로젝트에 존재하는 타입/함수명을 기준으로 수정
- Targets 구간은 기존 `Target(BaseModel)` 그대로 사용
- auth 미들웨어 의존을 최소화/정리

### 이슈 4) `/data/state/_targets.json`이 비어있거나 손상되면 `JSONDecodeError`

**증상**

- `GET /targets`가 500 오류 반환

**대응**

- `state_store.load_json()`이 비정상 JSON에 대해 기본값을 반환하도록 보강
- `_targets.json` 최초 생성/초기값 보장 로직 추가

### 이슈 5) 원격 SubAgent endpoint 혼동 (`/run` vs `/a2a/run_script`)

**증상**

- `404 Not Found`

**대응**

- SubAgent OpenAPI에서 실제 path가 `/a2a/run_script`임을 확인
- Manager 라우팅도 해당 경로로 고정

## 6) 결과 상태 (현재 가능한 것)

### 지금 가능한 것

여러 서버에 SubAgent만 떠 있으면 다음이 가능하다.

- Manager에서 target 등록
- 프로젝트 실행 시 `target_id` 지정
- 원격 실행 결과 수집

### 아직 불완전한 점 / 제약

- `evidence_refs`는 원격 SubAgent의 `/data/evidence/...` 경로를 그대로 반환하므로, Manager 쪽 Evidence Pack에 원격 evidence를 자동으로 끌고 와서 합치기는 아직 미구현
- Playbook / LangGraph 실행이 아직 `target` 분산 실행까지는 확장되지 않음
- LLM이 알아서 서버별 작업을 배분하는 규칙/플래너 단계는 아직 없음
