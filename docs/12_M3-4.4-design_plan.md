# OpsClaw M3-4.4 계획 (Planner v0: 자동 타겟/플레이북 선택 + 질문 생성)

작성일: 2026-03-04
범위: **요청 텍스트(request_text)를 기반으로 target/playbook을 자동 선택하고, 모호하면 질문(next_questions)을 생성하는 Planner v0**

---

## 1) 하고자 하는 것 요약(Why / What)

M3-4.4에서 하고자 하는 핵심은:

* 사용자가 “자연어로 요청”만 던지면
  OpsClaw가 **어느 서버(target)에서** **어떤 플레이북(playbook)** 을 실행해야 할지 **자동으로 결정**한다.
* 요청이 모호하거나 매칭이 안 되면, 실행을 강행하지 않고
  **필요한 질문(next_questions)** 을 만들어서 “결정에 필요한 정보”를 받는다.

> 목표 수준은 LLM 기반 고급 플래너가 아니라, **태그 매칭 기반(룰 + 간단 점수)** 의 Planner v0.

---

## 2) 기능적 설명(Functional)

### 2.1 입력

* 프로젝트 생성 시 저장된 `request_text`
* 등록된 Targets (`/targets`, target.tags, target.name, notes)
* 등록된 Playbooks (`/playbooks`, playbook id/name, yaml 내용 일부)

### 2.2 출력(Planner 결과)

Planner는 아래 중 하나를 반환한다.

1. **실행 가능**

* `selected_target_ids`: 예) `["remote-1"]` 또는 `["local-agent-1", "remote-1"]`
* `selected_playbook_id`: 예) `"suricata-inline-install"` 또는 `"ops-health-dag"`
* `plan_rationale`: 왜 이렇게 골랐는지 근거(태그/키워드/점수)

2. **질문 필요**

* `next_questions`: 예)

  * “어느 서버에 설치할까요? (remote-1 / remote-2 / local-agent-1)”
  * “suricata 설치가 목적이면 인라인 설치와 모니터링 모드 중 무엇을 원하나요?”

### 2.3 동작 흐름(프로젝트 기준)

* `/projects/{id}/plan` (신규) 호출

  * `request_text` 기반으로 target/playbook 추천
* `/projects/{id}/run_auto` (신규, 선택)

  * plan 결과가 실행 가능하면 즉시 `run_playbook` 실행
  * 질문 필요면 `next_questions` 반환

## 3) 기술적 설명(Technical)

### 3.1 Planner v0 알고리즘(간단, 확실한 방식)

**(A) 키워드/태그 매칭 점수화**

* 요청 텍스트에서 키워드 추출(단순 토큰화/소문자/한글 포함)
* target.tags / target.name / notes에서 키워드 매칭 점수 계산
* playbook id/name 및 yaml 내부 “keywords” 섹션(추가 가능) 매칭 점수 계산

**점수 예시**

* exact tag match: +5
* 부분 일치: +2
* name/notes match: +1
* 다수 매칭 누적

**(B) 후보 선택 규칙**

* target:

  * 최고 점수 target 1개(기본)
  * “복수 대상” 키워드(“모든 서버”, “전체 점검”, “각 서버”)가 있으면 top N 선택(옵션)
* playbook:

  * 최고 점수 1개 선택
  * 동점/불확실하면 질문 생성

**(C) 임계값 기반 결정**

* `best_score < threshold` → 질문 필요
* `best_score - second_best < margin` → 질문 필요(동점/근소 차)

### 3.2 데이터/상태 저장

프로젝트 state에 아래 필드 추가:

* `plan`:

  * `status`: `"ready"` | `"needs_clarification"`
  * `selected_targets`: [...]
  * `selected_playbook_id`: str
  * `rationale`: {...}
  * `next_questions`: [...]
  * `created_at`

또는 `audit`에도 `PLAN_START/PLAN_DONE` 이벤트 기록.

### 3.3 파일/모듈 구조(권장)

* `api/planner.py` (신규)

  * `plan_request(request_text, targets, playbooks) -> PlanResult`
* `api/main.py`

  * `POST /projects/{id}/plan`
  * `POST /projects/{id}/run_auto` (선택)
* `api/playbook_store.py`

  * 목록 + (선택) playbook 메타 추출 함수
* `api/targets_store.py`

  * targets 조회(이미 있음)

> LLM 호출은 v0에서는 선택 사항.
> 먼저 deterministic planner를 구현하고, 나중에 M3-4.4.x에서 “LLM assist”로 확장.

## 4) 완성 시 변경되는 것(What Changes)

### 4.1 사용자 경험 변화

* 지금(M3-4.3): 사용자가 `target_id`, `playbook_id`를 직접 지정
* M3-4.4 이후:

  * `request_text`만 주면 OpsClaw가 추천/결정
  * 불확실하면 질문을 만들어서 확인 후 실행

### 4.2 API 변화(추가)

* `POST /projects/{project_id}/plan` (신규)
* `POST /projects/{project_id}/run_auto` (신규, 선택)

기존 API(`run_playbook`, `targets`, `playbooks`)는 변경 없음.

## 6) 완료 조건(Exit Criteria)

1. `POST /projects/{id}/plan`이 200으로 응답한다.
2. plan 결과가 **두 모드**로 안정적으로 나온다.

   * ready: `selected_targets`, `selected_playbook_id` 포함
   * needs_clarification: `next_questions` 포함
3. plan 결과가 프로젝트 state(`plan`)에 저장된다.
4. 최소 5개 테스트 케이스를 통과한다.
5. (선택) `run_auto`가 ready 상태에서 `run_playbook`을 자동 호출하여 실행까지 완료한다.
