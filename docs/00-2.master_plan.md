좋다.
아래는 **OldClaw 개발 계획서 v2**다.

이번 버전은 다음을 반영했다.

* 이름을 **OldClaw**로 변경
* 기존 계획의 중심축은 유지
* **새로운 업무가 들어와도 코어 수정 없이 skill / playbook 추가만으로 대응**하도록 명시 강화
* **배치 작업 / 지속 실행형 작업 / 상시 모니터링형 작업**을 정식 업무유형으로 포함
* **히스토리 관리 및 경험 축적 전략**을 독립 축으로 강화
* 기존 내용과 동일한 비중을 유지하면서 전체를 다시 정리

# OldClaw 개발 계획서 v2

## 부제

**pi runtime 기반 정보시스템 운영/보안 작업 오케스트레이션 플랫폼**

---

## 1. 시스템 한 줄 정의

**OldClaw**는 오픈소스 에이전트 **pi**의 핵심 실행 런타임을 재사용하되, 그 위에 **Master–Manager–SubAgent 구조, Asset Registry, Policy/Approval, Evidence/Validation, Experience/RAG, A2A 분산 실행, LangGraph 상태기계**를 얹어 **내부망 정보시스템 운영 업무를 안전하고 검증 가능하게 수행하는 전문 운영 에이전트 플랫폼**이다.

---

## 2. 왜 pi를 기반으로 만드는가

## 2.1 pi를 그대로 쓰지 않는 이유

pi는 에이전트 런타임과 명령 실행 능력은 강하지만, OldClaw가 목표로 하는 영역은 더 좁고 더 깊다.

OldClaw가 필요한 영역은 다음과 같다.

* 작업 대상이 불특정 인터넷이 아니라 **내부망 자산(asset)** 이다
* 대상 업무가 일반 코딩이 아니라 **정보시스템 운영 / 보안 / 온보딩 / 복구 / 점검 / 배포 / 폐기 / 모니터링** 이다
* **승인 / 정책 / 증빙 / 검수 / 롤백 / 감사 추적**이 필수다
* “실제로 했다”는 말을 믿는 게 아니라 **증빙과 검증을 통해 완료를 판정**해야 한다
* 쓰면 쓸수록 과거 히스토리와 경험을 활용해 더 잘하는 **운영 특화 자율 시스템**이 되어야 한다

즉 pi는 “잘 만든 범용 실행 엔진”이고, OldClaw는 “그 엔진을 내장한 정보시스템 운영 전문 장비”다.

## 2.2 pi를 재사용해야 하는 이유

반대로 밑바닥부터 에이전트 런타임을 다시 만드는 것은 비효율적이다. pi는 이미 다음을 제공한다.

* 모델 추상화 및 다중 모델 연결
* 도구 호출과 실행 루프
* 상태를 가진 에이전트 런타임
* CLI / SDK / 대화형 작업 흐름
* 명령 실행형 에이전트 기반

따라서 OldClaw는 **pi를 포크해서 다른 이름으로 바꾸는 프로젝트가 아니라**,
**pi를 실행 엔진으로 임베드하고 그 위에 운영 오케스트레이션 계층을 새로 구축하는 프로젝트**로 가야 한다.

---

## 3. OldClaw의 목표

OldClaw의 목표는 아래와 같다.

* 내부망 자산을 기준으로 작업 대상을 관리
* Manager가 작업을 orchestration
* SubAgent가 실제 시스템 작업 수행
* Master가 계획 / 고난도 추론 / 검수 담당
* 모든 작업의 **명령, stdout, stderr, exit code, 검증 결과, 보고서**를 남김
* 새로운 업무가 생겨도 코어를 뜯지 않고 **skill / playbook 추가로 확장**
* 과거 작업 히스토리와 경험을 smart retrieval로 꺼내 활용
* 일회성 업무뿐 아니라 **배치 / 주기 / 모니터링 / 상시 운영형 업무**까지 커버

---

## 4. 비목표

OldClaw는 다음을 목표로 하지 않는다.

* 범용 개인 챗봇
* 인터넷 일반 검색/브라우징 중심 에이전트
* 무한 자율 AI 실험 플랫폼
* 승인 없이 모든 작업을 즉시 실행하는 시스템
* 사람을 완전히 제거한 시스템

OldClaw는 **사람을 최소화하지만 제거하지 않는 운영 시스템**이다.

---

## 5. 핵심 설계 원칙

## 5.1 Asset-first

관리와 정책의 기준은 `target`이 아니라 **asset**이다.

* Asset = 관리 대상 자산의 정체성
* Target = 실행 시점에 asset에서 파생되는 실행 endpoint

즉 등록/정책/상태/기억의 기준은 asset이고, 실행 직전에만 target으로 resolve한다.

## 5.2 Evidence-first

모든 완료 주장은 반드시 증빙으로 닫아야 한다.

필수 증빙:

* 어떤 자산에
* 어떤 명령/도구를
* 어떤 입력으로
* 언제 실행했고
* stdout / stderr / exit code가 무엇이며
* 어떤 테스트로 검증했고
* 결과가 무엇인지

## 5.3 Human-minimized, not Human-eliminated

사람 질문은 최소화하되 다음은 반드시 사람 개입 가능해야 한다.

* 고위험 변경
* 생산계 DB/네트워크/보안정책 변경
* 민감정보 반출 가능성
* 불확실성이 큰 파괴적 조치
* 최종 검수

## 5.4 Graph-based orchestration

작업은 while loop 임시 구현이 아니라 **LangGraph 기반 상태기계**로 관리한다.

대표 상태:

* intake
* clarify
* plan
* select_assets
* resolve_targets
* execute
* validate
* review
* replan
* report
* close

## 5.5 Distributed execution

Manager가 모든 자산에 SSH로 직접 접속하는 구조가 아니라,
**A2A + SubAgent** 기반 분산 실행으로 간다.

## 5.6 Extensible by skill / playbook

새로운 업무가 나와도 **코어를 수정하지 않고 skill과 playbook 추가만으로 끝나야 한다.**

이 원칙은 OldClaw의 매우 중요한 설계 목표다.

즉 새로운 업무는 다음 조합으로 수용한다.

* 기존 Tool 재사용
* 필요한 Skill 추가
* Skill을 조합한 Playbook 추가
* Policy / Approval rule 연결

코어 런타임, 상태기계, 저장계, 증빙계는 그대로 유지한다.

## 5.7 History-aware, context-light

OldClaw는 과거를 “전부 context에 넣어 기억하는 시스템”이 아니다.
대신 raw history를 저장하고, 구조화 기억과 경험으로 승격한 뒤, 현재 작업에 필요한 것만 smart retrieval로 꺼내 사용하는 시스템이다.

## 7. 에이전트 역할 정의

## 7.1 Master

Master는 비용이 큰 고급 추론 에이전트다.

### 역할

* 사용자 요청을 더 정밀한 작업 목표로 정제
* 작업을 마일스톤/작업단위로 분해
* Manager에게 전략 지시
* 복잡한 RCA, 전략 전환, 예외 상황 해결
* Manager 종료 작업에 대한 **실제 완료 검수**
* 절차/도구/테스트/판정이 부적절하면 반려

### Master를 항상 쓰지 않는 이유

* 비용이 높다
* 모든 작업에 고급 추론이 필요한 것은 아니다

### Master 사용 조건

* 복합 계획 수립
* 고난도 replan
* 반복 실패
* 고위험 작업 전 검토
* 최종 검수
* 중요 보고서 품질 보정

---

## 7.2 Manager

Manager는 OldClaw의 실질적 본체다.

### 역할

* human과의 소통 창구
* project 생성 및 상태 관리
* asset 탐색 및 선택
* target resolve
* skill / playbook 선택
* subagent 작업 script 작성 및 전달
* 스케줄링, orchestration, 중재
* 증빙 수집
* 검증 수행
* 필요 시 Master escalation
* 최종 보고

### 본질

Manager는 단순히 말을 잘하는 모델이 아니라,
**작업 입출력과 작업 흐름을 통제하는 실행 관리자**다.

---

## 7.3 SubAgent

SubAgent는 각 자산에 설치되어 실제 시스템 작업을 수행하는 실행기다.

### 역할

* 로컬 명령 실행
* stdout / stderr / exit code 반환
* 파일 read/write/edit
* health / capabilities 제공
* 로컬 probe
* script 실행
* evidence 수집

### 제약

* 자의적 계획 수립 최소화
* manager 지시 우선
* 고위험 작업은 정책 따라 제한
* 결과 은폐 금지

## 8. pi 개념과 OldClaw 개념 매핑

## 8.1 Tool

가장 작은 원자 기능

예:

* run_command
* read_file
* write_file
* restart_service
* query_metric
* fetch_log
* open_ticket

## 8.2 Skill

재사용 가능한 단일 능력

예:

* probe_linux_host
* collect_web_latency_facts
* check_tls_cert
* analyze_wazuh_alert_burst
* validate_db_connection_pool
* monitor_disk_growth
* summarize_incident_timeline

Skill은 “할 수 있는 능력”이다.

## 8.3 Playbook

여러 skill/tool을 조합한 절차 단위

예:

* onboard_new_linux_server
* diagnose_web_latency
* patch_linux_wave
* investigate_intrusion_signs
* tune_siem_noise
* monitor_siem_and_raise_incident
* nightly_health_baseline_check

Playbook은 “능력을 엮은 운영 절차”다.

## 8.4 Policy

실행 가능 범위와 승인 규칙

## 8.5 Experience

과거 작업에서 추출한 재사용 지식

## 8.6 결론

OldClaw는 `Tool < Skill < Playbook` 구조로 간다.

즉,

* Tool = primitive
* Skill = reusable capability
* Playbook = orchestrated procedure

이 분리는 끝까지 유지한다.

## 9. 확장성 원칙: 새로운 업무는 어떻게 추가되는가

OldClaw는 새로운 업무가 들어올 때마다 코어를 뜯는 시스템이 되면 안 된다.

이상적인 확장 방식은 아래다.

### 경우 1. 기존 도구와 skill만으로 충분

* 새 playbook만 추가

### 경우 2. 기존 tool은 충분하지만 새 능력이 필요

* skill 추가
* playbook 추가

### 경우 3. 정말 새로운 원자 기능이 필요

* tool 추가
* 그 tool을 쓰는 skill 추가
* playbook 추가

즉 대부분의 신규 업무는 **skill + playbook** 수준에서 끝나야 한다.

### 예시

“SIEM 상시 모니터링”이라는 새 업무가 생기면:

* `consume_wazuh_alerts` skill
* `correlate_alert_burst` skill
* `raise_incident_if_threshold` skill
* `monitor_siem_and_raise_incident` playbook

만 추가하면 된다.

코어 상태기계, project lifecycle, evidence pipeline은 수정하지 않는다.

## 10. 업무 유형 분류

OldClaw는 업무를 한 종류로 보지 않는다.

## 10.1 One-shot 작업

한 번 실행하고 종료

예:

* 온보딩
* 인증서 갱신
* 패치
* 폐기
* 장애 보고서 생성

## 10.2 Batch 작업

주기 또는 묶음 단위로 실행

예:

* 매일 아침 health 점검
* 주간 patch candidate 수집
* 월간 인증서 만료 점검
* 분기별 계정/권한 검토
* 주기적 백업 복구 테스트

## 10.3 Continuous 작업

계속 감시/평가/승격

예:

* SIEM 상시 모니터링
* 로그 이상 탐지
* drift detection
* 리소스 사용량 감시
* 장애 징후 사전 감시

즉 OldClaw는 단순 사건 대응 시스템이 아니라,
**one-shot + batch + continuous** 세 업무유형을 모두 다루는 운영 플랫폼으로 설계해야 한다.

## 12. 저장 전략

OldClaw는 **모든 대화와 실행이 저장되고 추적 가능해야 한다.**
따라서 저장 전략은 필수 축이다.

## 12.1 PostgreSQL

구조화 저장

* assets
* targets
* projects
* jobs
* approvals
* messages
* audit log
* playbook runs
* validation results
* task summaries

## 12.2 Blob/Object Store

원본 보관

* stdout/stderr
* full reports
* generated scripts
* diff files
* exported evidence packs

## 12.3 Vector DB

의미 검색

* 경험 패턴
* 과거 보고서 요약
* remediation 지식
* 조직 특화 운영 관행
* playbook 문서

## 13. 히스토리 관리 전략

이건 OldClaw에서 매우 중요한 축이다.

### 핵심 원칙

OldClaw는 “모든 과거를 context에 다 넣는 시스템”이 아니다.
대신 아래 4층 기억 구조로 간다.

## 13.1 Raw History

원본 로그를 절대 버리지 않는다.

저장 대상:

* user 요청
* manager/master/subagent 메시지
* 실행 명령
* stdout/stderr
* evidence
* approval
* report

용도:

* 감사
* 검색
* 사후 분석

## 13.2 Structured Task Memory

프로젝트 종료 시 자동으로 구조화 요약을 만든다.

포함:

* 목표
* 관련 asset
* 사용 playbook / skill
* 핵심 발견사항
* 실제 조치
* 검증 결과
* 최종 상태
* lesson learned

즉 raw history를 사람이/모델이 재사용 가능한 task memory로 압축한다.

## 13.3 Semantic Experience Memory

정말 재사용 가치가 있는 것만 경험으로 승격한다.

예:

* 특정 오류 패턴의 성공적 remediation
* 특정 자산군의 주의점
* 특정 승인 규칙 precedent
* 환경 특화 workaround

모든 기록이 경험이 되는 것이 아니라, **재사용 가치 있는 것만 승격**한다.

## 13.4 Working Context

현재 작업에 필요한 기억만 retrieval해서 context에 넣는다.

예:

* 관련 asset 최근 작업 3건
* 같은 playbook 성공/실패 5건
* 유사 오류 패턴 3건
* 관련 approval precedent
* 조직 정책

### 결론

OldClaw는 **history를 저장하고, 구조화하고, 경험으로 승격하고, 현재 작업에 필요한 것만 smart retrieval로 꺼내 쓴다.**

## 14. retrieval 전략

단순 vector search 하나로는 부족하다.
다음 축으로 검색한다.

* Asset 축: 이 자산의 과거 작업
* Playbook 축: 이 절차의 과거 성공/실패
* Error 축: 이 오류/증상과 유사한 과거 사례
* Change 축: 최근 변경 이후 발생한 문제
* Approval 축: 이 정도 위험의 선례
* Org 축: 우리 조직에서 통했던 방식

## 15. hallucination 방지와 완료 판정

이 프로젝트의 핵심 문제 중 하나는 “가짜로 작업하고 진짜처럼 보고하는 현상”이다.

## 15.1 완료 인정 조건

완료로 인정되려면 아래가 필요하다.

* 실행 명령 또는 tool action 존재
* 대상 asset 식별 가능
* stdout/stderr 존재
* exit code 존재
* 검증 명령 또는 테스트 존재
* 테스트 결과 기록
* validation 통과
* 필요 시 Master review 통과

## 15.2 완료로 인정하지 않는 경우

* 실행 증빙 없음
* stdout/stderr 없음
* 단순 추론만 있고 실제 실행 없음
* 검증 없이 “잘 됐다”고 주장
* 검증이 실행 결과 재진술 수준
* 사람이 중요한 승인 없이 위험행위를 했음

## 15.3 Master 검수

Manager는 작업을 끝냈다고 보고할 때 아래를 정리해 제출한다.

* 목표
* 사용한 도구와 이유
* 실제 실행 절차
* 핵심 명령
* 주요 stdout/stderr
* 테스트 케이스
* 테스트 결과
* 남은 위험
* 완료 판정 근거

Master는 이를 검토해 반려/승인한다.

# 30. 최종 판단

OldClaw의 가장 올바른 정의는 다음이다.

> **OldClaw는 pi의 실행 런타임을 재사용하되, 그 위에 Master–Manager–SubAgent, Asset-first 운영, Evidence/Validation, Skill/Playbook 확장성, Batch/Continuous 실행, History/Experience retrieval를 얹은 정보시스템 운영 전문 에이전트 플랫폼이다.**

즉 이 프로젝트는:

* pi를 버리는 프로젝트도 아니고
* pi를 이름만 바꾸는 프로젝트도 아니며
* 임시 stub를 계속 붙이는 프로젝트도 아니다.

정확히는:

> **pi를 embedded executor로 사용하는 운영 오케스트레이션 시스템**이다.

원하면 다음 단계로 바로 이어서
**repo 구조, 서비스 디렉터리 구조, DB 스키마, skill/playbook registry 명세**까지 상세 설계로 내려가겠다.
