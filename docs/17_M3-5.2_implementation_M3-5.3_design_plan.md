## OpsClaw M3-5.2 완료보고서 (PhaseA: master-guided probe loop for missing/ambiguous inputs)

작성일: 2026-03-06
브랜치: `main` (이전 wip에서 리베이스 후 main 반영)
범위: **run_auto 입력 해결(choices/missing → human 또는 probe) + audit/evidence/state 정착 + remote subagent(host network) 운용 가이드**

---

### 1) 목표(Goal)

M3-5.2 PhaseA의 목표는 **LLM이 “추측”하지 않고**, 실행에 필요한 정보가 부족/모호할 때

* (1) **먼저 안전한 probe**로 확인을 시도하고
* (2) 그래도 확정이 안 되면 **최소 질문만 human에게** 던지며
* (3) 결정된 입력값/실행 결과를 **state/audit/evidence에 재현 가능하게 남기는 것**

을 “돌아가게” 만드는 것이다.

---

### 2) 구현 결과 요약(What’s Done)

#### ✅ A. `run_auto` 흐름 정착

`POST /projects/{project_id}/run_auto`가 아래 흐름으로 동작한다.

1. 프로젝트 로드
2. plan 없으면 planner로 생성
3. playbook/target 확정(override 허용)
4. `input_resolver.resolve_inputs()`로 **missing/choices 계산 + discover 실행**
5. missing이면:

   * 가능한 건 자동 확정(후보 1개)
   * 아니면 choices + 질문 반환(needs_clarification)
6. 입력이 확보되면 `playbook_runner.run_playbook()` 실행
7. 결과를 `playbook_runs`에 저장 + audit 기록

---

#### ✅ B. “입력값 자동 발견(discover) → 최소 질문” 패턴 도입

Playbook YAML에 `inputs:` 섹션을 정의하면,

* 각 입력 키에 대해 discover commands를 실행해 후보를 뽑고
* exclude 필터로 의미 없는 후보를 제거하고
* 후보가 1개면 자동 선택, 여러 개면 choices 반환

즉, “**입력값을 human이 무조건 타이핑**”하는 구조가 아니라,
**시스템이 먼저 후보를 만들어** human의 결정을 최소화하도록 함.

---

#### ✅ C. remote 타깃 subagent 운영 방식 정리(Host network)

remote-1 타깃에서 ens33/ens34 같은 **호스트 NIC를 보려면**
remote 서버의 subagent 컨테이너가 **host network로 떠야 함**을 확인했고,
그에 맞춘 운영 절차를 정리했다.

---

#### ✅ D. 증빙(재현성) 확보: state + audit + evidence

* `/data/state`:

  * 프로젝트 state에 plan, 선택된 playbook/target, 입력값 상태 저장
  * `playbook_runs[-1].inputs`에 최종 입력 저장
* `/data/audit/audit.jsonl`:

  * INPUT_RESOLVE_START/DISCOVER/AUTO_CHOSEN/NEEDS/DONE
  * RUN_AUTO_START/RUN_AUTO_DONE
* `/data/evidence/*.log`:

  * run_script 실행 stdout/stderr 로그 파일

### 3) 완료 조건(Exit Criteria)

아래를 충족하면 M3-5.2 PhaseA 완료로 판단.

1. `run_auto`가 missing_inputs를 감지하면 `needs_clarification`을 반환한다. ✅

### 5) 주요 이슈사항(What Hurt)

#### 이슈 1) `eth0@ifXX` 같은 컨테이너 NIC가 자동 선택됨

* 원인: discover가 `ip -o link show`로 **이름만 나열**했고, 필터가 약해 veth/bridge류가 후보에 남음
* 영향: 잘못된 iface가 플레이북 템플릿에 주입되어 precheck 등에서 실패
* 현재 상태: PhaseA에서는 “choices 반환 + human 선택”으로 회피 가능.
  (PhaseB에서 자동 판별 probe 체인으로 해결 예정)

#### 이슈 2) remote에서 host NIC가 안 보임

* 원인: remote subagent가 docker bridge 네트워크로 떠 있으면 컨테이너 인터페이스만 보임
* 해결: remote subagent를 `--network host`로 기동

#### 이슈 3) run_auto에서 plan/missing 상태 갱신 일관성

* 1차 호출에서 `plan.status = needs_clarification`
* 2차 호출에서 실행은 됐는데 `plan.status`가 그대로 남는 혼동 가능성
* 해결/개선 방향: 실행 성공 시 `plan.status="executed"` + `plan.inputs` 반영을 확정적으로 저장(PhaseB에서 정리)

---

### 6) 설계 변경(정리)

#### 변경된 핵심

* “요청 → 바로 실행”이 아니라,
* “요청 → plan → missing/choices → (probe/human) → 실행 → 증빙”
  로 **운영체제/네트워크/보안 등 범용 작업의 기본 골격**을 확정했다.

#### YAML의 역할(현재)

* “작업을 하드코딩”하려는 목적이 아니라,
* **입력/출력/증빙을 통제**하기 위한 실행 명세(스텝/잡/검증/입력스펙)로 사용 중.

---

### 7) 반성문(요청 반영)

* “suricata만 되면 끝”처럼 보이게 만들었던 부분은 명백히 문제였다.
  특히 NIC 자동판별을 “목록에서 아무거나 고르기” 수준으로 두면, 네가 말한 비전(범용 자연어 제어)에서 바로 깨진다.
* 디버깅 과정에서 사소한 실수(함수 시그니처/저장 구조/plan 갱신)가 반복되어 불필요한 삽질을 유발했다.
  앞으로는 **(1) 함수 시그니처 고정 → (2) 최소 테스트 스크립트 고정 → (3) 변경 후 즉시 재현 테스트** 순으로만 진행해야 한다.
* 결론: PhaseA는 “흐름을 세우는 프로토타입”으로 의미가 있지만,
  PhaseB부터는 **OpenClaw 수준의 probe 기반 자동판별**로 끌어올리지 않으면 비전과 괴리가 생긴다.

## M3-5.3 계획 (OpenClaw급 NIC 자동 판별 + LLM probe loop 강화)

### 1) 하고자 하는 것 요약

* iface_in/iface_out 같은 “환경 의존 입력”을

  * **가능하면 자동으로 확정**
  * 불가능하면 **질문 1~2개로 최소화**
* 단순 필터링이 아니라 **검증 probe 체인**으로 신뢰도를 확보

---

### 2) 기능적/기술적 설명

#### A) NIC Auto-Resolve Probe Chain (non-guess)

입력키가 `iface_*` 류면 아래 probe를 순서대로 실행해서 점수화:

1. 인터페이스 후보 목록(정교):

* `ip -o link show`
* 제외: `lo`, `docker*`, `br-*`, `veth*`, `cni*`, `flannel*`, `kube*`, `virbr*`, `zt*`, `wg*`, `tun*`, `tap*`

2. IP가 붙은 NIC만 우선:

* `ip -o -4 addr show`

3. egress nic 추정(가장 강력):

* `ip route get 1.1.1.1` (또는 조직 환경에 맞는 라우팅 목적지)

  * 결과에서 `dev <iface>` 추출

4. 인라인 in/out sanity:

* in/out이 같은 경우 경고 + 추가 probe:

  * “브릿지 구성/인라인 구조” 관련 질문 or 추가 명령 제안

5. 최종 결정

* 점수 기반으로 1개로 좁혀지면 자동 확정
* 2개 이상이면 “왜 이 후보들인가”를 rationale로 포함해 질문

#### B) “Probe Loop”의 역할 정리

* **input_resolver**: deterministic + 안전한 probe로 최대한 해결
* **LLM probe loop**: deterministic로 해결 못한 경우에만

  * 추가 probe 생성(안전 우선)
  * 결과를 근거로 최종 결정을 하거나 질문 최소화

---

### 3) 완성하면 변경되는 것

* 지금: `choices`를 주고 사람이 고르는 경우가 많음
* M3-5.3 이후:

  * 대부분의 환경에서 **iface_out은 자동 확정**
  * iface_in은 상황에 따라 자동 또는 질문 1개로 축소
  * `eth0@ifXX` 같은 컨테이너 NIC 선택이 원천적으로 줄어듦

---

### 4) 테스트 케이스 / 완료 조건

#### 테스트 케이스

1. remote host network 환경에서:

* 입력 없이 run_auto 호출 → iface_out 자동 확정(ens33 등)
* iface_in/out이 둘 다 확정되면 질문 없이 실행까지 진행

2. docker bridge 환경에서(일부러):

* 후보가 컨테이너 NIC뿐이면

  * “원격 subagent가 host network인지” 질문 1개로 축소

3. 인라인 특수 구성(브릿지 등)에서:

* 자동 확정 실패 시:

  * “트래픽이 들어오는 NIC / 나가는 NIC를 선택” 질문 + 후보 2~4개로 제한

#### 완료 조건

* (1) iface_out 자동 확정 성공률 향상(bridge/veth 제외)
* (2) 실패 시 질문은 최대 1~2개
* (3) 결정 근거(rationale + probe 결과)가 audit/evidence에 남음
