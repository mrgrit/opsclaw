아래는 “완성된 OpsClaw” 기준으로 **사용 케이스 12개를 아주 상세하게** 쓴 버전이야. (각 케이스는 *자연어 목표 → 시스템 흐름(Plan/Probe/Decide/Exec/Validate/Replan) → 도구/OSS 활용(필요 시 discover→clone→학습→실행→랩핑) → human 최소 개입 → 산출물/증빙*까지 포함)

# 사용 케이스 1) 원인 모르는 서비스 지연/장애 자동 복구

## 사용자 목표(자연어)

* “웹이 갑자기 느려졌어. 원인 찾아서 복구해. 재발 방지까지.”

## 시스템 흐름

### Plan

* 서비스 경로(웹→API→DB→캐시) 가정 구성
* 우선순위: “가장 빠른 probe로 원인을 좁히기”
* 위험도 분류: 재시작/설정 변경/스케일 변경은 승인 필요 가능

### Probe (확인)

* 대상 타겟 자동 선택(웹 서버/DB 서버/로드밸런서)
* 읽기 전용 점검:

  * CPU/Load/Mem/Swap, Disk/IO, 네트워크 연결 수, 포트 리슨, 프로세스 상태
  * 최근 로그 tail(nginx/app/db), 에러율/p95 지표(연동 가능 시)
* 병목 후보가 여러 개면 추가 probe:

  * DB slow query, 커넥션 풀 상태, 파일 디스크립터, GC/메모리 누수

### Decide

* 증거 기반 원인 후보 ranking(예: DB 커넥션 고갈 vs 디스크 꽉 참 vs 특정 5xx 증가)
* “즉시 완화”와 “근본 수정”을 분리

### Exec (실행)

* 완화 조치(정책에 따라 자동 또는 승인):

  * 서비스 재시작, worker 증설, 캐시 플러시(승인), 임시 rate limit
* 근본 조치:

  * 설정 수정(PR 생성 가능), 테이블 인덱스 제안, 로그 레벨 조정

### Validate

* 헬스체크 + p95 개선 + 에러율 감소 확인
* 동일 probe 재수행으로 정상 범위 확인

### Replan

* 개선 안 되면 다음 원인 후보로 probe 확장

## 도구/OSS 활용(예시)

* 시스템 도구: `ss`, `journalctl`, `df`, `free`, `top`
* 필요 시 OSS:

  * 성능 프로파일러/벤치 툴을 GitHub에서 탐색→clone→사용법 학습→실행
  * 결과를 JSON으로 랩핑해 비교 가능하게 저장

## human 최소 개입

* “재시작/스케일 변경/캐시 플러시” 같은 고위험 조치만 승인

## 산출물/증빙

* Audit 타임라인(원인 후보→조치→검증 결과)
* Evidence pack(ZIP): probe 결과 로그/스크립트/변경 diff/최종 리포트

# 사용 케이스 2) 신규 서버 5대 온보딩(표준 셋업)

## 사용자 목표

* “신규 서버 5대 기본 셋업(계정/SSH/보안/모니터링/로그) 해줘. 증빙 남겨.”

## 흐름

### Plan

* 타겟 그룹 선택(5대)
* 단계 DAG:

  1. 사용자/키/권한
  2. 패키지 업데이트
  3. 필수 에이전트 설치(모니터링/로그)
  4. 보안 기본 설정
  5. 검증

### Probe

* OS/배포판/패키지 매니저 확인
* 현재 사용자/SSH 설정/방화벽 상태 확인

### Decide

* OS별 분기(ubuntu/rhel)
* 사전 충돌 해결(이미 계정 있음, 포트 변경됨 등)

### Exec

* 계정 생성, SSH hardening, 기본 패키지 설치, 에이전트 설정
* 실패한 노드만 재시도

### Validate

* SSH 접속/포트/서비스 상태 확인
* 에이전트 heartbeat 확인

## OSS 활용

* 기존 도구 없으면:

  * 모니터링/로그 수집 OSS를 검색→도입→설치 자동화
* 필요 시 wrapper로 표준화(“node_id→ok/metrics”)

## human 개입

* “기존 운영 정책과 충돌하는 설정”은 선택/승인

## 산출물

* 서버별 결과 요약 + 증빙 ZIP

---

# 사용 케이스 3) 전체 보안 점검 + 보고서 자동 생성(OSS 도입 포함)

## 사용자 목표

* “전체 서버 취약 설정/열린 포트/취약 패키지 점검해서 보고서 만들어.”

## 흐름

### Plan

* 스캔 범위 정의(타겟/포트/설정 파일)
* OSS 스캐너 후보가 필요하면 tool lifecycle 실행

### Probe

* 서버별 열린 포트, ssh 설정, sudoers, 패키지 목록 수집
* 필요 시 특정 CVE 체크를 위한 도구 탐색

### Tool Lifecycle(필요 시)

* GitHub에서 스캐너 후보 검색
* clone/빌드/실행
* 결과 포맷이 제각각이면 wrapper(API) 생성해서 JSON 통일

### Decide

* 위험도 분류, 우선순위(즉시 수정 vs 계획 수정)

### Exec(선택)

* 자동 수정은 정책/승인 하에만
* 수정안(PR/패치) 생성 중심

### Validate

* 재스캔으로 개선 확인

## 산출물

* 리포트(요약/근거/권고)
* evidence ZIP(스캐너 버전/커밋/명령/결과 포함)

# 사용 케이스 4) “네트워크 구조 파악”과 in/out 인터페이스 결정(LLM probe 기반)

## 사용자 목표

* “이 서버에서 인라인(in/out) 인터페이스를 정확히 판별해.”

## 흐름

### Plan

* 필요한 unknown: `iface_in`, `iface_out`, gateway, route, NAT 여부

### Probe (LLM이 직접 설계)

* `ip -j link show`, `ip -j addr`, `ip -j route`
* default route와 실제 트래픽 흐름 확인
* 필요 시 “실제 패킷 관찰” probe(짧은 tcpdump) 제안(정책 허용 시)

### Decide

* 증거 기반으로 in/out 지정
* 애매하면 “후보 2개 중 선택”만 human에게 질문

### Validate

* 선택된 인터페이스로 `ip link` 확인
* (가능하면) 간단한 통신 테스트로 경로 검증

## 산출물

* 결론 + 근거(어떤 probe로 결정했는지)
* 재현 가능한 명령 묶음(스크립트)

# 사용 케이스 5) IDS/로그 파이프라인 장애 추적(“어디서 끊겼는지”)

## 사용자 목표

* “Suricata 로그가 SIEM으로 안 올라가. 어디서 끊겼는지 찾고 복구해.”

## 흐름

### Plan

* 파이프라인 구성: suricata → file/agent → forwarder → collector → SIEM
* 각 구간 health probe 정의

### Probe

* suricata 서비스/로그 파일 생성 여부
* forwarder 상태/queue backlog
* collector 수신 여부(포트/로그)
* SIEM API/수집 상태

### Decide

* 장애 구간 pinpoint
* config mismatch vs permission vs disk full vs network block 판단

### Exec

* 설정/서비스 재시작(승인)
* 누락된 파일/권한 수정
* 재전송 트리거(가능 시)

### Validate

* end-to-end 이벤트 흐름 확인

## 산출물

* 장애 원인/조치/재발 방지
* evidence ZIP(각 구간 상태 증빙)

---

# 사용 케이스 6) 방화벽 정책 변경(승인 게이트 + 자동 검증 + 롤백)

## 사용자 목표

* “API 서버는 443만 열고 나머지는 차단. 변경 후 접속 테스트까지.”

## 흐름

### Plan

* 위험도 high → 승인 필요 자동 적용
* 롤백 플랜 포함(변경 전 정책 저장)

### Probe

* 현재 규칙/열린 포트/서비스 의존성 확인
* 변경 시 영향을 받을 내부 통신 점검

### Decide

* 최종 규칙안 생성 + 영향 분석 요약

### Exec(승인 후)

* 정책 적용
* 즉시 smoke test 실행(내부/외부에서 curl)

### Validate

* 허용된 포트 OK, 차단된 포트 BLOCK 확인
* 실패 시 자동 롤백

## 산출물

* 변경 내역(diff) + 테스트 결과 + 롤백 증빙

---

# 사용 케이스 7) “도구가 없는 문제”를 OSS로 해결(PCAP/포렌식/성능 등)

## 사용자 목표

* “PCAP에서 특정 패턴 트래픽만 추출해 통계 내줘. 도구 없으면 알아서 찾아 써.”

## 흐름

### Plan

* 필요한 기능 정의: filter, extract, aggregate
* OSS 후보 탐색 필요

### Tool Lifecycle

* GitHub/웹 검색으로 후보 수집
* clone/install
* README/--help 읽고 사용법 구성
* 샘플 실행으로 검증
* 결과를 JSON 포맷으로 wrapper 생성(필요 시)

### Exec

* 대상 PCAP 처리
* 결과 저장 + 리포트 생성

### Validate

* 작은 샘플로 교차 검증

## 산출물

* 사용한 도구 버전/커밋/명령 기록
* 결과 JSON + 요약 리포트

---

# 사용 케이스 8) DB 백업/복구 리허설(증빙 중심)

## 사용자 목표

* “백업 잘 되는지 확인하고, 복구 리허설까지 수행해서 증빙 남겨.”

## 흐름

### Plan

* 백업 체크 → 복구 환경 준비 → 복구 실행 → 무결성 검증
* 위험도 중간(운영 DB 직접 복구는 승인)

### Probe

* 백업 파일 존재/최근성/크기/암호화 확인
* DB 상태/버전 확인

### Exec

* 격리된 환경에 복구 실행
* 무결성 체크(테이블 수/샘플 쿼리)

### Validate

* 서비스 레벨 검증(쿼리 정상)

## 산출물

* 백업/복구 로그 + 체크리스트 + evidence ZIP

# 사용 케이스 9) IAM/권한 감사 + 최소 권한 정리(승인 워크플로)

## 사용자 목표

* “권한 과다 계정 찾고 정리안 만들고, 승인 후 적용.”

## 흐름

### Probe

* 사용자/그룹/권한 덤프 수집
* 위험 권한 패턴 탐지

### Decide

* 과다 권한 후보 리스트 + 근거
* 적용 시 영향도(서비스 계정 포함 여부)

### Exec(승인 후)

* 권한 조정 적용
* 롤백 포인트 저장

### Validate

* 정상 업무 영향 없는지 테스트

## 산출물

* 변경 전/후 권한 diff + 승인 기록 + 증빙

# 사용 케이스 10) 성능 튜닝(가설→실험→검증 반복)

## 사용자 목표

* “API p95가 튀어. 원인 찾고 개선해. 개선 전/후 증빙 남겨.”

## 흐름

### Probe

* 지표/로그/시스템 리소스 동시 수집
* 병목 후보(쿼리/락/GC/IO)별 추가 probe

### Decide

* 가설 2~3개 생성
* 실험 우선순위 결정

### Exec

* 설정 변경/캐시/리소스 조정(승인)
* 필요 도구(벤치/프로파일러) OSS로 도입

### Validate

* 동일 조건 벤치로 전/후 비교
* 회귀 위험 평가

## 산출물

* 전/후 그래프/로그/벤치 결과 + evidence ZIP

# 사용 케이스 11) 보안 이벤트 대응(조사 자동화 + 위험 조치는 승인)

## 사용자 목표

* “이상 트래픽. 감염 조사하고 필요한 조치 실행. 증빙 남겨.”

## 흐름

### Probe

* 의심 프로세스/연결/로그/무결성 체크
* IOC 기반 탐색(파일/레지스트리/네트워크)

### Decide

* 감염 가능성 판단 + 권고 조치(격리/차단/계정 잠금 등)

### Exec

* 조사/수집은 자동
* 격리/차단은 승인 후 실행

### Validate

* 의심 행위 중단/재발 없음 확인

## 산출물

* 타임라인(탐지→조사→조치→검증) + 증거 파일

# 사용 케이스 12) 주간 운영 리포트 자동 생성(감사/증빙 기반)

## 사용자 목표

* “이번 주 장애/변경/조치/승인 내역을 증빙과 함께 보고서로.”

## 흐름

### Probe

* 프로젝트 audit/evidence/index 수집
* 중요 이벤트 분류(실패→복구, 승인→변경)

### Decide

* 요약/핵심 리스크/개선사항 도출

### Exec

* 리포트 생성(마크다운/PDF)
* evidence pack 링크/첨부

## 산출물

* 운영 리포트 + 관련 evidence ZIP 모음
