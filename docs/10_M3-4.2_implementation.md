# OpsClaw M3-4.2 완료 보고서 (Playbook Target Routing)

작성일: 2026-03-04
범위: **Playbook 조회/실행 API 추가 + target_id 기반 Playbook 실행 라우팅 + 프로젝트 state 누적 저장**

---

## 1) 목표(Goal)

M3-4.2의 목표는 다음을 **실제로 동작**하게 만드는 것이다.

1. Manager API가 **Playbook 목록/단건 조회**를 제공한다.
2. Manager API가 **Playbook 실행 엔드포인트**를 제공한다.
3. Playbook 실행 시 요청의 `target_id`를 이용해 해당 Target의 `base_url`로 **SubAgent 라우팅**하여 step을 실행한다.
4. 실행 결과(step별 run 결과, stdout/stderr, evidence_refs, validate 결과)를 **프로젝트 state에 누적 저장**한다.
5. 로컬(`local-agent-1`)과 원격(`remote-1`)에서 모두 **동일 Playbook 실행이 성공**해야 한다.

---

## 2) 구현 결과 요약(What’s Done)

### ✅ A. Playbook API 추가 (Manager)

`api/main.py`에 Playbook 관련 엔드포인트 추가

* `GET /playbooks` : 플레이북 목록 조회
* `GET /playbooks/{playbook_id}` : 플레이북 단건 조회
* `POST /projects/{project_id}/run_playbook` : 프로젝트 컨텍스트에서 플레이북 실행

요청 모델(예: `RunPlaybookReq`)

* `playbook_id: str`
* `inputs: dict = {}`
* `target_id: str = "local-agent-1"`
* `timeout_s: int = 60`
* `approval_required: bool = False` *(현재 runner에서 미사용/무시)*

---

### ✅ B. target_id 기반 Playbook 실행 라우팅

Playbook 실행 요청이 들어오면:

1. `resolve_subagent_url(req.target_id)` 로 subagent_url 결정

* `local-agent-1` → 로컬 subagent
* `remote-1` → Target Registry에서 base_url resolve 후 원격 subagent

2. `playbook_runner.run_playbook(...)` 실행

* `project_id`, `audit_dir`를 runner 요구 시그니처에 맞게 전달
* runner가 각 step의 command를 `POST {subagent_url}/a2a/run_script`로 실행

---

### ✅ C. 프로젝트 state에 playbook_runs 누적 저장

Playbook 실행 결과는 프로젝트 state 파일에 누적 저장:

* 키: `playbook_runs[]`
* 저장 내용(최소):

  * `playbook_id`
  * `target_id`
  * `result` (runner가 반환한 runs + validate 등)
  * `created_at`

---

### ✅ D. 로컬/원격 모두 실행 성공 + 증빙(evidence) 생성 확인

* 로컬(`local-agent-1`)에서 ops-health-basic 실행 성공
* 원격(`remote-1`)에서도 동일 실행 성공
* 원격 SubAgent 서버의 `/data/evidence`에 step 실행 로그 파일들이 생성됨

---

## 3) 완료 조건(Exit Criteria)

아래 조건을 모두 만족하면 M3-4.2 완료로 판단한다.

1. `GET /playbooks`가 200으로 응답하고 플레이북 목록을 반환한다.
2. `GET /playbooks/{playbook_id}`가 존재하는 플레이북에 대해 200으로 응답한다.
3. `POST /projects/{project_id}/run_playbook` 호출 시:

   * 지정한 `target_id`의 SubAgent에서 step 실행이 수행되고
   * 응답에 step별 실행 결과(`exit_code/stdout/stderr/evidence_refs`)와 `validate` 결과가 포함된다.
4. 실행 결과가 프로젝트 state의 `playbook_runs[]`에 누적 저장된다.
5. 원격 target(`remote-1`) 실행 시 원격 서버의 `/data/evidence`에 로그 파일이 생성된다.

➡️ 위 조건 모두 충족됨.

## 5) 주요 이슈 / 트러블슈팅 (What Hurt)

### 이슈 1) `ModuleNotFoundError: No module named 'opsclaw'`

**원인**

* 컨테이너 내부는 `/app/*.py` 형태로 모듈이 배치되어 있는데, `from opsclaw.api import ...` 같은 패키지 import를 사용함.

**대응**

* `api/main.py` import를 로컬 모듈 기준으로 수정

  * `import playbook_store`
  * `import state_store`
  * `import audit_store`
  * `from workflows import playbook_runner`

---

### 이슈 2) runner 시그니처 불일치 (`approval_required` unexpected)

**증상**

* `TypeError: run_playbook() got an unexpected keyword argument 'approval_required'`

**원인**

* `workflows/playbook_runner.py`의 `run_playbook` 함수가 해당 인자를 받지 않음.

**대응**

* runner 호출에서 `approval_required=` 전달 제거(요청 모델은 유지하되 현재는 무시)

---

### 이슈 3) runner 필수 인자 누락 (`audit_dir`, `project_id`)

**증상**

* `TypeError: run_playbook() missing 2 required keyword-only arguments: 'audit_dir' and 'project_id'`

**원인**

* runner가 감사 기록용 인자를 필수로 요구.

**대응**

* 호출부에 `project_id=project_id`, `audit_dir=AUDIT_DIR` 추가

---

### 이슈 4) state 저장 함수 NameError (`load_project` 미정의)

**증상**

* `NameError: name 'load_project' is not defined`

**원인**

* 새로 추가한 `append_playbook_run`이 코드베이스에 없는 `load_project/save_project` 패턴을 사용함.

**대응**

* 기존 패턴(`load_json/project_path/update_project`)으로 `append_playbook_run(STATE_DIR, project_id, entry)` 재작성

---

### 이슈 5) jq parse error (Invalid numeric literal)

**원인**

* 서버가 500을 내며 `Internal Server Error`(text/plain)를 반환했고 jq가 JSON 파싱 시도.

**대응**

* 먼저 `curl -i`로 원문 확인 후, 서버 로그(`docker compose logs api`)로 실제 예외 확인
