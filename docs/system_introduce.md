# OpsClaw

## 한 줄 소개

**자율시스템 지향, 자연어로 서버·클라우드·네트워크·보안·엔드포인트의 모든 작업을 수행하는 멀티 AI 에이전트 오케스트레이터.**

## 두 줄 소개

사용자가 목표를 제시하면, OpsClaw는 필요한 사항을 최소 질의하고 계획을 수립한 뒤 작업을 쪼개어 대상 SubAgent에 배분하고 협업을 조율한다. 세부 계획과 Todo list를 생성해 작업 체계를 구축하고 오케스트레이션한다.
상황에 맞는 동적 프롬프트를 생성하고 쉘 스크립트를 작성하거나 Tool/OSS를 도입해 실행하며, 결과(Evidence)를 근거로 다음 스텝을 결정해 목표를 끝까지 완수한다.

## 세 줄 소개

경량 오픈 모델 오케스트레이션으로 폐쇄망 운용이 가능하며, 어려운 문제는 안전하게 상용 모델(Master)과 필요할 때 최적으로 소통해 해결한다.
LLM이 최대한 자율로 작업하되 폭주 방지를 위해 Playbook 기반 운영과 승인 게이트를 유지하고, 마일스톤별 테스트케이스·보고서·상위 모델 검수를 통해 품질을 보장한다.
LLM이 생성·관리하는 모듈형 **Playbook / Tool / Skill / Experience**를 통해 작업을 구조화하고, 누적 지식베이스와 RL/밴딧 기반 최적화로 궁극적 자율화를 달성한다.

# 핵심 철학

## Evidence-first (추측 금지)

OpsClaw는 “그럴 것이다”를 말하지 않는다. 필요한 사실은 **probe(관측 명령)**로 직접 확인하고, 결과를 **audit/evidence**로 남긴 후 판단한다.

## Human 최소 개입, Critical만 Confirm

* 자동으로 결론 낼 수 있는 사실은 자동(AUTO)
* 후보는 만들 수 있지만 중요한 결정은 사용자 확인(CONFIRM)
* 사용자만 아는 정책/의도는 질문(ASK)
* 위험한 실행은 승인(NEEDS_APPROVAL)

즉, 사람은 정말 필요한 결정만 한다. 나머지는 에이전트가 확인·정리·실행·검증을 반복한다.

# 구조

## 1) 모델 연계 에이전트: Master / Manager / SubAgent

### Master

* 사용자가 지정한 상용 모델(가장 스마트)
* 비용/보안 이유로 모든 작업을 직접 수행하지 않는다.
* Manager가 요청할 때만 “필요한 질문에 최적으로 응답”한다.
  (최소화가 목적이 아니라, **필요할 때 최적으로 쓰는 구조**)

### Manager

* 중심 에이전트(컨트롤 플레인)
* Master·SubAgent와 소통하며 계획·판단·오류 해결을 담당한다.
* 필요한 스크립트를 작성하고, 지속되는 오류는 Master에 질의한다.
* 작업 완료 시 Master에게 검수(Review)를 요청한다.
* 작업 컨텍스트 유지, 전체 히스토리 관리, 사용자의 질문에 응답한다.

### SubAgent

* 작업 대상 시스템(서버/엔드포인트)에 설치되는 실행 에이전트(데이터 플레인)
* Manager의 지시에 따라 스크립트를 실행하고 결과·증거를 보고한다.
* 실행 컨텍스트를 유지하며, 의문사항/선택 필요사항은 Manager에 질의한다.
* 설치가 어려운 환경(예: 얼라이언스 시스템)은 SSH 등으로 Manager가 직접 수행할 수 있다.

## 2) OpsClaw 중심뼈대 : LangGraph + A2A가

OpsClaw의 코어는 “에이전트가 알아서 일한다”가 아니라 **일이 실제로 끝날 때까지 안전하게 반복 실행되는 구조**다. 그 뼈대를 만드는 두 축이 **LangGraph(상태기계/워크플로 엔진)**와 **A2A(Agent-to-Agent 분산 실행)**다.

### LangGraph: 자율 루프를 상태기계로 고정한다

* **Plan → Probe → Resolve → Execute → Validate → Replan(Loop) → Report**
* 각 단계는 입력/출력 계약을 갖고, 실패 시 어디로 돌아갈지(재-probe/재-resolve/재-exec)가 규칙으로 정해진다.
* 폭주 방지(승인 게이트/정책), 실패 복구, 재현성(동일 그래프로 재실행)이 모두 여기서 나온다.
* 결과적으로 “프롬프트 잘 짜면 된다”가 아니라 **시스템이 항상 같은 방식으로 움직이며** audit/evidence가 축적된다.

### A2A: 실행을 대상 시스템으로 분산하고 협업을 가능하게 한다

* Manager가 만든 실행 단위를 SubAgent(대상 시스템)로 내려보낸다.
* SubAgent는 스크립트/Tool/OSS 실행 결과를 **evidence와 함께** Manager에 보고한다.
* 다수 타겟(서버 5대 온보딩, 파이프라인 장애 추적, 클라우드 리소스 생성/검증 등)을 병렬/순차로 조율할 수 있다.
* 설치가 어려운 환경은 Manager가 SSH로 직접 수행하는 예외 경로도 동일한 실행 계약으로 흡수 가능하다.

### 결론

* LangGraph는 “무슨 순서로/실패 시 어디로 돌아가서/언제 멈추는지”를 고정하고
* A2A는 “그 일을 실제로 어디서 실행하고 증빙을 가져오는지”를 담당한다.
  이 조합으로 OpsClaw는 단순 챗봇이 아니라 **범용 IT 업무 오케스트레이션 자율 시스템**이 된다.

## 3) 장착 모듈: Tool / Skill / Experience

### Tool

* 시스템 내부 명령어 및 즉시 생성한 쉘 스크립트
* GitHub 등에서 다운로드해 설치·실행하는 OSS
* “지금 이 환경에서 실제로 실행 가능한 단위”

### Skill

* 입력/출력 계약, 정책, 검증, 증빙을 포함하는 재사용 단위 → **검증된 실행 패키지**
* 도메인/작업 종류별 기본 정의를 YAML 형태로 정형화한다.
* 필요한 Skill이 없다면, 사용자와 소통해 LLM이 기존 양식을 참고하여 작성하고 검증 절차까지 포함해 등록한다.

### Experience

* 작업 중 사용자의 선택, 시행착오, 과정과 결과를 종합해 “경험”으로 만든다.
* 누적될수록 사용자에게 질문이 줄고, LLM이 더 자율적으로 결정할 수 있는 기반이 된다.
* 장기적으로 RL/밴딧을 통해 “사용자 결정을 예측하고 일치하면 보상”하는 방식으로 질문/후보/전략 선택을 최적화한다. (보조 메커니즘 병행 가능)

# 실행 흐름

## 표준 사이클

1. Plan: 목표를 작업 단위로 분해하고 Todo/DAG 생성
2. Probe: 필요한 unknown을 최소 명령으로 확인(sys/svc/net/fs 등)
3. Resolve: 입력/정책/검증조건을 만족하도록 AUTO/CONFIRM/ASK/APPROVAL 처리
4. Execute: 스크립트/Tool/OSS 실행(SubAgent 분산)
5. Validate: 동일 probe 재수행 및 규칙으로 성공/실패 판정
6. Replan: 실패 시 원인 후보 갱신 후 다음 시도 자동 수행
7. Report: audit/evidence 기반 보고서·증빙팩 생성

## 폭주 방지(안전장치)

* Playbook 기반 IR(입출력/정책/검증/증빙)을 최소한으로 유지
* 위험 실행(재시작, 방화벽 변경, 계정 잠금, 삭제 등)은 승인 게이트 적용
* validation은 “구라 방지”가 최우선(명령 실패는 실패로 기록)

---

# 폐쇄망 + 상용 모델 연계

* 기본 운용: 폐쇄망에서 오픈 모델(경량 모델)로 Manager/SubAgent 오케스트레이션 수행
* 어려운 문제: 정책에 따라 Master(상용 모델)에 최소 질의

  * 개인정보/기밀 필터링(거버넌스 레이어) 후 요청
  * 필요할 때 최적으로 사용(비용·보안 최적화)

# 산출물(Evidence / Audit)

OpsClaw는 작업 결과뿐 아니라 “왜 그렇게 했는지”를 남긴다.

* Audit 타임라인: 계획 → probe → 결정 → 실행 → 검증 → 재계획
* Evidence 파일: stdout/stderr, 로그, diff, 스냅샷, 스캐너 결과
* Evidence Pack(ZIP): 제출 가능한 형태로 자동 묶음
* Master Review 기록: 상위 모델 검수 결과 및 코멘트(선택)

# User Case 상세 (보존)

## 1) 원인 모르는 서비스 지연/장애 자동 복구

**목표:** “웹이 갑자기 느려졌어. 원인 찾아서 복구해. 재발 방지까지.”
**흐름:** Planner가 경로(웹→API→DB→캐시) 가정 → `svc/sys/fs`(필요 시 `net`) probe → 병목 후보 ranking → 위험 조치(재시작/스케일/flush)는 승인 → 동일 probe로 검증 → 실패 시 replan
**산출물:** 장애 타임라인, 원인 후보/조치/검증 로그, evidence ZIP

## 2) 신규 서버 5대 온보딩

**목표:** “서버 5대 기본 셋업(계정/SSH/보안/모니터링/로그) 해줘. 증빙 남겨.”
**흐름:** 대상 선택 → `sys.probe`로 OS/패키지/서비스 매니저 확인 → SSH/계정/에이전트 확인 → 공통 DAG 실행 → 실패 노드 재시도 → 충돌은 human 질문
**산출물:** 서버별 성공/실패 표, 설정 diff, heartbeat/포트 증빙

## 3) 전체 서버 보안 점검 및 보고서 생성

**목표:** “취약 설정/열린 포트/취약 패키지 점검해서 보고서 만들어.”
**흐름:** inventory 수집 → `sys/svc/net/fs` facts → OSS 스캐너 도입·실행 → 결과 JSON 통일 → 위험도 분류 → 수정은 PR/패치 중심
**산출물:** 점검 보고서, 리스크 목록, 스캐너 증빙팩

## 4) 네트워크 구조 파악과 in/out 인터페이스 결정

**목표:** “인라인 in/out 인터페이스 정확히 판별해.”
**흐름:** `net.probe`로 NIC/ADDR/ROUTE/GW/DNS/Reachability → 후보 생성 → 애매하면 사용자 2지선다 → 결론 Experience 저장
**산출물:** 후보/근거, 최종 결정, 재현 가능한 probe 명령 모음

## 5) IDS/로그 파이프라인 장애 추적

**목표:** “Suricata 로그가 SIEM으로 안 올라가. 어디서 끊겼는지 찾아 복구해.”
**흐름:** 파이프라인 구간 모델링 → 구간별 `svc/fs/net` → 끊긴 지점 pinpoint → 수정 조치 제안 → 승인 후 재시작/권한/재전송 → end-to-end 검증
**산출물:** 장애 구간, 전후 검증, 로그 증빙

## 6) 방화벽 정책 변경 + 자동 검증 + 롤백

**목표:** “443만 열고 나머지 차단. 변경 후 테스트까지.”
**흐름:** 현재 규칙/의존 통신/영향 분석 → high-risk 승인 → 변경 전 백업 → 적용 후 smoke test → 실패 시 자동 롤백
**산출물:** diff, 롤백 포인트, 검증 결과

## 7) OSS 도구가 없는 문제 해결

**목표:** “PCAP 특정 패턴 추출 통계. 도구 없으면 알아서 찾아 써.”
**흐름:** 요구 기능 정의 → OSS 탐색/설치 → README/--help 학습 → 샘플 probe → 실제 처리 → 필요 시 wrapper
**산출물:** 도구 버전/커밋, 실행 명령, JSON 결과 + 요약

## 8) DB 백업/복구 리허설

**목표:** “백업 확인하고 복구 리허설까지 증빙 남겨.”
**흐름:** `fs.probe`로 백업 파일 점검 → `sys.probe`로 DB 환경 확인 → 격리 복구 → 무결성 검증 → 운영 영향 시 승인
**산출물:** 백업 증빙, 복구 로그, 무결성 결과

## 9) IAM/권한 감사 + 최소 권한 정리

**목표:** “과다 권한 계정 찾아 정리안 만들고 승인 후 적용.”
**흐름:** 권한 구조 수집 → 위험 패턴 탐지 → 승인 후 변경 → 영향 검증 → rollback 저장
**산출물:** 권한 diff, 승인 로그, 검증 결과

## 10) 성능 튜닝

**목표:** “API p95 튀어. 개선 전/후 증빙까지.”
**흐름:** 지표/로그/리소스 동시 수집 → 가설 생성 → 실험 순서 → 벤치/프로파일러 도입 → 조정 → 전후 비교
**산출물:** 가설 목록, 전후 벤치, 그래프/로그 증빙

## 11) 보안 이벤트 대응

**목표:** “이상 트래픽. 감염 조사하고 조치. 증빙 남겨.”
**흐름:** IOC 조사(파일/프로세스/연결/로그) → 판단 → 조사 자동 / 차단·격리·잠금은 승인 후 → 재발 확인
**산출물:** 타임라인, 증거 파일

## 12) 주간 운영 리포트 생성

**목표:** “이번 주 장애/변경/조치/승인 내역을 증빙과 함께 보고서로.”
**흐름:** audit/state/evidence 수집 → 중요 이벤트 분류 → 요약/개선사항 → Markdown/PDF 생성
**산출물:** 운영 리포트, evidence ZIP 링크/첨부

## 13) 이메일 읽기·요약·분류·답장 초안 생성

**목표:** “메일 요약하고 중요하지 않은 건 답장 초안 만들어.”
**흐름:** 메일 시스템 확인 → 권한/정책 확인 → 메일 수집 → 중요도 분류 → low-risk 초안 생성 → 발송은 정책에 따라 결정
**산출물:** 요약, 분류 결과, 초안 목록

## 14) 이메일 자동 삭제/휴지통 정책 적용

**목표:** “광고성/의미없는 메일 자동 정리.”
**흐름:** 정책 확인(삭제 vs trash) → 샘플 분류 human 확인 → Experience 저장 → 반복적으로 자동화 강화
**산출물:** 정리 대상, 처리 로그, 정책 기록

## 15) 디렉토리 파일 정리

**목표:** “오래된 것/중복/의미없는 것 구분해 정리.”
**흐름:** `fs.probe`로 목록/크기/시간/확장자/중복 후보 → 보존 규칙 최소 질문 → 정리안 → 승인 후 이동/압축/삭제
**산출물:** 전후 목록, 기록, 아카이브 파일

## 16) WAF 룰 생성 및 적용

**목표:** “공격 패턴 막는 WAF 룰 만들어 적용.”
**흐름:** 로그/패턴 수집 → 룰 초안 → detect-only 테스트 → FP 검토 → 승인 후 차단 적용
**산출물:** 룰 diff, 테스트 결과, 적용/롤백 로그

## 17) 서드파티 연동 장애 분석

**목표:** “외부 API 실패 원인(네트워크/인증) 찾아.”
**흐름:** DNS/route/reachability → 토큰 만료 → http probe/로그 → 원인 분류 → 수정 또는 human 확인
**산출물:** 원인 분류, 재현 명령, 수정 후 검증

## 18) 배포 후 헬스체크 및 회귀 탐지

**목표:** “배포했는데 문제 없는지 확인.”
**흐름:** 포트/프로세스/health 확인 → 오류/latency 비교 → baseline 비교 → 이상 시 rollback 제안
**산출물:** 검증 리포트, 회귀 여부, rollback 판단

## 19) 규정 준수/설정 표준화 점검

**목표:** “운영 표준에 맞게 SSH/sudoers/로그/패키지 점검.”
**흐름:** 표준 정책 로드 → `sys/fs/svc` 현상 수집 → diff → 자동수정 vs 승인 필요 분리
**산출물:** 위반 항목, 수정안, 검증 로그

## 20) 클라우드/가상화 자원 운영 자동화

**목표:** “신규 VM 생성 후 기본 셋업 + heartbeat 확인.”
**흐름:** 자원 생성 API/CLI → 접근 확인 → `sys.probe` → 온보딩 DAG → heartbeat 확인
**산출물:** 자원 목록, 셋업 결과, 최종 상태

## 21) 구성 파일 위치 탐색 및 자동 수정 제안

**목표:** “설정 파일 위치 찾고 변경안 제안.”
**흐름:** `fs.probe` 후보 탐색 → 프로세스 인자/서비스 파일 조사 → 최종 파일 결정 → diff 생성 → 적용은 승인 후
**산출물:** 후보/결론/변경 diff

## 22) 감사 대응용 증빙 패키지 자동 생성

**목표:** “감사 제출 가능한 증빙팩으로 묶어줘.”
**흐름:** project state/audit/evidence 수집 → 명령/결과/승인/diff/로그 정리 → ZIP + Markdown/PDF 생성
**산출물:** 제출용 evidence pack, 실행 요약, 감사 타임라인

# 종합

OpsClaw는 **Playbook 중심 실행기에서 출발**해, **Probe/Evidence 기반 의사결정 구조**를 만들고, **Tool / Skill / Experience**로 작업을 구조화하며 확장한다. 장기적으로는 RL/밴딧으로 질문·후보·전략 선택을 최적화해 **완전 자율화**로 나아간다.

핵심은 **LLM이 Tool과 Skill을 활용해 필요한 사실을 직접 확인하고**, 인간은 정말 필요한 결정만 하며, 그 과정 전체가 evidence와 experience로 축적되어 시스템이 계속 진화한다는 점이다.
