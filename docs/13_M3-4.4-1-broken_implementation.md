# OpsClaw M3-4.4 문서화 (Planner v0 + RunAuto + Input Resolver v0)

작성일: 2026-03-05
범위: **M3-4.4 Planner v0(자동 선택) + run_auto(자동 실행) + input_resolver(자동 확인/질문 루프 v0)**
상태: **기능은 동작하나, “LLM probe loop(확인→판단→재계획)”을 아직 엔진 레벨로 완성하지 못함(WIP 성격 포함)**

---

## 1) M3-4.4 구현한 내용 요약

### 1.1 Planner v0 (`/projects/{id}/plan`)

* 프로젝트의 요청 텍스트를 기반으로:

  * 가장 적합한 **playbook_id**
  * 가장 적합한 **target_id(들)**
    를 점수화로 선택.
* 결과를 프로젝트 state에 저장:

  * `plan.status = ready | needs_clarification`
  * `selected_playbook_id`, `selected_target_ids`
  * `rationale`(점수/근거)

**중요:** 이 레포의 프로젝트 구조에서는 `request_text`가 state 최상단이 아니라 `todos[0].detail`로 들어가는 경우가 있어, plan 단계에서 이를 읽도록 수정함.

---

### 1.2 Auto-run (`/projects/{id}/run_auto`)

* 계획(plan)이 없으면 내부에서 plan을 먼저 수행
* plan이 `ready`면:

  * `resolve_subagent_url(target_id)`로 실행 대상 결정
  * playbook 실행 (`run_playbook`) 호출
  * 결과를 `playbook_runs[]`에 누적 저장
* plan이 `needs_clarification`면 질문 반환

---

### 1.3 Input Resolver v0 (`api/input_resolver.py`)

* playbook 실행 전에 **필수 입력값(템플릿 변수 `{{var}}` 및 playbook.inputs 선언)을 추출**
* 누락된 입력값이 있으면:

  * playbook에 `inputs.<key>.discover.commands`가 있으면 이를 실행해 후보를 찾고
  * 후보 1개면 자동 채움
  * 후보 여러 개면 `choices`로 반환(최소 질문)
  * 후보 0개면 질문으로 반환
* 결과:

  * `status: ready`이면 채워진 inputs로 실행
  * `status: needs_clarification`이면 질문/선택지를 반환

> v0에서는 “LLM이 확인 명령을 동적으로 만들고 실행하는 loop”는 아직 구현하지 않음.
> 지금 resolver는 “플레이북에 미리 선언된 discovery command” 기반으로만 작동함.

---

## 2) M3-4.4에서 변경된 동작(완성해서 바뀐 것)

### 이전(M3-4.3까지)

* 사용자가 `playbook_id`, `target_id`, `inputs`를 직접 지정해서 실행해야 안정적으로 돌아감.

### 이후(M3-4.4)

* 사용자는 자연어 요청만 주면:

  * **Planner가 playbook/target 자동 선택**
  * **run_auto가 실행까지 자동 시도**
* 실행에 필요한 입력값이 빠져 있으면:

  * **resolver가 자동 발견하거나**
  * **최소 질문(choices 포함)으로 human에게만 묻고 중단**

## 3) 주요 이슈사항(What Hurt)

### 3.1 코드베이스 시그니처/리턴 형태 불일치로 인한 반복 크래시

* `list_targets()`가 dict 리턴이라고 가정했다가 실제는 list 리턴인 케이스 등
* import 경로/모듈명 충돌(`planner`, `planner_v0`)로 함수 미인식
* FastAPI 500이 text로 반환되면서 jq parse error가 연쇄로 발생

**교훈:** “레포의 실제 반환 타입/시그니처를 확인한 다음” 패치를 제공해야 함.

---

### 3.2 COPY 기반 컨테이너 구조로 인한 playbook/YAML 반영 지연

* 파일을 추가해도 컨테이너에 자동 반영되지 않아, 리빌드 없으면 `/playbooks`에서 안 보임.
* 이로 인해 “playbook not found”가 반복 발생.

---

### 3.3 Input Resolver가 “질문”까지는 되지만 “확정”이 부족

* 예: 네트워크 인터페이스 후보가 이상한 형태로 잡히는 상황(`eth0@if9`)
* 현재 v0 resolver는 “선언된 discovery” 기반이라, 애매하면 human에 질문을 남김
* 그러나 사용자 요구는:

  * LLM이 **추가 확인 명령**을 만들고 실행해서
  * 사람에게는 **정말 최소만** 질문하는 구조임

➡️ 이 지점이 M3-4.4의 핵심 미완성 영역.

---

### 3.4 run_auto 결과/plan/state의 표시 위치 혼란

* 실행 성공(`executed`)인데 plan 필드에 inputs가 없어서 “안 된 것처럼” 보이는 혼란 발생
* 실제로는 `playbook_runs[-1].inputs`에 저장됨

---

## 4) 설계 변경/조정(Design Change)

### 4.1 “YAML이 표준 프로토콜”이 아니라 “입력 포맷 중 하나”로 격하

* M3-4.2~4.3에서 playbook YAML이 중심이었으나,
* 목표(범용 자연어 시스템 제어 AI) 관점에서 YAML 하나로 모든 작업을 표현하는 건 불가능/위험
* **장기적으로 실행 표준은 IR(JSON 중간표현)로 고정**하고,

  * YAML/자연어/LLM-output은 모두 IR로 변환하여 실행하는 구조가 되어야 함.

### 4.2 Input Resolver의 방향 전환

* v0(현재): playbook에 선언된 discovery command 기반
* 목표(사용자 요구): **LLM이 상황에 맞는 probe 명령을 동적으로 생성→실행→결정**
* 따라서 resolver는 “하드코딩/규칙”이 아니라 **Probe→Decide 루프 엔진**이 되어야 함.

## 5) 반성문(필수)

이번 M3-4.4 진행에서 내가 가장 크게 잘못한 점은 세 가지다.

1. **레포의 실제 소스를 충분히 확인하지 않고** “일반적인 FastAPI 패턴”을 전제로 코드를 던져서
   함수 시그니처/리턴 타입 mismatch(`list_targets`, playbook_store 반환 등)를 반복 발생시켰다.
   결과적으로 사용자가 “복붙→빌드→크래시→로그 확인”을 여러 번 반복하게 만들었다.

2. 문제를 해결하는 과정에서, 사용자 목표(“LLM이 확인 명령을 직접 실행해 판단하는 범용 시스템”)보다
   당장의 통과를 위해 “YAML에 discovery 커맨드 박기/정규화 땜질” 같은 **임시방편을 먼저 제시**했다.
   이는 장기 비전과 어긋났고, 사용자가 느끼는 ‘이번만 넘어가려는’ 불신을 키웠다.

3. 디버깅 안내가 일관되지 못했다.
   `curl -i`로 원문 확인 → 로그 tail로 원인 확정 → 최소 패치 적용이라는 루틴을 처음부터 강제하지 않아,
   jq parse error 같은 2차 증상이 문제를 가리며 불필요한 혼란이 생겼다.

앞으로는:

* **패치 제공 전 반드시 현 소스/컨테이너 내 파일을 확인**하고,
* 수정은 **통째 던지기 금지**, **diff/교체 블록**으로만 제공하며,
* 목표(범용 자연어 제어 AI)의 핵심(Probe→Decide→Replan loop)을 항상 중심에 놓고 진행하겠다.

## 6) 이후 TODO 리스트 (우선순위 순)

### P0 — “비전 구현의 핵심” (다음 단계에서 바로 해야 함)

1. **LLM Probe Loop 엔진(v0)** 구현

   * 입력: request_text, missing_inputs, 후보(target/playbook), 현재 상태(state), 정책
   * 출력: probes(확인 명령들), decisions(선택), need_human(최소 질문)
   * 실행: probes는 SubAgent `/a2a/run_script`로 실행, 결과는 evidence/audit에 저장

2. **Human 최소 질의 엔드포인트 추가**

   * `/projects/{id}/answer` (or `/clarify`)
   * previous plan/questions/choices + user answers를 state에 저장
   * run_auto 재호출 시 answers를 inputs로 반영

3. **IR(JSON) 중간표현 도입**

   * YAML/자연어/LLM-output을 모두 IR로 변환
   * IR 기반으로 `probe→decide→exec→validate`를 실행
   * state/audit에 IR을 저장해 재현성 확보

## 7) 현재 상태 결론

M3-4.4는 “자동 선택 + 자동 실행”의 골격을 만들었고,
입력값 부족 시 “최소 질문으로 중단”하는 v0 resolver까지 연결했다.

그러나 목표 비전(LLM이 확인 명령을 동적으로 생성/실행/판단/재계획하는 범용 시스템 제어)은
**LLM Probe Loop + IR 기반 실행**이 들어가야 비로소 충족된다.

즉, M3-4.4는 **‘자동 실행 파이프라인’의 뼈대**까지는 완료했고,
다음 단계에서 **‘지능(확인→판단→재계획)’ 엔진**을 반드시 붙여야 한다.
