# M3-5.2 구현 범위(v0)

## A) Action IR v0 (LLM 출력 포맷 고정)

LLM이 반환할 수 있는 건 딱 3가지 중 하나(또는 조합):

### 1) actions

* `shell`: bash 스크립트(여러 줄 가능)
* `http`: curl/requests 실행(초기엔 shell로 curl만 써도 됨)

```json
{
  "actions": [
    {
      "id": "p1",
      "type": "shell",
      "target_id": "remote-1",
      "timeout_s": 30,
      "script": "ip -o link show\nip route\n"
    }
  ]
}
```

### 2) resolved_inputs

LLM이 probe 결과 기반으로 “결정된 값”만 반환

```json
{"resolved_inputs": {"iface_in":"ens33","iface_out":"ens34"}}
```

### 3) question

정말 애매하거나 위험하면 human에게 최소 질문

```json
{
  "question": {
    "next_questions": ["iface_in/out 중 무엇이 외부망인가?"],
    "choices": {"iface_out": ["ens33","ens34"]}
  }
}
```

## B) Probe Loop 엔진(v0)

새 모듈 `api/probe_loop.py` 하나 만들고, 여기서만 루프를 돈다.

### 입력(ProbeLoopCtx)

* `project_id`
* `request_text`
* `plan`(selected_playbook_id/target_ids)
* `missing_inputs`, `choices`(있다면)
* `last_runs`(최근 stdout/stderr 요약)
* `policy`(denylist/allowlist)

### 출력(ProbeLoopResult)

* `status: ready|needs_clarification|failed`
* `resolved_inputs`(있다면)
* `question`(있다면)
* `executions`: 실행한 action 결과(증빙 refs)

---

## C) run_auto 연결(v0)

현재 `run_auto`가

* plan이 ready면 playbook 실행
* inputs 부족하면 input_resolver로 질문

이 흐름을 M3-5.2에서는 이렇게 바꾼다:

1. plan 생성/확인(기존 유지)
2. **missing/choices/validate fail이면 → probe_loop 호출**
3. probe_loop가 resolved_inputs를 내면 → inputs에 merge하고 계속
4. question이면 그대로 반환

> 즉 “하드코딩된 YAML discovery”는 v0에서 보조로 남겨도 되고,
> 핵심은 probe_loop가 **LLM 기반으로 해결**하는 것.

# 핵심 구현 포인트(실수 방지)

## 1) 어떤 LLM을 쓰나?

* 무조건: `llm_registry.resolve_llm_conn_for_role("master", target_id)`
* 그 결과 conn을 `master_clients.call_conn(conn, prompt)`로 호출

## 2) LLM이 “명령을 만든다”는 뜻

* LLM은 **actions JSON을 생성**한다.
* 실행은 항상 SubAgent(`/a2a/run_script`)가 한다.
* 모든 action 결과(stdout/stderr/evidence_refs)를 다시 LLM에 feed한다.

## 3) 무한루프 방지

* max_iter 기본 3
* 각 iter마다 audit 남김
* 매 iter마다 “목표 달성 조건/중단 조건” 체크
