# Chapter 9: 자동화의 꿈

---

월요일 오전. 최민재는 관제실 한쪽에서 Playbook을 만들고 있었다.

"민재 선배, Playbook이 뭐예요?" 가람이 물었다.

민재가 화면을 돌려 보여주었다.

"**같은 작업을 매번 사람이 직접 하면** 시간도 걸리고, 실수도 하고, 기분에 따라 결과가 달라져. Playbook은 '이 순서대로, 이 도구로, 이 파라미터로 실행해라'를 **미리 정의**해두는 거야."

```
예: "서버 보안 점검" Playbook
  Step 1: hostname && uptime       (tool: run_command)
  Step 2: df -h                    (tool: run_command)
  Step 3: ss -tlnp                 (tool: run_command)
  Step 4: sudo -l                  (tool: run_command)
```

"이걸 한 번 만들어두면, **1 API call로 재실행** 가능해. 사람이 개입하지 않아도 항상 같은 결과가 나와."

가람이 생각했다. "그럼 LLM에게 '서버 점검해'라고 말하면 LLM이 알아서 하는 거랑 뭐가 달라요?"

민재가 웃었다. "좋은 질문이야."

그가 화이트보드에 적었다.

```
LLM adhoc 실행:
  "서버 점검해" → LLM이 매번 다른 명령을 생성할 수 있음
  예1: hostname && uptime && df -h
  예2: uname -a && free -h && top -bn1
  → 비결정론적 (매번 다를 수 있다)

Playbook 실행:
  "서버 점검 Playbook 실행" → 항상 동일한 명령 시퀀스
  → 결정론적 (매번 같다)
```

"**재현성**이 차이야." 민재가 말했다. "인시던트 대응에서 '지난번과 같은 점검을 다시 해'라고 하면, Playbook은 100% 같은 결과를 보장해. LLM은 보장 못 해."

"100%요?" 가람이 놀랐다.

"우리가 실험했어. 같은 Playbook을 **10번 반복** 실행했는데, 10번 전부 동일한 출력이 나왔어. LLM adhoc은? 시도할 때마다 다른 명령을 생성했어."

> **가람의 메모:**
> Playbook = 결정론적 재현 (100%)
> LLM adhoc = 비결정론적 (매번 다를 수 있음)
> 새로운 상황 → LLM이 유리 (적응적)
> 반복 운영 → Playbook이 유리 (재현성)

---

민재는 OpsClaw에 Playbook을 등록하는 과정을 보여주었다.

```bash
# Playbook 생성
curl -s -X POST http://localhost:8000/playbooks \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "name": "daily-server-health",
    "version": "1.0",
    "category": "operations",
    "description": "일일 서버 상태 점검"
  }'

# Step 추가
curl -s -X POST "http://localhost:8000/playbooks/$PBID/steps" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "step_order": 1,
    "step_type": "tool",
    "name": "시스템 현황",
    "ref_id": "run_command",
    "params": {"command": "hostname && uptime && uname -r"}
  }'
```

"Playbook이 등록되면, 프로젝트에 연결하고 `playbook/run`으로 실행할 수 있어."

---

"근데 민재 선배." 가람이 물었다. "Playbook이 결정론적이라면... SIGMA 룰도 자동으로 만들 수 있어요?"

민재의 눈이 빛났다.

"그것도 해봤어. Ollama LLM에게 공격 패턴을 설명하고, SIGMA 룰을 생성하라고 시켰지."

```bash
curl -s http://192.168.0.105:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma3:12b",
    "messages": [
      {"role": "system", "content": "SIGMA 룰 전문가입니다. 유효한 SIGMA YAML만 출력하세요."},
      {"role": "user", "content": "SSH 브루트포스를 탐지하는 SIGMA 룰을 작성하세요. 5분 내 동일 IP에서 10회 이상 SSH 인증 실패."}
    ],
    "temperature": 0.2
  }'
```

"LLM이 5초 만에 SIGMA 룰을 생성했어."

```yaml
title: SSH Brute Force Detection
status: experimental
logsource:
    product: linux
    service: sshd
detection:
    selection:
        eventid: 4625
    condition: selection | count(srcip) > 10
    timeframe: 5m
level: high
tags:
    - attack.credential_access
    - attack.t1110.001
```

"이걸 Wazuh 룰로 변환하면, 실시간 탐지가 가능해."

가람은 감탄했다. "LLM이 탐지 룰까지 만들어요?"

"만들어. 하지만—" 민재가 손가락을 들었다. "LLM이 만든 룰을 **그대로 배포하면 안 돼**. 오탐이 있을 수 있고, 문법이 틀릴 수도 있어. 반드시 `wazuh-logtest`로 테스트한 후에 배포해야 해."

"사람이 검증하는 단계가 필요하다?"

"맞아. **LLM은 초안을 만들고, 사람이 검증한다.** 이게 자동화의 올바른 방식이야."

---

점심 시간. 민재와 가람은 식당에서 대화를 이어갔다.

"민재 선배, 추팀장님이 OpsClaw를 설계할 때 가장 중요하게 생각한 게 뭐였어요?"

민재가 젓가락을 멈추었다.

"세 가지." 그가 손가락을 폈다.

"첫째, **모든 것을 기록한다.** 이건 여러 번 말했지."

"둘째, **기계가 할 일과 사람이 할 일을 나눈다.** 반복 실행은 기계(Playbook), 판단은 사람(Master). 이것도 말했지."

"셋째는..." 민재가 잠시 생각했다. "셋째는 좀 이상한 얘긴데."

"뭔데요?"

"팀장님이 이런 말을 했어. **'이 시스템은 나를 증명해야 한다.'**"

"나를 증명한다?"

"그래. 처음에는 '시스템이 제대로 동작한다는 것을 증명한다'는 뜻인 줄 알았어. 근데 지금 생각하면..."

민재가 말을 흐렸다.

"뭐예요?"

"아니, 아마 그런 뜻이겠지." 민재가 웃으며 화제를 돌렸다. "디저트 먹을래?"

가람은 수첩에 적었다.

> `"이 시스템은 나를 증명해야 한다." — 추팀장`
> `무엇을 증명? 시스템의 동작을? 아니면... 자기 자신의 행동을?`
