자정이 넘었다. 도윤은 커피를 마시며 Suricata fast.log를 확인했다.

```bash
sshpass -p1 ssh -o StrictHostKeyChecking=no secu@10.20.30.1 \
  "echo 1 | sudo -S tail -f /var/log/suricata/fast.log" 2>/dev/null
```

로그가 천천히 흘렀다. 대부분은 ET 기본 룰의 정보성 알림이었다.

```
03/28/2026-00:15:22  [**] [1:2200121:1] SURICATA Ethertype unknown [**]
03/28/2026-00:20:35  [**] [1:2200121:1] SURICATA Ethertype unknown [**]
```

`Ethertype unknown`. LLDP(Link-Layer Discovery Protocol) 패킷이었다. 네트워크 장비가 자동으로 보내는 것으로, 위협이 아니다.

> **도윤의 메모:** LLDP → Suricata가 모르는 이더넷 프레임 타입. 정상. 무시.

잠시 후 다른 유형의 로그가 나타났다.

```
03/28/2026-00:35:53  [**] [1:2200121:1] SURICATA Ethertype unknown [**]
```

같은 패턴. 도윤은 다시 무시하려다 멈추었다.

*잠깐. 이 패킷의 소스 MAC이...*

도윤은 raw 패킷 데이터를 확인했다.

```
[Raw pkt: 01 80 C2 00 00 0E 68 1D EF 59 40 55 88 CC ...]
```

`01:80:C2:00:00:0E` — 이것은 LLDP 멀티캐스트 MAC 주소다. 정상.

`68:1D:EF:59:40:55` — 이것은 소스 MAC. 도윤은 ARP 테이블을 확인했다.

```bash
arp -a | grep "68:1d:ef:59:40:55"
```

결과가 없었다. 이 MAC 주소는 현재 ARP 테이블에 등록되어 있지 않다.

> **정상적인 경우:** 네트워크에 있는 장비의 MAC이라면 ARP에 기록되어야 한다.
> **이상한 경우:** LLDP는 보내고 있는데 ARP에는 없다? 물리적으로 연결되어 있지만 IP 통신은 하지 않는 장비?

도윤은 잠시 고민했다. 이것이 정상인지 비정상인지 확신이 없었다. 네트워크 장비일 수도 있고, 관리용 인터페이스가 IP 없이 연결된 것일 수도 있다.

*정규 선배라면 어떻게 할까?*

도윤은 수첩에 적고 넘어가기로 했다.

> `00:35 — 미등록 MAC에서 LLDP 패킷. 장비 확인 필요. 긴급 아님.`

새벽 2시. 도윤은 nftables 방화벽 설정을 점검하고 있었다. 주간에 김영훈이 지시한 정기 점검 항목이었다.

```bash
sshpass -p1 ssh -o StrictHostKeyChecking=no secu@10.20.30.1 \
  "echo 1 | sudo -S nft list ruleset 2>/dev/null"
```

화면에 규칙이 쏟아졌다.

```
table inet filter {
    chain input {
        type filter hook input priority filter; policy drop;
        ct state established,related accept
        iif "lo" accept
        iif "enp4s0" accept
        tcp dport { 22, 8000, 8002 } accept
        icmp type echo-request accept
        ip6 nexthdr ipv6-icmp accept
        tcp dport { 80, 443, 514, 1514, 1515, 8080, 8443, 9200, 9392, 55000 } accept
        udp dport { 53, 514, 1514 } accept
        log drop
    }
    ...
}
```

도윤은 규칙을 하나씩 확인했다.

> **policy drop** — 기본 정책이 차단. 화이트리스트 방식. 좋다.
> **ct state established,related accept** — 이미 수립된 연결은 허용. 필수.
> **iif "lo" accept** — 루프백 허용. 필수.
> **tcp dport { 22, 8000, 8002 }** — SSH(22), OpsClaw Manager(8000), SubAgent(8002). 필요한 포트.
> **tcp dport { 80, 443, 514, 1514... }** — 웹, HTTPS, syslog, Wazuh 포트. 정상.

도윤은 포워드 체인도 확인했다.

```
    chain forward {
        type filter hook forward priority filter; policy drop;
        ...
    }
```

> **forward도 drop.** 좋다. secu를 경유하는 트래픽도 기본 차단.

여기까지는 정상이었다. 도윤은 NAT 테이블도 확인했다.

```
table ip nat {
    chain prerouting {
        type nat hook prerouting priority dstnat; policy accept;
        iif "enp1s0" tcp dport 8080 dnat to 10.20.30.80:80
        iif "enp1s0" tcp dport 8443 dnat to 10.20.30.80:443
    }
    chain postrouting {
        type nat hook postrouting priority srcnat; policy accept;
        oif "enp1s0" masquerade
    }
}
```

> **DNAT:** 외부에서 secu:8080 → web:80 포트 포워딩. 외부 사용자가 웹에 접근하는 경로.
> **masquerade:** 내부에서 외부로 나가는 트래픽의 IP 변환. 공유기와 같은 원리.

도윤은 점검 결과를 인수인계 문서에 정리했다.

```
[야간 점검] 방화벽(nftables) 상태
- filter: input/forward policy=drop ✓
- 허용 포트: 22, 80, 443, 514, 1514, 8000, 8002 등 ✓
- NAT: DNAT(8080→80, 8443→443), masquerade ✓
- 비인가 규칙: 없음 ✓
- 점검 시각: 02:15
```
