# Chapter 4: 가용성의 대가

---

목요일 오전 9시. 넥서스AI 인프라 운영본부.

정태식 본부장의 사무실은 서버실 바로 옆에 있었다. 벽 너머로 냉각 팬의 웅웅거리는 소리가 들렸다. 사무실에는 모니터 4대가 놓여 있었고, 그중 하나에는 항상 **서버 가동 현황(uptime dashboard)**이 띄워져 있었다.

```
[NexusAI Infrastructure Status]
Web Cluster:  ████████████████████ 99.97% uptime (30d)
DB Cluster:   ████████████████████ 99.99% uptime (30d)
AI Training:  ██████████████████── 98.12% uptime (30d)
SOC Services: ████████████████████ 99.95% uptime (30d)
```

정태식은 이 화면을 **아침에 가장 먼저** 확인했다. 99.9% 이상이면 좋은 아침, 99% 미만이면 누군가 혼나는 아침이었다.

오늘은 좋은 아침이었다 — 인프라 측면에서는.

---

"본부장님, 보안팀에서 왔습니다."

김영훈과 가람이 정태식의 사무실에 들어섰다. 정태식은 의자를 돌리며 두 사람을 바라보았다.

"또 보안 얘기야?"

"네. 긴급 보안 패치 관련입니다." 김영훈이 서류를 꺼냈다.

```
[긴급 보안 패치 요청]

대상: 전 서버 (opsclaw, secu, web, siem)
항목:
  1. OpenSSH 8.9p1 → 9.7p1 업그레이드 (CVE-2024-6387, regreSSHion)
  2. Apache 2.4.52 → 2.4.62 업그레이드 (CVE-2024-38475)
  3. Linux Kernel 6.8.0-106 → 6.8.0-115 (CVE-2024-26581)
  4. sudo 설정 강화 (NOPASSWD 제거 — 이미 완료)
  5. 전 서버 SSH 키 기반 인증 전환 (비밀번호 인증 비활성화)

요청 일정: 이번 주 내 적용
```

정태식이 서류를 읽고 얼굴을 찡그렸다.

"**이번 주 내?** 김 과장, 지금 Nexus-1 릴리즈가 2주 남았어. 서버를 재시작하면 학습 파이프라인이 **4시간** 멈춰. 서문기 상무가 뭐라 하겠어?"

"본부장님." 김영훈이 차분하게 말했다. "CVE-2024-6387은 **원격 코드 실행** 취약점입니다. SSH가 열려 있는 모든 서버가 영향을 받아요."

"CVSS 점수가?"

"**8.1.** High입니다."

정태식이 손을 흔들었다. "High? Critical이 아니잖아. Critical이면 바로 하는데, High는 **다음 정기 패치 주기에** 하면 돼."

가람은 이 대화를 들으며 수첩에 적었다.

> `CVSS(Common Vulnerability Scoring System):`
> `0.0~3.9: Low (참고 사항)`
> `4.0~6.9: Medium (주의 필요)`
> `7.0~8.9: High (긴급 대응)`
> `9.0~10.0: Critical (즉시 대응)`
> `정태식 본부장: "High는 다음 정기 패치에" — 김영훈과 충돌`

---

"본부장님." 김영훈의 목소리가 낮아졌다. "지난달에 SSH 비밀번호 '1'이 뚫려서 10일간 C2 비콘이 동작했습니다. 그때도 **'서비스 먼저'**라고 패치를 미뤘었죠."

정태식의 얼굴이 붉어졌다. "그건—"

"그리고 공유 계정 admin/admin123으로 secu 서버까지 접근당했습니다. 그것도 2년간 방치된 **'관리 편의'** 때문이었죠."

정태식이 일어섰다. "김 과장. 지금 나한테 **과거 일을 들춰서** 압박하는 건가?"

"사실을 말씀드리는 겁니다."

"사실? 사실은 이거야." 정태식이 모니터를 가리켰다. "**99.97% 가동률.** 이게 내 책임이야. 서버가 한 시간 죽으면 **매출이 날아가**. 보안 패치 때문에 서비스가 죽으면, 그 책임은 **네가 져**?"

---

가람은 두 사람 사이의 긴장을 느꼈다. 이것은 기술적 논쟁이 아니었다. **조직의 우선순위** 논쟁이었다.

김영훈이 말했다. "본부장님. 패치를 안 하면 **서버가 뚫려서** 서비스가 죽을 수도 있습니다."

"**뚫릴 수도 있다**와 **확실히 멈춘다**는 다른 거야." 정태식이 말했다.

"지난달에 **실제로 뚫렸**잖아요."

정태식이 잠시 침묵했다.

김영훈이 한 발 물러섰다. "절충안을 제안합니다."

```
[절충안]

1. SSH 업그레이드: 야간 시간(새벽 2~4시) + 롤링 재시작
   → 서비스 다운타임 0 (서버 하나씩 순차 재시작)

2. Apache 업그레이드: 주말 야간
   → Nexus-1 학습 파이프라인에 영향 없음 (웹 서비스만)

3. 커널 업그레이드: Nexus-1 릴리즈 후로 연기 (2주 후)
   → 리스크 수용, 단 Suricata 모니터링 강화

4. SSH 키 인증 전환: 1주 내 단계적 적용
   → 관리자 키 배포 → 테스트 → 비밀번호 비활성화
```

정태식이 절충안을 읽었다. 표정이 누그러졌다.

"롤링 재시작이면... 서비스 다운은 없어?"

"없습니다. 로드밸런서가 트래픽을 돌리는 동안 한 서버씩 재시작하면 됩니다."

"... 좋아. 이 안으로 해."

---

회의실을 나오며 가람이 물었다.

"과장님, 왜 처음부터 절충안을 내지 않으셨어요? 정면 충돌하시다가..."

김영훈이 걸으며 대답했다.

"처음에 절충안을 내면 **그것마저 깎으려** 해. '야간 재시작? 그것도 좀...' 이러면서. 먼저 **최대치를 요구**하고, 그 다음에 '이만큼 양보하겠다'고 해야 **원래 원하던 수준**을 얻을 수 있어."

"그건... 협상 기법이잖아요."

"보안 엔지니어는 반만 기술자고, 반은 **외교관**이야." 김영훈이 씁쓸하게 웃었다.

가람은 수첩에 적었다.

> `보안 패치 vs 서비스 가용성 — 영원한 트레이드오프`
> `정태식: "99.97% 가동률 = 내 책임"`
> `김영훈: "패치 안 하면 뚫려서 서비스가 죽을 수도"`
> `절충안: 롤링 재시작(다운타임 0) + 단계적 적용`
>
> `[교훈] "보안 패치를 미루는 가장 흔한 이유는 '서비스가 죽으면 어쩌나'이다.`
> `하지만 해킹으로 서비스가 죽으면, 패치보다 훨씬 오래 멈춘다."`
>
> `[CVSS 점수 기준]`
> `8.1(High) → 정태식은 "다음 정기 패치에"`
> `김영훈은 "즉시 대응"`
> `→ 조직의 리스크 수용 기준(risk appetite)이 다르다.`

### 실제 패치 적용

그날 밤, 김영훈과 도윤이 야간 패치를 진행했다. OpsClaw로 기록하면서.

```bash
# 패치 프로젝트 생성
PID=$(curl -s -X POST http://localhost:8000/projects \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"name":"patch-20260418-ssh-apache","request_text":"긴급 보안 패치 — CVE-2024-6387, CVE-2024-38475","master_mode":"external"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['project']['id'])")

curl -s -X POST "http://localhost:8000/projects/$PID/plan" -H "X-API-Key: opsclaw-api-key-2026" > /dev/null
curl -s -X POST "http://localhost:8000/projects/$PID/execute" -H "X-API-Key: opsclaw-api-key-2026" > /dev/null
```

패치 순서: secu → web → siem → opsclaw (하나씩 순차).

```bash
# Step 1: secu 서버 SSH 업그레이드
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S apt-get update && echo 1 | sudo -S apt-get install -y openssh-server 2>&1 | tail -5","subagent_url":"http://10.20.30.1:8002"}'
```

"과장님, 이걸 OpsClaw로 하면 **패치 이력이 전부 기록**되는 거죠?" 도윤이 물었다.

"맞아. 나중에 '언제, 어떤 서버에, 어떤 패치를, 누가 적용했는지' 감사할 때 replay 한 번이면 돼."

```bash
# 패치 후 SSH 버전 확인
curl -s -X POST "http://localhost:8000/projects/$PID/execute-plan" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "tasks": [
      {"order":1,"title":"secu SSH 버전","instruction_prompt":"ssh -V 2>&1","risk_level":"low","subagent_url":"http://10.20.30.1:8002"},
      {"order":2,"title":"web SSH 버전","instruction_prompt":"ssh -V 2>&1","risk_level":"low","subagent_url":"http://10.20.30.80:8002"},
      {"order":3,"title":"siem SSH 버전","instruction_prompt":"ssh -V 2>&1","risk_level":"low","subagent_url":"http://10.20.30.100:8002"},
      {"order":4,"title":"opsclaw SSH 버전","instruction_prompt":"ssh -V 2>&1","risk_level":"low","subagent_url":"http://localhost:8002"}
    ],
    "subagent_url":"http://localhost:8002",
    "parallel":true
  }'
```

도윤이 화면을 보며 배웠다. "4대 서버의 SSH 버전을 **한 번에 병렬로** 확인하는 거네요."

"직접 SSH로 하면 4번 접속해야 하지만, execute-plan으로 하면 **0.5초에 4대 동시 확인**이야." 김영훈이 말했다. "그리고 결과가 전부 evidence로 남아."

```
결과:
  secu:    OpenSSH_9.7p1  ← 업그레이드 완료
  web:     OpenSSH_9.7p1  ← 업그레이드 완료
  siem:    OpenSSH_9.7p1  ← 업그레이드 완료
  opsclaw: OpenSSH_9.7p1  ← 업그레이드 완료
```

"**4대 전부 9.7p1.** CVE-2024-6387 패치 완료." 김영훈이 완료 보고를 작성했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/completion-report" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "summary": "SSH 긴급 패치 완료 — CVE-2024-6387 대응",
    "outcome": "success",
    "work_details": ["secu/web/siem/opsclaw 4대 OpenSSH 9.7p1 업그레이드","롤링 재시작으로 다운타임 0"],
    "issues": [],
    "next_steps": ["Apache 패치 주말 예정","커널 패치 2주 후 예정","SSH 키 인증 전환 1주 내"]
  }'
```

---

새벽 4시. 패치가 끝났다. 도윤이 기지개를 켰다.

"과장님. 보안 패치를 하면서 느낀 건데요."

"뭐?"

"**패치 자체보다 정태식 본부장 설득이 더 어려웠어요.**"

김영훈이 웃었다. 처음으로 도윤 앞에서 웃는 것을 가람은 보았다.

"그게 보안의 현실이야. 기술은 10%고, 사람을 설득하는 게 90%야."

"추팀장님도 비슷한 말씀하셨어요. '보안은 원래 정치야'라고."

"..." 김영훈이 잠시 멈추었다. "팀장님 말이 맞아. 인정하기 싫지만."

---

가람은 그날의 교훈을 수첩에 정리했다.

```
[보안 패치 실습 정리]

1. 취약점 식별:
   CVE-2024-6387 (OpenSSH, CVSS 8.1, 원격 코드 실행)
   → 모든 SSH 서버가 영향

2. 리스크 평가:
   - 가용성 리스크: 서버 재시작 시 서비스 일시 중단
   - 보안 리스크: 패치 안 하면 원격 공격에 취약
   → 보안 리스크 > 가용성 리스크

3. 절충안 설계:
   - 롤링 재시작 (다운타임 0)
   - 단계적 적용 (SSH → Apache → Kernel)
   - 커널은 릴리즈 후로 연기 (리스크 수용)

4. 패치 적용:
   - OpsClaw execute-plan으로 4대 동시 패치
   - 패치 전후 버전 확인 (evidence)
   - completion-report로 기록

5. 교훈:
   "기술은 10%, 설득은 90%"
   "패치를 미루는 이유는 항상 '서비스'"
   "하지만 해킹으로 서비스가 죽으면, 패치보다 훨씬 오래 멈춘다"
```
