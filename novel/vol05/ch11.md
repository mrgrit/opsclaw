# Chapter 11: 룰 생성기

월요일 오전 9시. 넥서스AI SOC 관제실. 5월 첫째 주.

"오늘은 **탐지 룰 자동 생성**이야." 김영훈이 말했다.

정규가 눈을 빛냈다. 탐지 룰은 **정규의 영역**이었다. 1권에서 Suricata 룰을 수작업으로 작성하고, 3권에서 Wazuh 커스텀 룰을 만들고, 4권에서 SIGMA 룰을 작성한 — 그 정규.

"과장님. LLM이 **제 일**을 빼앗으려는 건가요?" 정규가 무표정으로 물었다.

"빼앗는 게 아니라 **도와주는** 거야." 김영훈이 말했다.

"... 제 룰이 더 정확합니다."

"그걸 증명해봐."

정규의 눈이 가늘어졌다. **도전**이 들어왔다.

### 대결: 사람 vs LLM

김영훈이 규칙을 정했다.

```
[탐지 룰 생성 대결 — 정규 vs LLM]

과제: 아래 공격 시나리오에 대한 탐지 룰을 작성하라.

시나리오 1: SSH 키 기반 무단 접근
  → 비인가 SSH 키가 authorized_keys에 추가됨
  → Wazuh 룰 (local_rules.xml 형식)

시나리오 2: DNS 터널링
  → 비정상적으로 긴 DNS 쿼리 (64자 이상)
  → Suricata 룰 형식

시나리오 3: 웹셸 업로드
  → PHP/JSP/ASP 파일이 웹 디렉토리에 생성됨
  → SIGMA 룰 형식

평가 기준:
  1. 정확성: 공격을 정확히 탐지하는가
  2. 오탐률: 정상 행위를 오탐하지 않는가
  3. 속도: 얼마나 빨리 작성하는가
  4. 완성도: 메타데이터, 설명, 참조 포함 여부
```

"정규는 수작업으로, LLM은 OpsClaw dispatch로 동시에 시작." 김영훈이 말했다.

"시작!" 도윤이 외쳤다. 비공식 심판.

---

**시나리오 1: SSH 키 무단 접근 — Wazuh 룰**

정규가 타이핑을 시작했다. 빠르고 정확했다. 10년의 근육 기억.

```xml
<!-- 정규 작성 — 4분 32초 -->
<rule id="100201" level="12">
  <if_sid>550</if_sid>
  <match>authorized_keys</match>
  <description>Unauthorized SSH key modification detected</description>
  <mitre>
    <id>T1098.004</id>
  </mitre>
  <group>authentication_modified,pci_dss_10.2.7,</group>
</rule>
```

동시에 가람이 LLM에 프롬프트를 보냈다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "command": "curl -s http://192.168.0.105:11434/v1/chat/completions -H \"Content-Type: application/json\" -d \"{\\\"model\\\":\\\"gemma3:12b\\\",\\\"messages\\\":[{\\\"role\\\":\\\"system\\\",\\\"content\\\":\\\"당신은 Wazuh SIEM 전문가입니다. Wazuh local_rules.xml 형식으로 탐지 룰을 작성하세요.\\\"},{\\\"role\\\":\\\"user\\\",\\\"content\\\":\\\"비인가 SSH 키가 authorized_keys에 추가되는 것을 탐지하는 Wazuh 룰을 작성해주세요. MITRE ATT&CK 매핑 포함.\\\"}]}\" 2>/dev/null | python3 -c \"import sys,json; r=json.load(sys.stdin); print(r[\\\"choices\\\"][0][\\\"message\\\"][\\\"content\\\"])\"",
    "mode": "shell",
    "subagent_url": "http://localhost:8002"
  }'
```

```xml
<!-- LLM 생성 — 18초 (네트워크 포함) -->
<rule id="100501" level="12">
  <if_sid>550</if_sid>
  <field name="audit.file.name">.ssh/authorized_keys</field>
  <description>SSH authorized_keys file modified - possible unauthorized key addition (T1098.004)</description>
  <mitre>
    <id>T1098.004</id>
  </mitre>
  <options>no_full_log</options>
  <group>syscheck,authentication_modified,pci_dss_10.2.7,gdpr_IV_35.7.d,</group>
</rule>

<rule id="100502" level="14">
  <if_sid>100501</if_sid>
  <time>10:00pm - 6:00am</time>
  <description>SSH key modification during off-hours - HIGH RISK</description>
  <group>syscheck,authentication_modified,</group>
</rule>
```

"**18초.** 정규 4분 32초." 도윤이 시간을 쟀다.

"하지만 내용을 비교하자." 김영훈이 두 룰을 나란히 놓았다.

```
비교:
                     정규                    LLM
─────────────────────────────────────────────────
시간              4분 32초                 18초
정확도            ✅ 정확                  ✅ 정확
field 매칭        generic (match)         specific (audit.file.name)
MITRE 매핑        ✅ T1098.004            ✅ T1098.004
추가 룰           없음                    ✅ 야간 시간 룰 (100502)
PCI-DSS 매핑      ✅                      ✅ + GDPR 추가
오탐 방지         보통                    더 정밀 (field 지정)
```

"**LLM이 더 정밀해.**" 진호가 말했다. "field name을 구체적으로 지정했고, 야간 시간 탐지 룰까지 추가했어."

정규가 LLM의 룰을 노려보았다. 10초.

"... **field 지정**은 내가 빼먹었어." 정규가 인정했다. "그리고 야간 시간 룰은 좋은 아이디어야."

"**하지만.**" 정규가 손가락을 들었다. "이 룰을 `wazuh-logtest`로 테스트해봐야 해. LLM이 생성한 XML이 **문법적으로 올바른지** 확인 안 했잖아."

---

민재가 LLM 룰을 siem 서버에서 테스트했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S /var/ossec/bin/wazuh-logtest < /tmp/test_sshkey_log.txt 2>&1 | tail -5","subagent_url":"http://10.20.30.100:8002"}'
```

"**문법 정상, 룰 매칭 성공.**" 민재가 말했다.

정규가 입술을 깨물었다. 자존심이 상하는 표정.

"... 인정해." 정규가 말했다. "LLM이 이번 라운드는 이겼어."

"정규 선배." 가람이 말했다. "하지만 LLM이 만든 룰이 **실환경에서 오탐을 일으키지 않는지**는 선배가 확인해야 해요."

"맞아." 정규의 표정이 풀렸다. "LLM이 만들고, 내가 검증하는 거지. **만드는 건 AI, 책임은 사람.**"

### 시나리오 2, 3 결과

나머지 두 시나리오도 진행했다.

```
[대결 종합 결과]

시나리오 1 (Wazuh): LLM 승 — 더 정밀, 추가 룰 포함
시나리오 2 (Suricata): 무승부 — 둘 다 정확, LLM이 5배 빠름
시나리오 3 (SIGMA): 정규 승 — LLM이 SIGMA 문법 오류 발생

종합:
  속도: LLM 압도적 승 (평균 15초 vs 5분)
  정확성: 비슷 (2:1로 LLM 소폭 우위)
  완성도: LLM 우위 (메타데이터, 매핑이 더 풍부)
  안정성: 정규 우위 (LLM의 SIGMA 문법 오류)
```

"**속도는 LLM, 안정성은 사람.**" 김영훈이 정리했다.

"최적의 워크플로는 뭘까?" 가람이 물었다.

"**LLM이 초안(draft)을 생성하고, 사람이 검토(review)하는 것.**" 김영훈이 말했다. "LLM이 15초에 초안을 만들면, 정규가 2분에 검토해서 수정. 총 2분 15초. 정규 혼자 하면 5분."

"**2배 빨라지면서 품질은 유지.**" 정규가 고개를 끄덕였다. "이건 인정할 수 있어."

민재가 웃으며 말했다. "정규야, 네 일자리는 안전해. 다만 이제 **동료가 AI**인 거지."

"AI 동료가 커피를 안 망치면 더 좋겠다." 정규가 무표정으로 말했다.

"그건 내 얘기야?" 민재가 물었다.

"**커피를 망치는 건** 인간만의 고유 능력이잖아." 도윤이 웃었다.
