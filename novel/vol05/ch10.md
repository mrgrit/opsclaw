# Chapter 10: 로그를 읽는 AI

---

금요일 오전 9시. 넥서스AI SOC 관제실.

어제 프롬프트 엔지니어링으로 LLM 기반 경보 분석을 성공시킨 후, 김영훈이 다음 단계를 제시했다.

"137건의 경보를 LLM이 분석한 건 **단일 시점**이야. 오늘은 **실시간 로그 스트림**을 LLM이 분석하는 파이프라인을 만든다."

"실시간이요?" 민재가 눈을 빛냈다. "OpsClaw의 **Watcher**와 **Schedule**을 결합하면 될 것 같은데요."

"맞아. Watcher가 로그를 수집하고, LLM이 분석하고, 위협이 발견되면 자동으로 인시던트를 생성하는 **자동 관제 파이프라인**이야."

```
[LLM 로그 분석 파이프라인 — 설계]

                   ┌─── Wazuh ───┐
Suricata logs ─────┤              ├──→ 로그 수집기
Apache logs ───────┤  syslog      │    (OpsClaw Watcher)
auth.log ──────────┤              │         ↓
                   └──────────────┘    LLM 분석
                                       (invoke_llm)
                                           ↓
                                    ┌── 결과 분류 ──┐
                                    │               │
                              true_positive    false_positive
                                    ↓               ↓
                              인시던트 생성      로그 기록만
                              Slack 알림
                              OpsClaw evidence
```

### 파이프라인 구현

민재가 구현을 이끌었다.

"OpsClaw의 **Schedule**로 5분마다 로그를 수집하고, **dispatch auto 모드**로 LLM 분석을 실행하고, 결과에 따라 **Notification**으로 알림을 보내는 거야."

```bash
# 1. 로그 수집 + LLM 분석 스케줄 등록
curl -s -X POST "http://localhost:8000/schedules" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "name": "llm-log-analysis",
    "cron": "*/5 * * * *",
    "project_template": {
      "name": "auto-log-analysis",
      "request_text": "최근 5분 Wazuh 경보 LLM 분석"
    },
    "task_template": {
      "instruction_prompt": "echo 1 | sudo -S tail -100 /var/ossec/logs/alerts/alerts.json 2>/dev/null | python3 -c \"import sys,json; lines=sys.stdin.readlines(); alerts=[json.loads(l) for l in lines if l.strip()]; high=[a for a in alerts if int(a.get(\\\"rule\\\",{}).get(\\\"level\\\",0))>=8]; print(f\\\"Total: {len(alerts)}, High(>=8): {len(high)}\\\"); [print(json.dumps({\\\"id\\\":a[\\\"id\\\"],\\\"rule\\\":a[\\\"rule\\\"][\\\"description\\\"],\\\"level\\\":a[\\\"rule\\\"][\\\"level\\\"],\\\"src\\\":a.get(\\\"srcip\\\",\\\"N/A\\\")})) for a in high[:5]]\"",
      "risk_level": "low",
      "subagent_url": "http://10.20.30.100:8002"
    }
  }'
```

"**5분마다** siem 서버에서 Wazuh 경보를 수집하고, level 8 이상만 필터링해서 보여주는 거야." 민재가 설명했다.

"이게 LLM 분석은 아닌데?" 정규가 지적했다.

"이건 **1단계 — 수집**이야. 2단계에서 LLM이 분석해."

```bash
# 2. 수집된 경보를 LLM에 전달하여 분류
# (OpsClaw의 invoke_llm을 사용 — DGX Spark의 Ollama)
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "command": "curl -s http://192.168.0.105:11434/v1/chat/completions -H \"Content-Type: application/json\" -d \"{\\\"model\\\":\\\"gemma3:12b\\\",\\\"messages\\\":[{\\\"role\\\":\\\"system\\\",\\\"content\\\":\\\"당신은 SOC 분석가입니다. 아래 Wazuh 경보를 true_positive/false_positive로 분류하세요. JSON으로 출력.\\\"},{\\\"role\\\":\\\"user\\\",\\\"content\\\":\\\"rule.level=10, description=Multiple authentication failures, srcip=185.220.101.33, count=23\\\"}]}\" 2>/dev/null | python3 -c \"import sys,json; r=json.load(sys.stdin); print(r[\\\"choices\\\"][0][\\\"message\\\"][\\\"content\\\"][:300])\"",
    "mode": "shell",
    "subagent_url": "http://localhost:8002"
  }'
```

결과:

```json
{
  "classification": "true_positive",
  "severity": "high",
  "technique": "T1110.001 - Brute Force: Password Guessing",
  "reasoning": "외부 IP(185.220.101.33)에서 23회 인증 실패. Tor exit node 대역으로 알려진 IP.",
  "recommendation": "IP 즉시 차단. SSH fail2ban 설정 확인."
}
```

"**Ollama의 gemma3:12b 모델**이 경보를 분석했어." 민재가 말했다. "외부 IP 185.220.101.33을 **Tor exit node**로 식별했어."

"gemma3:12b가 Tor IP 대역을 알고 있어?" 도윤이 놀랐다.

"학습 데이터에 보안 관련 정보가 포함되어 있으니까." 정규가 말했다. "하지만 IP가 현재도 Tor exit node인지는 **실시간 확인**이 필요해. LLM의 지식은 학습 시점 기준이거든."

"그래서 **CTI(위협 인텔리전스)**와 결합해야 해." 김영훈이 말했다. "LLM이 1차 분류하고, CTI 피드로 IP 평판을 **2차 검증**하는 구조."

### 실전 적용: 오후의 경보

오후 2시. LLM 분석 파이프라인이 돌아가는 동안, 실제 경보가 발생했다.

```
#bot-cc [OldClaw]
⚠️ LLM Alert Analysis: true_positive detected
Alert: rule.level=12, "Web server 500 error rate spike"
src: 10.20.30.80 (web)
LLM: "웹 서버의 500 에러가 급증. 가능한 원인: 1) DoS 공격, 2) 앱 버그,
      3) 백엔드 서비스 장애. 추가 확인 권고."
severity: high
```

"LLM이 **실시간으로** 경보를 분석해서 Slack에 알렸어!" 도윤이 말했다.

가람이 원인을 확인했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S tail -20 /var/log/apache2/error.log 2>/dev/null","subagent_url":"http://10.20.30.80:8002"}'
```

"JuiceShop의 DB 연결 풀이 소진됐어요. **앱 버그**예요, 공격이 아니에요."

"LLM의 분류는 **true_positive (high)**였지만, 실제로는 **앱 버그**야." 김영훈이 말했다. "이걸 어떻게 봐야 할까?"

"LLM이 **보수적으로** 분류한 거예요." 가람이 말했다. "500 에러 급증을 DoS 공격 가능성으로 봤으니까. 사람이 확인해서 앱 버그로 재분류하면 돼요."

"맞아. **보수적 오분류(conservative false positive)**는 받아들일 수 있어." 정규가 말했다. "위험한 건 반대 — 진짜 공격을 false_positive로 분류하는 거야."

"LLM이 '추가 확인 권고'라고 했잖아요." 도윤이 말했다. "자기도 확신이 없으니까 사람에게 넘긴 거예요."

"**좋은 LLM이란, 확신이 없을 때 '모르겠다'고 말하는 LLM**이야." 추박하가 만화책 뒤에서 말했다.

### 파이프라인 평가

하루 동안 LLM 분석 파이프라인이 처리한 결과:

```
[LLM 로그 분석 파이프라인 — 1일 운영 결과]

수집 경보:        287건
LLM 분석:         287건 (100%)
true_positive:     8건
suspicious:       12건
false_positive:   267건

사람 검증 결과:
  true_positive 8건 중 실제 위협: 3건 (정확도 37.5%)
  suspicious 12건 중 실제 위협: 1건 (정확도 8.3%)
  false_positive 267건 중 놓친 위협: 0건 (재현율 100%)

종합:
  정밀도(Precision): 4/20 = 20%   ← 낮음
  재현율(Recall): 4/4 = 100%     ← 완벽
  F1 Score: 0.33

분석 시간: LLM 2분 + 사람 검증 40분 = 42분
수동 분석 예상: 12시간+
→ 시간 절약: 약 94%
```

"**재현율 100%.** 실제 위협을 하나도 놓치지 않았어." 김영훈이 말했다. "정밀도가 20%로 낮지만, 이건 **오탐이 많다**는 뜻이지 **위험하다**는 뜻은 아니야."

"287건을 42분에 처리한 거잖아요." 도윤이 감탄했다. "제가 혼자 하면 12시간이에요."

"**사람의 시간을 94% 절약.** 이게 LLM 보안 도구의 가치야." 김영훈이 말했다.

가람이 적었다.

> `[LLM 로그 분석 — 교육 정리]`
>
> `파이프라인: 로그 수집 → LLM 분석 → 분류 → 인시던트/알림`
> `OpsClaw 기능 활용: Watcher(수집) + dispatch(분석) + Notification(알림)`
>
> `평가 지표:`
> `  정밀도(Precision): 탐지한 것 중 실제 위협의 비율`
> `  재현율(Recall): 실제 위협 중 탐지한 비율`
> `  → 보안에서는 재현율이 더 중요 (놓치면 안 되니까)`
> `  → "오탐이 많아도 괜찮지만, 미탐은 안 된다"`
>
> `LLM의 한계:`
> `  - IP 평판 정보가 학습 시점 기준 (실시간 CTI 필요)`
> `  - 앱 버그 vs DoS를 구분하지 못함 (도메인 지식 부족)`
> `  - 정밀도 20% → 오탐이 많음 → 프롬프트 개선 필요`
>
> `LLM의 강점:`
> `  - 재현율 100% → 위협을 놓치지 않음`
> `  - 시간 94% 절약 → 사람은 20건만 검증`
> `  - 지치지 않음 → 24시간 동작 가능`
>
> `"확신이 없을 때 '모르겠다'고 말하는 LLM이 좋은 LLM이다."`
