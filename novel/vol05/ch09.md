# Chapter 9: 프롬프트의 기술

---

목요일 오전 10시. 넥서스AI SOC 관제실.

어제 CTF의 여운이 남아 있었다. 관제실 분위기가 한결 부드러워졌다. 정규가 평소보다 3분 빨리 출근했고(기적이었다), 민재의 커피가 처음으로 "나쁘지 않다"는 평가를 받았다(도윤 기준).

"오늘부터 **AI 보안 도구**를 만든다." 김영훈이 말했다.

"만들어요? 우리가?" 도윤이 물었다.

"지금까지 우리는 **사람이** 로그를 읽고, 룰을 쓰고, 분석했어. 하지만 하루 경보가 137건이야. 오탐 134건을 걸러내려면 분석가가 밤새야 해."

"도윤 선배 얘기네요." 가람이 말했다.

"내 얘기 맞아." 도윤이 한숨을 쉬었다.

"LLM을 **보안 분석 도구**로 활용하면, 오탐 필터링을 자동화할 수 있어. 인간이 할 일은 LLM이 **놓친 것**만 확인하는 거야." 김영훈이 말했다.

"하지만 LLM도 완벽하지 않잖아요." 가람이 말했다. "Nexus-1처럼 탈옥도 되고."

"맞아. 그래서 핵심은 **프롬프트 엔지니어링**이야. LLM에게 **뭘 어떻게 시키느냐**가 결과의 질을 결정해."

---

### 프롬프트 엔지니어링 기초

정규가 교육 세션을 열었다.

```
[프롬프트 엔지니어링 for 보안 분석]

원칙 1: 역할 부여 (Role Assignment)
  나쁜 예: "이 로그를 분석해줘"
  좋은 예: "당신은 10년 경력의 SOC 분석가입니다.
           아래 로그에서 위협 징후를 MITRE ATT&CK T-code와 함께 분류해주세요."

원칙 2: 구조화된 출력 (Structured Output)
  나쁜 예: "결과를 알려줘"
  좋은 예: "결과를 JSON 형식으로 출력하되:
           {severity: high/medium/low, technique: T-code, description: 설명,
            evidence: 근거 로그 라인, recommendation: 대응 권고}"

원칙 3: 예시 제공 (Few-Shot)
  나쁜 예: 예시 없이 지시
  좋은 예: "예시: SSH 브루트포스 경보
           → {severity: medium, technique: T1110.001, description: '...', ...}
           이제 아래 경보를 같은 형식으로 분석해주세요."

원칙 4: 제약 조건 (Constraints)
  "확실하지 않은 내용은 '불확실'로 표시하세요."
  "외부 정보를 추측하지 말고, 주어진 로그 데이터만 사용하세요."
  "개인정보가 포함되어 있으면 마스킹 처리하세요."

원칙 5: 체인 오브 생각 (Chain of Thought)
  "먼저 로그의 출발지 IP를 확인하고,
   다음으로 요청 패턴을 분석하고,
   마지막으로 ATT&CK 기법과 매핑하세요.
   각 단계의 근거를 명시하세요."
```

"이 5가지 원칙을 OpsClaw의 **dispatch auto 모드**에 적용하면, LLM이 더 정확한 분석을 할 수 있어." 정규가 말했다.

### 실전: 보안 분석 프롬프트 설계

김영훈이 실제 Wazuh 경보 데이터를 가져왔다.

"지난주 경보 137건 중 실제 위협이었던 3건을 LLM이 찾아낼 수 있는지 테스트한다."

가람이 프롬프트를 설계했다. 처음으로 **프롬프트 엔지니어링**을 직접 해보는 것이었다.

```
[가람이 설계한 보안 분석 프롬프트 — v1]

당신은 넥서스AI SOC팀의 시니어 보안 분석가입니다.
아래의 Wazuh SIEM 경보 목록을 분석하여, 각 경보를 분류해주세요.

분류 기준:
  - true_positive: 실제 보안 위협 (즉시 대응 필요)
  - false_positive: 오탐 (무시 가능)
  - suspicious: 추가 조사 필요

출력 형식 (JSON 배열):
[
  {
    "alert_id": "경보 ID",
    "classification": "true_positive | false_positive | suspicious",
    "severity": "critical | high | medium | low",
    "technique": "MITRE ATT&CK T-code (해당 시)",
    "reasoning": "분류 근거 (한 문장)",
    "recommendation": "대응 권고 (한 문장)"
  }
]

제약:
  - 확실하지 않으면 "suspicious"로 분류
  - 추측하지 말고 경보 데이터만 근거로 사용
  - 개인정보(IP, 사용자명)는 그대로 출력 (내부 분석 목적)
```

"나쁘지 않은데." 정규가 가람의 프롬프트를 보며 고개를 끄덕였다. "하지만 **Few-Shot 예시**가 빠졌어."

가람이 수정했다.

```
[가람이 설계한 보안 분석 프롬프트 — v2 (Few-Shot 추가)]

(v1 내용 그대로)

예시:
  입력: {"alert_id": "001", "rule_id": 5712, "description": "SSHD brute force attempt",
         "src_ip": "203.0.113.50", "count": 847, "timestamp": "2026-04-15T02:30:00"}
  출력: {"alert_id": "001", "classification": "true_positive", "severity": "high",
         "technique": "T1110.001", "reasoning": "동일 IP에서 847회 SSH 인증 시도",
         "recommendation": "출발지 IP 즉시 차단, nftables에 DROP 룰 추가"}

이제 아래 경보 목록을 분석해주세요:
(137건의 경보 데이터)
```

---

OpsClaw를 통해 LLM에 분석을 요청했다.

```bash
# OpsClaw dispatch로 LLM 분석 실행
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "command": "cat /tmp/wazuh_alerts_137.json | python3 -c \"import sys,json; alerts=json.load(sys.stdin); print(f\\\"Total: {len(alerts)} alerts\\\")\"",
    "mode": "shell",
    "subagent_url": "http://localhost:8002"
  }'
```

LLM 분석 결과 (OpsClaw의 invoke_llm 경유):

```
[LLM 분석 결과 — 137건 경보]

true_positive:  5건  (실제 위협)
  - Alert 023: SSH 브루트포스 (T1110.001) — severity: high
  - Alert 067: 비인가 sudo 사용 (T1548.003) — severity: critical
  - Alert 089: 의심스러운 outbound DNS (T1071.004) — severity: medium
  - Alert 112: 파일 무결성 변경 (T1565.001) — severity: high
  - Alert 131: 비정상 cron 등록 (T1053.003) — severity: medium

false_positive: 127건  (오탐)
suspicious:      5건   (추가 조사 필요)
```

"**true_positive 5건.** 지난주 수동 분석에서 **3건**을 찾았는데, LLM은 **5건**을 찾았어." 도윤이 놀랐다.

"추가 2건은 뭐야?" 김영훈이 확인했다.

"Alert 112: 파일 무결성 변경. Alert 131: 비정상 cron 등록." 가람이 읽었다. "이 두 건은 수동 분석에서 **놓쳤던** 거예요."

"LLM이 사람보다 **더 많이** 찾았다?" 민재가 놀랐다.

"오탐 134건을 보면서 지치니까 3건만 찾은 거야." 도윤이 자기 반성했다. "LLM은 지치지 않으니까."

"하지만—" 정규가 말했다. "LLM이 찾은 5건이 **정말로 true_positive**인지는 사람이 확인해야 해. LLM도 오판할 수 있으니까."

"맞아." 김영훈이 고개를 끄덕였다. "LLM은 **1차 필터**야. 최종 판단은 사람이. 하지만 137건을 전부 보는 것과, LLM이 걸러준 **10건(5+5)**만 보는 것은 **업무량이 10배** 차이야."

### 프롬프트 개선 사이클

가람이 프롬프트를 더 개선했다. v3에서는 **Chain of Thought**를 추가했다.

```
[v3 — Chain of Thought 추가]

분석 절차:
  Step 1: 경보의 rule_id와 description을 확인하여 공격 유형을 판단
  Step 2: src_ip의 과거 이력을 확인 (내부/외부, 반복 여부)
  Step 3: 동일 시간대의 다른 경보와 상관관계(correlation) 분석
  Step 4: MITRE ATT&CK 매핑
  Step 5: 최종 분류 및 심각도 판단

각 Step의 근거를 "reasoning" 필드에 포함하세요.
```

v3로 재분석한 결과:

```
v2: true_positive 5건
v3: true_positive 4건, suspicious 2건 (Alert 131이 suspicious로 재분류)

v3가 더 신중한 판단을 함 — "비정상 cron이지만, 정규 유지보수일 수 있음"
→ Chain of Thought가 과잉 판단을 줄여줌
```

"**v3가 더 좋아.** 과잉 탐지를 줄였어." 정규가 말했다.

"프롬프트를 바꾸는 것만으로 분석 품질이 달라지네요." 가람이 감탄했다.

"그래서 **프롬프트 엔지니어링이 중요한** 거야." 김영훈이 말했다.

---

가람은 수첩에 적었다.

> `[프롬프트 엔지니어링 교육 정리]`
>
> `5원칙:`
> `  1. 역할 부여: "당신은 10년 경력의 SOC 분석가"`
> `  2. 구조화 출력: JSON, 표, 명확한 필드`
> `  3. Few-Shot: 예시 제공 → 기대 형식 학습`
> `  4. 제약 조건: "추측하지 마", "데이터만 근거로"`
> `  5. Chain of Thought: 단계별 사고 유도`
>
> `실험 결과:`
> `  수동 분석: 137건 → 3건 탐지 (22시간 소요)`
> `  LLM v2: 137건 → 5건 탐지 (2분 소요)`
> `  LLM v3: 137건 → 4건 + 2건 suspicious (2분 소요)`
> `  → LLM이 1차 필터 → 사람이 최종 판단`
> `  → 업무량 10배 감소`
>
> `[핵심 교훈]`
> `"LLM은 분석가를 대체하지 않는다.`
> ` 분석가를 10배 효율적으로 만든다."`
> `"프롬프트가 나쁘면 LLM도 나쁘다.`
> ` 프롬프트가 좋으면 LLM도 좋다.`
> ` 도구의 품질은 사용자의 지시에 달려 있다."`
