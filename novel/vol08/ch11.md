# Chapter 11: 클라우드 설계

---

목요일 아침. 6월 18일.

김영훈이 아침 회의에서 새로운 과제를 꺼냈다.

"증거를 모았어. genesis, IaC, 보안 점검 보고서, Nexus-1 평가. 이제 문제는 **이 증거들을 어떻게 보존하느냐**야."

"보존이요?" 가람이 물었다.

"법적 절차는 **몇 년**이 걸릴 수 있어." 김영훈이 말했다. "그 기간 동안 증거가 손상되거나 접근 불가능해지면 안 돼."

"현재 genesis는 이 건물의 서버에서 Docker 컨테이너로 돌아가고 있잖아요." 정규가 말했다. "화재, 물리적 접근, 장비 고장 — 리스크가 있어요."

"맞아." 김영훈이 말했다. "**증거 보존을 위한 클라우드 아키텍처**를 설계해야 해."

---

김영훈이 설계 원칙을 정리했다.

```
[증거 보존 클라우드 아키텍처 — 설계 원칙]

1. 불변성 (Immutability)
   - 저장된 데이터는 수정/삭제 불가
   - Write Once Read Many (WORM) 스토리지

2. 다중화 (Redundancy)
   - 최소 3개 리전에 복제
   - 단일 장애점(SPOF) 제거

3. 접근 통제 (Access Control)
   - 최소 권한 원칙 (Least Privilege)
   - 법무팀 + SOC팀 이중 승인 필요

4. 감사 추적 (Audit Trail)
   - 모든 접근/시도를 로깅
   - 로그 자체도 WORM 스토리지에 보존

5. 암호화 (Encryption)
   - 저장 시(at-rest): AES-256
   - 전송 시(in-transit): TLS 1.3
   - 키 관리: AWS KMS + HSM

6. 법적 준수 (Legal Compliance)
   - 데이터 보존 정책(Retention Policy) 설정
   - Legal Hold 기능 — 삭제 방지 잠금
   - 관할권 문제: 한국 리전 우선
```

"WORM 스토리지." 가람이 말했다. "한 번 쓰면 읽기만 가능한 건가요?"

"맞아." 김영훈이 말했다. "법적 증거 보존의 핵심이야. **아무도** 수정하거나 삭제할 수 없어. 관리자 권한으로도."

---

정규와 도윤이 아키텍처를 설계했다.

```
[증거 보존 클라우드 아키텍처 — 설계안]

┌─────────────────────────────────────────────┐
│                  접근 계층                     │
│  법무팀/SOC팀 → MFA + 이중 승인 → API Gateway  │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│              애플리케이션 계층                  │
│  OpsClaw Manager → 증거 조회/검증 API          │
│  (읽기 전용 — 쓰기는 초기 마이그레이션 시만)      │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│               스토리지 계층                    │
│                                              │
│  [Primary] S3 Glacier Deep Archive           │
│    - WORM (Object Lock, Compliance 모드)      │
│    - 리전: ap-northeast-2 (서울)              │
│    - Retention: 10년                          │
│                                              │
│  [Secondary] S3 Standard (읽기용 캐시)         │
│    - 자주 접근하는 evidence 캐싱               │
│    - Cross-Region: ap-northeast-1 (도쿄)      │
│                                              │
│  [Tertiary] Azure Blob (오프사이트)            │
│    - 다른 CSP에 추가 복제                      │
│    - Immutable Storage 정책                    │
│    - 리전: Korea Central                       │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│                보안 계층                       │
│  AWS KMS (키 관리) + CloudHSM                  │
│  CloudTrail (API 감사)                         │
│  GuardDuty (이상 탐지)                         │
│  Config (설정 변경 추적)                        │
│  → 모든 보안 로그도 WORM 스토리지에 보존          │
└─────────────────────────────────────────────┘

비용 추정 (월):
  S3 Glacier Deep Archive (100GB): ~$0.10
  S3 Standard 캐시 (10GB): ~$0.23
  Azure Blob (100GB): ~$1.80
  KMS + CloudTrail: ~$5.00
  총합: ~$10/월 (약 13,000원)
```

"월 **만 삼천 원**으로 법적 증거를 10년간 보존할 수 있어요?" 민재가 놀랐다.

"Glacier Deep Archive가 싸거든." 정규가 말했다. "접근 빈도가 낮은 데이터에 최적화된 스토리지야."

"하지만 **WORM이 핵심**이야." 김영훈이 말했다. "S3 Object Lock의 Compliance 모드를 사용하면, **AWS 루트 계정으로도** 데이터를 삭제할 수 없어."

"루트 계정으로도?" 가람이 놀랐다.

"Compliance 모드는 **보존 기간이 끝날 때까지** 누구도 삭제할 수 없어." 김영훈이 말했다. "법적 증거 보존의 골드 스탠다드야."

---

강혜윤이 법적 관점에서 아키텍처를 검토했다.

```
[법적 검토 — 클라우드 증거 보존]

1. Legal Hold (법적 보존 잠금)
   - 소송 예상 시 관련 데이터 삭제 금지
   - AWS S3 Legal Hold + Azure Legal Hold
   - 법무팀이 설정/해제 권한 보유

2. Chain of Custody (증거 관리 연속성)
   - 증거의 수집 → 보관 → 제출 과정을 문서화
   - 클라우드 이전 시 해시값 비교로 무결성 확인
   - "이 증거가 원본과 동일하다"를 증명

3. 관할권 (Jurisdiction)
   - 한국 법원 관할 → 한국 리전 우선
   - 해외 리전은 백업 목적
   - GDPR, 개인정보보호법 준수

4. 제3자 검증
   - 외부 감사인이 독립적으로 검증 가능한 구조
   - API 기반 검증 (8장에서 확인)
   - 감사인 전용 읽기 권한 IAM 역할
```

"Chain of Custody." 가람이 되뇌었다.

"증거 관리 연속성이에요." 강혜윤이 말했다. "누가, 언제, 어떻게 증거를 다뤘는지 전 과정이 기록되어야 해요. **빈틈이 있으면** 법정에서 증거 능력이 부정될 수 있어요."

"지금 genesis를 클라우드로 옮기면 그 과정도 기록해야 하는 거죠?" 가람이 물었다.

"맞아요." 강혜윤이 말했다. "마이그레이션 전 해시, 마이그레이션 과정 로그, 마이그레이션 후 해시 — 세 가지가 일치해야 해요."

---

김영훈이 마이그레이션 계획을 OpsClaw로 실행했다.

```bash
# 증거 보존 클라우드 마이그레이션 프로젝트
curl -X POST http://localhost:8000/projects \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $OPSCLAW_API_KEY" \
  -d '{
    "name": "evidence-cloud-preservation",
    "request_text": "genesis evidence 클라우드 보존 아키텍처 구축 및 마이그레이션",
    "master_mode": "external"
  }'
```

```bash
# 마이그레이션 전 무결성 기록
curl -X POST http://localhost:8000/projects/proj_cloud_001/plan \
  -H "X-API-Key: $OPSCLAW_API_KEY"
curl -X POST http://localhost:8000/projects/proj_cloud_001/execute \
  -H "X-API-Key: $OPSCLAW_API_KEY"

curl -X POST http://localhost:8000/projects/proj_cloud_001/execute-plan \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $OPSCLAW_API_KEY" \
  -d '{
    "tasks": [
      {
        "order": 1,
        "instruction_prompt": "sha256sum /var/lib/genesis/evidence.db > /tmp/pre_migration_hash.txt && cat /tmp/pre_migration_hash.txt && echo [Pre-migration hash recorded]",
        "risk_level": "low",
        "subagent_url": "http://localhost:8002"
      },
      {
        "order": 2,
        "instruction_prompt": "curl -s -H \"X-API-Key: $OPSCLAW_API_KEY\" http://localhost:8000/pow/verify?agent_id=genesis | jq \".valid, .blocks, .chain_hash\" && echo [Chain integrity verified before migration]",
        "risk_level": "low",
        "subagent_url": "http://localhost:8002"
      }
    ],
    "subagent_url": "http://localhost:8002"
  }'
```

"마이그레이션 전 상태를 **증거로 기록**하는 거예요." 김영훈이 말했다. "genesis의 증거를 보존하기 위한 작업 자체가, 다시 genesis에 기록되는 거야."

"메타 증거." 가람이 말했다. "증거를 보존하는 과정의 증거."

"맞아." 김영훈이 미소를 지었다.

---

추박하가 자리에서 일어나 창가로 갔다. 오후 햇살이 들어오고 있었다.

"genesis가 클라우드로 가는 거구나." 추박하가 말했다.

"복제본이에요." 김영훈이 말했다. "원본은 그대로 여기 있어요."

"20년간 이 건물 지하 서버실에 있었는데." 추박하가 말했다. "이제 세상 어딘가의 데이터센터에도 존재하게 되는 거야."

"안전하게 보존될 거예요." 정규가 말했다.

"알아." 추박하가 말했다. "그냥... **아이가 독립하는 기분**이야."

민재가 웃으려다 참았다. 추박하의 표정이 진지했으니까.
