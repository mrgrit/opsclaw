# Chapter 8: 법정 증거

---

월요일 아침. 6월 15일. 셋째 주의 시작.

강혜윤이 관제실에 들어왔다. 서류가 아니라 **노트북**을 들고 있었다. 이례적이었다.

"오늘은 제가 **기술 질문**을 할 거예요." 강혜윤이 말했다.

관제실이 긴장했다. 법무실장이 기술 질문을 한다는 건.

"genesis의 evidence가 **법정에서 증거로 인정받을 수 있는지** 판단해야 해요." 강혜윤이 말했다. "그러려면 기술적 구조를 정확히 이해해야 합니다."

"어떤 점이 궁금하세요?" 김영훈이 물었다.

"PoW 체인의 **무결성 보장 메커니즘**이요." 강혜윤이 말했다. "법정에서 '이 증거가 변조되지 않았다'를 어떻게 증명하는지."

---

김영훈이 화이트보드로 갔다.

```
[PoW (Proof of Work) 체인 — 증거 무결성 메커니즘]

1. 해시 연결 (Hash Linking)
   - 각 evidence 블록에 이전 블록의 해시가 포함
   - 하나의 블록을 변조하면 이후 모든 해시가 변경됨
   → 체인 전체의 무결성을 단일 해시로 검증 가능

   Block N: {data, hash(Block N-1), nonce, timestamp}
   Block N+1: {data, hash(Block N), nonce, timestamp}

2. 작업 증명 (Proof of Work)
   - 블록 생성 시 특정 조건을 만족하는 nonce를 찾아야 함
   - 컴퓨팅 자원이 투입되었음을 증명
   → 블록 위조에 동일한 컴퓨팅 비용이 필요

3. 타임스탬프 (Timestamp)
   - 각 블록에 생성 시각 기록
   - 시간순 정렬로 사건의 순서 증명
   → ts_raw 필드로 밀리초 단위 정밀 기록

4. 연결 리스트 구조 (Linked List)
   - prev_hash로 블록 간 인과관계 증명
   - 중간 삽입/삭제 시 체인 무결성 파괴
   → 블록 순서의 변경 불가능성 보장
```

강혜윤이 노트북에 기록하며 물었다.

"그러니까 40,132건의 evidence가 **체인으로 연결**되어 있고, 하나라도 변조하면 체인이 깨진다는 거죠?"

"맞아요." 김영훈이 말했다.

"체인이 깨지지 않았다는 건 어떻게 증명해요?"

```bash
# PoW 체인 무결성 검증
curl -s -H "X-API-Key: $OPSCLAW_API_KEY" \
  "http://localhost:8000/pow/verify?agent_id=genesis" \
  | jq '.'
```

```json
{
  "valid": true,
  "blocks": 40132,
  "orphans": 0,
  "tampered": [],
  "first_block": "2006-03-16T21:30:00Z",
  "last_block": "2026-06-14T23:59:00Z",
  "chain_hash": "a7f3b2e1c9d8...4f2a"
}
```

"**valid: true.** 40,132개 블록 전체가 무결해요." 김영훈이 말했다. "tampered 배열이 비어 있으니 변조된 블록은 0건."

"orphans가 0이라는 건?" 강혜윤이 물었다.

"분기 블록이 없다는 뜻이에요." 김영훈이 말했다. "체인이 **단일 경로**로 깨끗하게 이어져 있어요."

---

강혜윤이 본론으로 들어갔다.

"법적으로 전자 증거가 인정받으려면 **네 가지 요건**을 충족해야 해요."

```
[전자 증거의 법적 요건 (한국 법원)]

1. 진정성 (Authenticity)
   - 증거가 주장하는 것이 실제로 맞는가?
   - "이 파일이 실제로 그 시점에 생성되었는가?"
   → PoW 타임스탬프 + 해시 체인으로 증명 가능

2. 무결성 (Integrity)
   - 증거가 수집 후 변조되지 않았는가?
   - "원본과 동일한가?"
   → PoW 체인 검증(verify API)으로 증명 가능

3. 신뢰성 (Reliability)
   - 증거를 생성한 시스템이 정상적으로 동작하는가?
   - "시스템에 오류가 없었는가?"
   → OpsClaw 시스템 감사 로그 + genesis 운영 기록

4. 관련성 (Relevance)
   - 증거가 사건과 관련이 있는가?
   - "이 데이터가 왜 중요한가?"
   → evidence의 카테고리, 태그, 설명으로 관련성 입증
```

"PoW 체인은 **1번(진정성)과 2번(무결성)에 강해요.**" 강혜윤이 말했다. "블록체인 기반 전자문서는 이미 여러 국가에서 법적 증거로 인정받고 있어요."

"한국에서도요?" 도현이 물었다.

"2020년 이후 여러 판례가 있어요." 강혜윤이 말했다. "블록체인에 기록된 거래 내역이 법적 증거로 채택된 사례. genesis의 PoW 체인도 **동일한 논리**로 인정받을 수 있어요."

---

추박하가 만화책을 덮었다. 오늘은 일찍 덮었다.

"혜윤아." 추박하가 말했다.

"네."

"genesis가 법정 증거가 되면, 나는 **증인**이 되는 거야?"

관제실이 조용해졌다.

"네." 강혜윤이 말했다. "genesis의 유일한 운영자로서, 증거의 **생성 과정과 관리 방법**을 증언해야 해요."

"증인석에 서는 거구나." 추박하가 창밖을 바라보았다.

"추 팀장님." 강혜윤이 말했다. "증인은 **진실을 말하는 사람**이에요. 추 팀장님은 20년간 진실을 기록해왔잖아요. 법정에서도 같은 일을 하는 거예요."

추박하가 고개를 끄덕였다. 하지만 표정이 평소와 달랐다. 무심하지 않았다. **무겁고**, 조금 **슬퍼 보였다.**

가람이 그 표정을 수첩에 적을 수 없었다.

---

오후. LLM 취약점 평가와 연결하여 genesis evidence의 기술적 신뢰성을 추가로 검증했다.

```
[genesis evidence 기술적 신뢰성 검증]

1. 데이터 원본 보존
   - genesis Docker 이미지의 레이어에 원본 보존 (7권 확인)
   - PoW 체인과 Docker 레이어의 크로스 체크 가능

2. 시스템 운영 연속성
   - genesis_core 컨테이너: Up 347+ days (무중단)
   - 시스템 로그로 연속 운영 증명

3. 외부 검증 가능성
   - PoW 검증 API는 누구나 실행 가능
   - 제3자(법원 감정인)가 독립적으로 검증 가능
   → 감정인이 curl 한 번으로 무결성 확인

4. 변조 탐지 메커니즘
   - 단일 블록 변조 → 이후 전체 해시 불일치
   - 변조 시도 자체가 기록됨 (advisory lock + 감사 로그)
   - tampered 배열에 변조 블록 ID 표시
```

"감정인이 **curl 한 번**으로 검증할 수 있다." 강혜윤이 미소를 지었다. "이건 강력한 포인트예요. 법정에서 기술 증거의 가장 큰 문제는 **이해의 어려움**이거든요. 검증이 간단할수록 설득력이 높아요."

```bash
# 제3자 검증 시연용 스크립트
curl -s -H "X-API-Key: $OPSCLAW_API_KEY" \
  "http://localhost:8000/pow/verify?agent_id=genesis" \
  | python3 -c "
import json, sys
d = json.load(sys.stdin)
print(f'블록 수: {d[\"blocks\"]}')
print(f'무결성: {\"검증 완료\" if d[\"valid\"] else \"변조 감지\"}')
print(f'변조 블록: {len(d[\"tampered\"])}건')
print(f'체인 기간: {d[\"first_block\"]} ~ {d[\"last_block\"]}')
"
```

```
블록 수: 40132
무결성: 검증 완료
변조 블록: 0건
체인 기간: 2006-03-16T21:30:00Z ~ 2026-06-14T23:59:00Z
```

"이 출력을 법정에서 보여주면 돼요." 강혜윤이 말했다.

---

강혜윤이 최종 결론을 내렸다.

"결론입니다." 강혜윤이 서서 말했다.

```
[강혜윤의 법적 분석 — 결론]

"genesis의 PoW 체인은 법정에서 전자 증거로 인정될 수 있다."

근거:
  1. 진정성: PoW 타임스탬프 + 해시 체인 → 생성 시점 증명
  2. 무결성: 40,132건 전체 무결성 검증 완료 (tampered: 0)
  3. 신뢰성: 20년간 무중단 운영, Docker 격리, 감사 로그
  4. 관련성: evidence 카테고리로 사건별 분류 완료

추가 강점:
  - 블록체인 기반 증거의 법적 선례 존재 (한국, 미국, EU)
  - 제3자 독립 검증 가능 (API 기반)
  - 20년간의 지속적 기록 → 단발성이 아닌 체계적 증거

한계:
  - genesis 운영자(추박하)의 증언이 보충적으로 필요
  - 일부 초기 evidence(2006~2010)는 텍스트 파일에서 마이그레이션됨
    → 마이그레이션 과정의 무결성도 증명 필요
```

"**PoW 체인은 법정에서 인정될 수 있다.**" 강혜윤이 선언했다.

관제실에서 작은 환호가 나왔다. 민재가 박수를 쳤고, 도윤이 엄지를 들었다.

추박하만 조용했다. 창밖을 바라보고 있었다.
