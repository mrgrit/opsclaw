---

수요일 아침. 6월 17일.

정규가 아침에 조용히 손을 들었다. 정규가 먼저 발언하는 건 드문 일이었다. 관제실이 주목했다.

"genesis 외에 **증거가 될 수 있는 것**을 찾았습니다." 정규가 말했다.

"뭔데?" 김영훈이 물었다.

"**IaC 코드.**" 정규가 말했다. "Terraform state 파일과 Ansible playbook의 git 히스토리."

"IaC?" 가람이 물었다.

"Infrastructure as Code." 정규가 말했다. "인프라를 **코드로 관리**하는 방식이야. 서버, 네트워크, 보안 설정을 코드로 정의하고, 변경 이력이 git에 남아."

---

정규가 화이트보드에 IaC 개념을 정리했다.

```
[Infrastructure as Code (IaC) — 핵심 개념]

정의:
  인프라 구성을 코드(선언적/명령적)로 정의하고 관리하는 방식.
  수동 설정 대신 코드로 자동화 → 재현성, 일관성, 추적성

주요 도구:

1. Terraform (HashiCorp)
   - 선언적 방식 (HCL 언어)
   - 클라우드 리소스 프로비저닝 (AWS, GCP, Azure)
   - State 파일: 현재 인프라 상태를 JSON으로 기록
   - Plan → Apply → Destroy 워크플로우

2. Ansible (Red Hat)
   - 명령적/선언적 혼합 (YAML playbook)
   - 서버 설정 관리, 배포 자동화
   - Agentless — SSH 기반
   - Idempotent — 여러 번 실행해도 동일 결과

3. Kubernetes manifests (YAML)
   - 컨테이너 오케스트레이션 설정
   - Deployment, Service, ConfigMap 등

IaC 보안 위협:
  1. State 파일에 민감 정보 노출 (비밀번호, 키)
  2. 코드에 하드코딩된 비밀 (secret in code)
  3. 과도한 권한의 IAM 정책
  4. 미검증 모듈/플러그인 사용
  5. Drift — 코드와 실제 인프라의 불일치
```

"IaC의 장점은 **추적성**이에요." 정규가 말했다. "모든 변경이 git commit으로 남으니까."

"그리고 그게 **증거**가 된다는 거지." 김영훈이 말했다.

관제실이 얼어붙었다.

"**서문기가 직접 인프라를 변경했어요.**" 가람이 말했다.

"CloudTrail 로깅 비활성화." 김영훈이 말했다. "**감사 추적을 끄는** 행위야."

"KMS 키 삭제." 정규가 말했다. "training-data-key. **학습 데이터 암호화 키를 삭제**한 거예요. 암호화된 원본 데이터에 접근할 수 없게 만든 거죠."

"증거 인멸이잖아요." 진호가 말했다.

추박하가 만화책을 덮었다.

"서문기는 알고 있었어." 추박하가 말했다. "자진 신고가 올 수 있다는 걸. 그래서 **흔적을 지우기 시작한** 거야."

"언제부터요?" 가람이 물었다.

"2022년부터. genesis에도 기록이 있어. 서문기가 '리스크 정리'라는 이름으로 **증거 인멸 작업**을 시작한 시점."

---

김영훈이 IaC 변경 이력에서 보안 위반 패턴을 분류했다.

```
[서문기의 IaC 변경 — 보안 위반 패턴 분류]

패턴 1: 감사 추적 무력화
  - CloudTrail 비활성화 (2023-12)
  - VPC Flow Logs 삭제 (2024-01)
  - S3 접근 로그 비활성화 (2024-03)
  → 목적: 추후 조사 시 증거를 남기지 않기 위해

패턴 2: 데이터 접근 확대
  - S3 버킷 퍼블릭 접근 (2024-11)
  - IAM admin 권한 확대 (2024-09)
  - VPC 보안그룹 전체 허용 (2024-06)
  → 목적: 데이터 반출 경로 확보

패턴 3: 증거 인멸
  - KMS 키 삭제 (2023-08)
  - RDS 백업 보존 기간 축소 (2023-03)
  - ECS 보안 컨텍스트 제거 (2022-07)
  → 목적: 원본 데이터에 접근 불가능하게 만듦

발견 방법: git log + Terraform state 비교
법적 의미: 계획적 증거 인멸 → 가중 처벌 사유
```

"**계획적**이에요." 강혜윤이 들어오며 말했다. 언제 왔는지 모르게.

"패턴이 명확해요." 강혜윤이 말했다. "2022년부터 체계적으로 감사 추적을 무력화하고, 데이터 접근을 확대하고, 증거를 인멸했어요. **우발적 행위가 아니라 계획적 행위**예요."

"이건 genesis에 없는 증거인가요?" 가람이 물었다.

"genesis에도 서문기의 행동 기록이 있어." 추박하가 말했다. "하지만 **IaC git 히스토리는 genesis와 독립적인 증거**야. 서로를 교차 검증할 수 있어."

"**두 가지 독립적인 증거가 같은 것을 가리킨다.**" 강혜윤이 말했다. "이건 법정에서 매우 강력해요."

---

오후. IaC 보안 점검을 추가로 수행했다.

```bash
# IaC 보안 스캔 (Checkov)
curl -X POST http://localhost:8000/projects/proj_iac_evidence/execute-plan \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $OPSCLAW_API_KEY" \
  -d '{
    "tasks": [
      {
        "order": 1,
        "instruction_prompt": "cd /opt/nexus-infra && checkov -d . --framework terraform --compact --output json 2>/dev/null | jq \"{passed: .summary.passed, failed: .summary.failed}\" && echo [Checkov scan done]",
        "risk_level": "low",
        "subagent_url": "http://localhost:8002"
      },
      {
        "order": 2,
        "instruction_prompt": "cd /opt/nexus-infra && git log --all --diff-filter=D --name-only --pretty=format:\"%H %an %ad\" -- \"*.tf\" | head -30 && echo [Deleted Terraform files listed]",
        "risk_level": "low",
        "subagent_url": "http://localhost:8002"
      }
    ],
    "subagent_url": "http://localhost:8002"
  }'
```

```
[IaC 보안 스캔 결과]

Checkov 스캔:
  Passed: 127
  Failed: 43
  → 43건의 보안 위반 (서문기 변경분 포함)

삭제된 Terraform 파일:
  monitoring.tf — 서문기 삭제 (2023-12-10)
  encryption.tf — 서문기 삭제 (2023-08-15)
  audit-trail.tf — 서문기 삭제 (2024-01-20)
  → 보안 관련 설정 파일을 선택적으로 삭제
```

"보안 관련 .tf 파일만 **골라서** 삭제했네요." 민재가 말했다.

"서비스 코드는 안 건드리고 **보안 설정만** 지웠어." 정규가 말했다. "서비스가 중단되면 들키니까."

"영리한 범죄자." 진호가 말했다.

"하지만 git은 **삭제도 기록해.**" 정규가 말했다. "IaC의 아이러니야. 코드로 관리하면 효율적이지만, **모든 행위가 기록되는** 양날의 검."

---

추박하가 커피를 마시며 말했다.

"정규야."

"네."

"IaC 히스토리는 내가 몰랐던 거야." 추박하가 말했다. "genesis에는 서문기의 **행동**만 기록했지, **인프라 변경 코드**는 없어. 네가 찾은 거야."

정규가 살짝 놀랐다. 추박하가 인정하는 건 드문 일이었다.

"genesis가 완벽한 건 아니에요." 정규가 말했다. "genesis 밖에도 증거는 있습니다."

"그래." 추박하가 고개를 끄덕였다. "나 혼자서는 다 기록할 수 없었어. **팀이 필요한 이유**야."
