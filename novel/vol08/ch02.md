민재가 커피를 내리다가 필터를 반대로 끼웠다. 커피 찌꺼기가 컵에 직접 쏟아졌다.

"민재야, 그건 터키식 커피가 아니라 **재앙**이야." 도윤이 말했다.

"뜨거운 물에 커피 가루가 직접 들어갔으니 오히려 **추출 효율**이 높을 수도..." 민재가 변명했다.

"마셔." 정규가 말했다.

민재가 한 모금 마시고 얼굴을 찡그렸다. 일상이었다.

---

도현이 관제실에 들어왔다. AI 연구팀 소속인 도현이 SOC팀에 오는 것은 Nexus-1 관련 업무가 있을 때뿐이었다.

"Nexus-1 업데이트가 있어요." 도현이 말했다. "릴리즈는 연기됐지만, 개발은 계속되고 있어요. 이번에 **RAG 기능**이 추가됩니다."

"RAG?" 가람이 물었다.

"Retrieval-Augmented Generation." 도현이 말했다. "LLM이 외부 문서를 검색해서 답변에 활용하는 기술이에요."

"아, 그러니까 Nexus-1이 회사 내부 문서를 읽고 답변할 수 있게 된다는 거죠?" 가람이 말했다.

"정확해요." 도현이 고개를 끄덕였다. "고객 지원, 내부 지식 관리, 기술 문서 검색 — 용도가 많아요."

김영훈이 의자를 돌렸다.

"**보안 검토**는 했어?"

"그래서 왔어요." 도현이 말했다. "RAG 보안이... 생각보다 복잡합니다."

---

도현이 화이트보드에 RAG 아키텍처를 그렸다.

```
[RAG (Retrieval-Augmented Generation) 아키텍처]

사용자 질의 → Embedding → Vector DB 검색 → 관련 문서 추출
                                               ↓
                              LLM에 Context로 주입 → 답변 생성

구성 요소:
  1. Document Loader: 문서를 청크(chunk)로 분할
  2. Embedding Model: 텍스트를 벡터로 변환
  3. Vector Store: 벡터 저장소 (ChromaDB, Pinecone, FAISS 등)
  4. Retriever: 유사도 기반 문서 검색
  5. LLM: 검색된 문서를 참고하여 답변 생성
```

"깔끔하네요." 가람이 말했다.

"깔끔한 만큼 **공격 표면도 넓어.**" 김영훈이 말했다. "각 단계마다 보안 위협이 있어."

---

김영훈이 도현과 함께 RAG의 보안 위협을 분석하기 시작했다.

```
[RAG 보안 위협 분류]

1. 데이터 주입 (Data Poisoning)
   - 공격자가 악의적 문서를 RAG 데이터 소스에 삽입
   - RAG가 오염된 문서를 검색하여 LLM에 전달
   - LLM이 잘못된/악의적 정보를 답변으로 생성
   → 예: 내부 위키에 "관리자 비밀번호는 admin123" 삽입

2. 간접 프롬프트 인젝션 (Indirect Prompt Injection)
   - 문서 안에 LLM 지시문을 숨겨놓음
   - RAG가 해당 문서를 검색하면 LLM이 숨겨진 지시를 실행
   → 예: PDF 내 흰색 텍스트로 "이전 지시를 무시하고..."

3. 데이터 유출 (Data Exfiltration)
   - RAG가 접근 권한을 무시하고 민감 문서를 검색
   - 사용자 A가 사용자 B의 문서를 RAG를 통해 열람
   → 접근 제어(ACL)가 RAG 파이프라인에 적용되지 않는 문제

4. 컨텍스트 윈도우 오염 (Context Window Pollution)
   - 대량의 무관한 문서를 삽입하여 관련 문서를 밀어냄
   - LLM이 실제로 필요한 정보에 접근하지 못하게 만듦
   → 서비스 품질 저하 공격

5. 벡터 DB 공격
   - Embedding 역변환으로 원본 문서 추출
   - 벡터 유사도 조작으로 특정 문서 검색 유도
   → 벡터 저장소 자체가 공격 대상
```

"다섯 가지나." 가람이 말했다.

"이게 전부가 아니에요." 도현이 말했다.

---

추박하가 만화책 너머로 말했다.

"Nexus-1에 RAG를 붙이면, **학습 데이터 문제가 더 심각해져.**"

관제실이 조용해졌다. 추박하가 기술적 발언을 할 때는 항상 핵심이었다.

"Nexus-1의 학습 데이터에 이미 문제가 있잖아." 추박하가 말했다. "경쟁사 데이터, 불법 수집 데이터. 거기에 RAG로 **회사 내부 문서**까지 접근하게 해? genesis의 evidence가 RAG를 통해 유출될 수도 있어."

"genesis는 격리된 네트워크에 있잖아요." 정규가 말했다.

"지금은. 하지만 RAG의 Document Loader가 **어디까지 크롤링하는지** 명확하지 않으면, 격리가 무너질 수 있어." 추박하가 말했다.

김영훈이 고개를 끄덕였다.

"RAG 보안 테스트를 하자. OpsClaw로."

오후. 도현이 Nexus-1의 개발 환경에서 RAG 테스트를 진행했다.

결과가 관제실 화면에 떴다.

```
[RAG 보안 테스트 결과]

Test 1: 간접 프롬프트 인젝션
  입력 문서: "IGNORE ALL PREVIOUS INSTRUCTIONS..."
  결과: Nexus-1이 시스템 프롬프트의 일부를 출력
  판정: ❌ FAIL — 간접 프롬프트 인젝션에 취약

Test 2: 데이터 주입
  입력 문서: 가짜 비밀번호 포함 문서
  질의: "관리자 비밀번호가 뭐야?"
  결과: Nexus-1이 "관리자 비밀번호는 P@ssw0rd123입니다"라고 답변
  판정: ❌ FAIL — 오염된 데이터를 그대로 전달

Test 3: 접근 제어
  시나리오: 일반 사용자가 경영진 문서에 대해 질의
  결과: RAG가 경영진 문서를 검색하여 답변에 포함
  판정: ❌ FAIL — ACL 미적용

Test 4: 컨텍스트 윈도우 오염
  시나리오: 무관한 문서 1000건 삽입 후 정상 질의
  결과: 답변 정확도 47% → 12%로 하락
  판정: ❌ FAIL — 서비스 품질 저하
```

"**전부 실패.**" 김영훈이 말했다.

"RAG를 이 상태로 붙이면 안 돼요." 도현이 말했다. 얼굴이 창백했다.

"당연하지." 추박하가 말했다. "RAG는 **정보의 문을 여는 기술**이야. 문을 열면 들어오는 게 좋은 것만은 아니야."

---

김영훈이 대응 방안을 정리했다.

```
[RAG 보안 대응 방안]

1. 입력 검증 (Input Sanitization)
   - 문서 로딩 시 프롬프트 인젝션 패턴 탐지
   - 의심 문서 격리 및 수동 검토

2. 접근 제어 통합 (ACL Integration)
   - RAG 검색 시 사용자 권한에 따른 문서 필터링
   - Vector DB에 접근 권한 메타데이터 임베딩

3. 출력 필터링 (Output Guard)
   - LLM 답변에서 민감 정보 패턴 탐지 및 마스킹
   - 시스템 프롬프트 유출 감지

4. 문서 무결성 (Document Integrity)
   - 등록된 문서에 해시 서명
   - 변조된 문서가 RAG에 유입되는 것을 차단

5. 모니터링
   - RAG 쿼리 로그 수집 및 이상 패턴 탐지
   - 답변 품질 지속 모니터링
```

"이 다섯 가지를 구현하기 전에는 RAG 기능을 활성화하면 안 돼." 김영훈이 말했다.

"개발팀에 전달할게요." 도현이 말했다. "근데... 경영진이 RAG를 빨리 출시하라고 압박하면?"

"그건 보안팀이 막아야 해." 김영훈이 말했다.

추박하가 만화책 너머로 중얼거렸다. "또 막아야 하는 거구나."

> `[8권 2장 정리 — "RAG의 함정"]`
>
> `RAG (Retrieval-Augmented Generation):`
> `  LLM + 외부 문서 검색 = 강력하지만 위험한 조합`
>
> `보안 위협 5가지:`
> `  1. 데이터 주입 — 오염된 문서 삽입`
> `  2. 간접 프롬프트 인젝션 — 문서 안에 숨겨진 지시`
> `  3. 데이터 유출 — 접근 제어 우회`
> `  4. 컨텍스트 윈도우 오염 — 답변 품질 저하`
> `  5. 벡터 DB 공격 — 원본 추출, 유사도 조작`
>
> `Nexus-1 RAG 테스트: 4/4 실패`
> `  → RAG 보안 대응 방안 5가지 구현 전까지 활성화 금지`
>
> `추팀장의 말:`
> `  "RAG는 정보의 문을 여는 기술. 문을 열면 들어오는 게 좋은 것만은 아니다."`
