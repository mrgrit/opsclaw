# Chapter 9: 선언서

---

화요일 오전. 6월 16일.

강혜윤이 관제실에 다시 왔다. 이번에는 **빈 노트북**을 들고 있었다. 화면이 하얗게 비어 있었다.

"오늘 **SoA**를 작성합니다." 강혜윤이 말했다.

"SoA?" 가람이 물었다.

"Statement of Applicability." 강혜윤이 말했다. "**적용성 선언서**. ISO 27001에서 가장 중요한 문서 중 하나예요."

"어제는 법정 증거 분석이었고, 오늘은 ISO 문서요?" 민재가 물었다.

"연결돼 있어요." 강혜윤이 말했다. "자진 신고 서류에는 **현재의 보안 체계**를 증명하는 문서가 필요해요. SoA가 그 핵심이에요."

---

김영훈이 SoA의 개념을 설명했다.

```
[Statement of Applicability (SoA) — 적용성 선언서]

정의:
  ISO 27001 Annex A의 통제 항목 중 어떤 것을 적용했고,
  어떤 것을 제외했는지를 선언하는 문서.

구성:
  1. 통제 항목 ID (예: A.5.1)
  2. 통제 항목 설명
  3. 적용 여부 (적용 / 미적용)
  4. 적용 이유 또는 미적용 사유
  5. 구현 상태 (구현 완료 / 진행 중 / 미착수)
  6. 구현 방법 (간략 설명)

중요성:
  - ISO 27001 인증 심사의 핵심 문서
  - 조직이 어떤 보안 통제를 선택했는지를 한눈에 파악
  - 법적 분쟁 시 "보안 조치를 했는가?"의 증거
  - 리스크 평가 결과와 연계

예시:
  A.8.2 정보 분류: 적용 | 구현 완료 | 4단계 분류 체계 운영
  A.10.1 암호 통제: 적용 | 구현 완료 | AES-256, TLS 1.3
  A.14.2 보안 테스팅: 적용 | 진행 중 | 분기별 취약점 점검
```

"SoA는 **회사가 무엇을 하고 있고, 무엇을 안 하고 있는지** 선언하는 문서야." 김영훈이 말했다. "정직하게 작성해야 해."

"1장에서 봤듯이 넥서스AI의 과거 ISO 인증은 **부정**이었잖아요." 가람이 말했다. "이번에는 제대로 작성하는 거군요."

"그래. **이번에는 진짜**야." 김영훈이 말했다.

---

강혜윤이 SoA 작성의 법적 맥락을 설명했다.

"자진 신고 서류에 SoA를 포함하는 이유는 세 가지예요."

```
[SoA의 법적 역할 — 자진 신고 맥락]

1. 현재 상태의 정직한 선언
   - "우리는 이것을 하고 있고, 이것은 못 하고 있다"
   - 미적용 사항을 숨기지 않음 → 성실성 증명

2. 개선 로드맵의 기준선
   - SoA의 "미착수" 항목 = 향후 개선 대상
   - "우리는 부족한 점을 인지하고, 개선할 계획이 있다"

3. 과거와 현재의 대비
   - 2006년 SoA (부정 인증 시) vs 2026년 SoA (현재)
   - 20년간의 변화를 구조적으로 보여줌
```

"2006년 SoA가 있어요?" 정규가 물었다.

"genesis에 있어." 추박하가 말했다.

```bash
curl -s -H "X-API-Key: $OPSCLAW_API_KEY" \
  "http://localhost:8000/projects/genesis/evidence?category=audit_record&keyword=SoA_2006" \
  | jq '.items[0]'
```

```json
{
  "id": "EVD-2006-0320-009",
  "timestamp": "2006-03-20T11:00:00Z",
  "title": "ISO 27001 SoA — 2006년 버전",
  "summary": "서문기 작성. 93개 통제 항목 중 89개 '적용 및 구현 완료'로 표시.",
  "note": "실제 구현율은 30% 미만. 허위 SoA.",
  "risk_tag": "critical"
}
```

"**89개 적용 완료?**" 가람이 놀랐다. "실제로는 30%도 안 됐는데?"

"서문기가 **허위 SoA**를 만들어서 심사원에게 제출한 거야." 추박하가 말했다. "뇌물과 허위 문서. 이중 부정."

"그리고 genesis에 그 사실이 기록되어 있고." 강혜윤이 말했다. "이번에 우리가 작성하는 SoA는 **진짜**여야 해요. 거짓 없이."

---

SOC팀이 분담하여 현재 SoA를 작성하기 시작했다.

김영훈이 OpsClaw로 기존 보안 상태를 수집했다.

```bash

  -H "Content-Type: application/json" \
  -H "X-API-Key: $OPSCLAW_API_KEY" \
  -d '{
    "tasks": [
      {
        "order": 1,
        "instruction_prompt": "cat /etc/security/policy.conf 2>/dev/null || echo [No security policy file found]",
        "risk_level": "low",

      },
      {
        "order": 2,
        "instruction_prompt": "wazuh-control status 2>/dev/null && echo [Wazuh SIEM active] || echo [Wazuh not running on this host]",
        "risk_level": "low",
        "subagent_url": "http://192.168.208.152:8002"
      },
      {
        "order": 3,
        "instruction_prompt": "nft list ruleset 2>/dev/null | head -30 && echo [nftables rules collected]",
        "risk_level": "low",
        "subagent_url": "http://192.168.208.150:8002"
      },
      {
        "order": 4,
        "instruction_prompt": "cat /etc/bunkerweb/variables.env 2>/dev/null | grep -i security && echo [BunkerWeb WAF config collected]",

      }
    ],
```

각 팀원이 담당 영역을 작성했다.

```
[넥서스AI SoA 2026 — 주요 항목 발췌]

A.5.1 정보보안 정책
  적용: ✅  상태: 구현 완료
  방법: 정보보안 정책 문서 v3.2, 연 1회 검토
  담당: 김영훈

A.6.1 정보보안 조직
  적용: ✅  상태: 진행 중
  방법: SOC팀 + 법무팀 협업 체계 (AI 거버넌스 위원회 설립 예정)
  담당: 강혜윤

A.8.1 자산 관리
  적용: ✅  상태: 구현 완료
  방법: OpsClaw 자산 인벤토리 (Asset API)
  담당: 정규

A.9.1 접근 통제
  적용: ✅  상태: 구현 완료
  방법: RBAC + MFA, API Key 인증 (M28 패치)
  담당: 김영훈

A.12.4 로깅 및 모니터링
  적용: ✅  상태: 구현 완료
  방법: Wazuh SIEM + OpsClaw PoW 증적
  담당: 정규, 도윤

A.14.2 보안 테스팅
  적용: ✅  상태: 진행 중

  담당: 민재, 도현

A.18.1 법적 규정 준수
  적용: ✅  상태: 진행 중
  방법: EU AI Act 준수 평가 진행 중, 자진 신고 준비 중
  담당: 강혜윤

[미적용 항목 — 사유]
A.11.1 물리적 보안 구역: 미적용
  사유: 클라우드 전환 진행 중, 물리 서버룸 폐쇄 예정

A.17.1 사업 연속성: 부분 적용
  사유: DR 사이트 구축 진행 중 (목표: 2026 Q4)
```

추박하가 자리에서 일어났다. SoA 작성에 참여하지 않고 있던 추박하가.

"하나 추가해." 추박하가 말했다.

"뭐요?" 김영훈이 물었다.

"**genesis.**" 추박하가 말했다. "genesis도 SoA에 포함해야 해. 내부 감사 및 증적 관리 시스템으로."

"genesis를 **공식 통제 항목**에 넣는 건가요?" 가람이 물었다.

"genesis는 20년간 비공식이었어." 추박하가 말했다. "이제 공식으로 만들 때야."

강혜윤이 고개를 끄덕였다.

```
[추가 항목]
A.12.7 내부 감사 증적 시스템 (genesis)
  적용: ✅  상태: 구현 완료 (20년 운영)
  방법: OpsClaw PoW 체인 기반, 40,132건 evidence
  담당: 추박하
  비고: 기존 비공식 → 이번 SoA에서 공식 등록
```

"20년 만에 양지로 나오는 거네요." 가람이 말했다.

"빛이 싫은데." 추박하가 만화책을 펼쳤다.

---

오후. 2006년 허위 SoA와 2026년 현재 SoA를 나란히 놓았다.

```
[SoA 비교 — 2006 vs 2026]

                    2006 (허위)     2026 (현재)
통제 항목 수:        93              93
적용 항목:          89 (허위)       78 (실제)
구현 완료:          89 (허위)       61 (실제)
진행 중:            0               12
미착수:             0               5
미적용:             4               15

정직도:             ❌ (허위)        ✅ (실제)
```

"2006년에는 89개 구현 완료라고 **거짓말**했고, 2026년에는 61개 구현 완료라고 **솔직하게** 말한다." 강혜윤이 말했다.

"숫자는 2006년이 더 좋은데, 실제로는 2026년이 **압도적으로** 나은 거죠." 가람이 말했다.

"**정직한 61이 거짓된 89보다 강해요.**" 강혜윤이 말했다. "법정에서도 마찬가지예요."
