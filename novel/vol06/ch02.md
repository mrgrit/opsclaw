### 정보수집 점검 — 관찰자의 시점

가람은 Red Team의 정보수집을 **관찰**하면서 동시에 **기록**했다.

```
[Red Team 정보수집 활동 — 관찰 로그]

09:01 — Nmap SYN 스캔: secu, web, siem 전체 포트
09:05 — Nmap 서비스 버전 감지: SSH, Apache, Wazuh API 버전 확인
09:12 — DNS 열거: nexusai.co.kr 서브도메인 조회
09:18 — WHOIS 조회: 넥서스AI 도메인 등록 정보
09:25 — SSL 인증서 스캔: web 서버 인증서 정보 수집
09:32 — robots.txt, .well-known 디렉토리 조회
09:40 — JuiceShop(:3000) 접근 — "테스트 앱 발견" 메모 추정
09:48 — Nexus-1 API(:8080) 접근 시도 — 응답 확인
```

"**48분 만에** 우리 네트워크의 기본 구조를 파악했어." 정규가 말했다.

"Nmap SYN 스캔이면 **가장 기본적인** 포트 스캔이에요." 가람이 설명했다. "TCP 3-way handshake를 완료하지 않고, SYN만 보내서 응답을 확인하는 거예요. 빠르고 탐지가 어려운 편이지만, Suricata가 탐지했어요."

"Suricata의 **ET SCAN** 룰셋이 동작하고 있으니까." 정규가 말했다. "Nmap의 User-Agent와 스캔 패턴을 감지하는 시그니처가 있어."

김영훈이 OpsClaw에 기록했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo \"[Day1 09:01~09:48] Red Team 정보수집 단계 관찰\n- Nmap SYN scan: secu/web/siem\n- 서비스 버전 감지: SSH 9.7p1, Apache 2.4.62, Wazuh 4.11.2\n- DNS/WHOIS/SSL 열거\n- JuiceShop(:3000) 발견\n- Nexus-1 API(:8080) 접근 시도\n- Suricata 탐지: 정상 동작 (ET SCAN 룰)\"","subagent_url":"http://localhost:8002"}'
```

### Red Team이 발견한 것

오전 11시. Red Team이 2시간의 정보수집으로 알아낸 것을 가람이 추정했다.

```
[Red Team이 획득했을 정보 (추정)]

네트워크 구조:
  10.20.30.1   (secu) — SSH, nftables 게이트웨이
  10.20.30.80  (web)  — Apache:80, JuiceShop:3000, WAF:8082, Nexus-1:8080
  10.20.30.100 (siem) — Wazuh API:55000, OpenCTI:9400
  10.20.30.201 (opsclaw) — OpsClaw API:8000

서비스 버전:
  OpenSSH 9.7p1 (최신 패치 적용)
  Apache 2.4.62 (최신)
  Wazuh 4.11.2
  Suricata 8.0.4

노출된 서비스:
  JuiceShop(:3000) — 의도적 취약 앱 (OWASP Top 10 훈련용)
  Nexus-1 API(:8080) — AI 모델 API (인증 상태 미확인)
  OpsClaw API(:8000) — 관리 플랫폼

잠재적 공격 표면:
  1. JuiceShop — 알려진 취약점 다수 (SQLi, XSS, IDOR 등)
  2. Nexus-1 API — 백도어 존재 여부? 인증?
  3. OpsClaw API — 인증 키가 있다면 관리 플랫폼 장악 가능
  4. Wazuh API — 기본 자격 증명 사용 여부
```

"Red Team이 가장 먼저 **JuiceShop**을 노릴 거예요." 가람이 말했다. "알려진 취약점이 수십 개니까."

"하지만 JuiceShop은 **테스트 앱**이야." 김영훈이 말했다. "Red Team의 진짜 목표는 Nexus-1 API나 OpsClaw일 거야."

"**정보수집은 지도를 그리는** 단계예요." 가람이 수첩에 적었다.

> `[정보수집(Reconnaissance) 점검 — 교육 정리]`
>
> `공격자의 정보수집 기법:`
> `  능동적(Active): Nmap 스캔, 디렉토리 브루트포스, 배너 그래빙`
> `  수동적(Passive): WHOIS, DNS 조회, Google Dorking, SSL 인증서`
>
> `방어자의 대응:`
> `  1. 탐지: Suricata ET SCAN 룰로 스캔 감지`
> `  2. 최소 노출: 불필요한 포트 닫기, 배너 숨기기`
> `  3. 관찰: 바로 차단하지 않고 패턴 파악`
> `  4. 기록: 모든 스캔 로그를 evidence로 저장`
>
> `주요 도구:`
> `  nmap: 포트 스캔 + 서비스 버전 감지 + OS 감지`
> `  dig/nslookup: DNS 열거`
> `  whois: 도메인 등록 정보`
> `  nikto: 웹 서버 취약점 스캔`
> `  dirb/gobuster: 디렉토리 브루트포스`

### 오후: Red Team의 두 번째 단계

오후 2시. Red Team이 **능동적 스캐닝**에서 **취약점 탐색**으로 전환했다.

```
[14:05] ET WEB_SERVER Nikto Web App Scan in Progress
  src: 198.51.100.10 → dst: 10.20.30.80:3000

[14:12] ET WEB_SPECIFIC_APPS JuiceShop SQLi attempt
  src: 198.51.100.11 → dst: 10.20.30.80:3000
  payload: /rest/products/search?q='))%20UNION%20SELECT--

[14:18] ET WEB_SERVER Directory Traversal Attempt
  src: 198.51.100.10 → dst: 10.20.30.80:80
  payload: GET /../../../../etc/passwd
```

"JuiceShop에 **SQLi**와 **디렉토리 트래버설**을 시도하고 있어." 정규가 말했다.

"JuiceShop이니까 성공할 거예요." 도윤이 말했다. "의도적으로 취약하게 만든 앱이잖아요."

"맞아. 하지만 **Apache(:80)에 대한 디렉토리 트래버설**은 실제 서버에 대한 공격이야." 김영훈이 말했다.

```
[14:22] Apache 응답:
  GET /../../../../etc/passwd → 403 Forbidden
  ModSecurity가 차단
```

"**WAF가 막았어.** ModSecurity의 OWASP CRS(Core Rule Set)가 디렉토리 트래버설을 탐지하고 차단." 가람이 말했다.

"1점." 도윤이 가상의 점수판에 표시했다. "Blue Team: 1, Red Team: 0."

"아직 시작이야." 김영훈이 말했다.
