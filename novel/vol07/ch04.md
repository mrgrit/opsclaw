### Docker 네트워크 유형

민재가 커피를 들고 왔다. 오늘은 드립백이었다. 드립백 포장을 뜯다가 필터가 찢어졌고, 민재는 찢어진 필터를 컵 위에 올려놓고 그 위에 물을 부었다. 커피 가루가 섞인 탁한 액체가 컵에 담겼다.

"민재야, 그건 커피가 아니라 하수야." 도윤이 말했다.

"맛은 똑같아." 민재가 한 모금 마시고 미간을 찌푸렸다. "...거의."

가람이 웃음을 참으며 정규에게 물었다. "선배, Docker 네트워크 유형을 정리해주실 수 있어요? 보고서에 genesis_net이 왜 특별한지 설명하려면 비교가 필요해서요."

정규가 화이트보드에 적었다.

```
[Docker 네트워크 유형]

1. bridge (기본)
   - Docker 설치 시 자동 생성되는 기본 네트워크
   - 같은 bridge 안의 컨테이너끼리 IP로 통신 가능
   - 호스트와는 포트 매핑(-p)으로 통신
   - 예: docker run --network bridge nginx
   ⚠ 기본 bridge는 모든 컨테이너가 공유 → 격리 약함

2. host
   - 컨테이너가 호스트의 네트워크를 직접 사용
   - 포트 매핑 불필요, 성능 좋음
   - 하지만 격리가 전혀 없음 → 보안 취약
   - 예: docker run --network host nginx
   ⚠ 컨테이너가 호스트의 모든 포트에 접근 가능

3. none
   - 네트워크 인터페이스 없음 (loopback만)
   - 완전한 네트워크 격리
   - 하지만 외부 통신이 필요한 서비스에는 사용 불가
   - 예: docker run --network none alpine
   ✓ 보안은 최고, 하지만 실용성 제한

4. overlay
   - 여러 Docker 호스트를 하나의 논리 네트워크로 연결
   - Docker Swarm / Kubernetes 환경에서 사용
   - 예: docker network create -d overlay my_overlay
   ⚠ 멀티호스트 통신 → 공격 표면 넓음

5. macvlan
   - 컨테이너에 물리 네트워크의 MAC/IP 할당
   - 호스트와 같은 서브넷에 위치하는 것처럼 보임
   - 레거시 시스템 연동에 유용
   ⚠ 물리 네트워크에 직접 노출 → 방화벽 필수
```

"genesis_net은 **사용자 정의 bridge**인데, Internal 모드와 ICC 비활성화를 조합해서 **none에 가까운 수준의 격리**를 달성하면서도, docker exec를 통한 관리 접근은 유지한 거야." 정규가 설명했다.

"**none을 쓰면 되지 않았나요?**" 가람이 물었다.

"genesis가 OpsClaw Manager의 내부 API 호출을 받아야 해서." 추박하가 말했다. "완전한 none으로는 Manager와의 통신이 불가능하거든. Internal bridge가 **정확히 필요한 만큼만 열어두는** 설계야."

### 네트워크 격리 강화

김영훈이 이사회 보고서에 넣을 genesis 네트워크 보안 요약을 작성했다.

```bash
# genesis 네트워크 격리 현황 보고서 생성
curl -X POST http://localhost:8000/projects/genesis-audit/dispatch \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $OPSCLAW_API_KEY" \
  -d '{
    "command": "docker network inspect genesis_net -f \"{{.Internal}} {{.Options}}\"",
    "subagent_url": "http://localhost:8002"
  }'
```

```json
{
  "status": "success",
  "output": "true map[com.docker.network.bridge.enable_icc:false com.docker.network.bridge.enable_ip_masquerade:false]",
  "exit_code": 0
}
```

"Internal: true, ICC: false, IP Masquerade: false. **삼중 잠금.**" 김영훈이 보고서에 적었다.

추박하가 한 가지를 덧붙였다.

"genesis_net을 만들 때, 나는 **iptables 규칙도 추가**했어."

```bash
# 호스트의 iptables에서 genesis_net 관련 규칙 확인
sudo iptables -L DOCKER-ISOLATION-STAGE-2 -n -v 2>/dev/null | grep 172.28
```

```
   0     0 DROP       all  --  br-genesis   *       172.28.0.0/16        0.0.0.0/0
   0     0 DROP       all  --  *            br-genesis  0.0.0.0/0       172.28.0.0/16
```

"genesis_net(172.28.0.0/16)에서 나가는 트래픽과 들어오는 트래픽 **모두 DROP.** Docker의 Internal 설정 위에 **iptables 레벨에서도 이중으로 차단**한 거야." 추박하가 말했다.

"집착이시네요." 민재가 탁한 커피를 마시며 말했다.

"20년간 지켜야 했으니까." 추박하가 대답했다. 그 말에는 가벼움이 없었다.

가람은 수첩에 적었다.

> `[Docker 네트워크 보안 — 격리 설계]`
>
> `Docker 네트워크 유형 5가지:`
> `  bridge: 기본, 격리 약함 (같은 bridge 내 컨테이너끼리 통신 가능)`
> `  host: 격리 없음 (호스트 네트워크 직접 사용) → 보안 위험`
> `  none: 완전 격리 (네트워크 없음) → 실용성 제한`
> `  overlay: 멀티호스트 (Swarm/K8s) → 공격 표면 넓음`
> `  macvlan: 물리 네트워크 노출 → 방화벽 필수`
>
> `genesis_net 설계 (사용자 정의 bridge + 보안 강화):`
> `  Internal: true → 외부 아웃바운드 차단`
> `  ICC: false → 컨테이너 간 통신 차단`
> `  IP Masquerade: false → NAT 우회 차단`
> `  + iptables DROP 규칙 → 이중 차단`
>
> `교훈:`
> `  "네트워크 격리는 컨테이너 보안의 첫 번째 계층"`
> `  하지만 네트워크 격리만으로는 부족 → 물리적 접근 통제 필요`
> `  서문기가 원격 접근 실패 후 물리 접근을 시도한 것이 증거`
> `  → Defense in Depth: 네트워크 + 물리 + 인증 + 격리 = 다계층 방어`
