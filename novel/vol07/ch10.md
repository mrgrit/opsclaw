### K8s 기초 강의

김영훈이 화이트보드 앞에 섰다. "K8s 전환 얘기가 나왔으니, 기초부터 정리하자. 모르는 사람?"

도윤이 손을 들었다. 민재도 들었다. 가람도 들었다. 진호가 망설이다가 들었다.

"정규는?" 김영훈이 물었다.

"CKA 자격증 있습니다." 정규가 무표정하게 말했다.

"... 언제?" 김영훈이 놀랐다.

"작년에. 심심해서."

민재가 중얼거렸다. "심심해서 CKA를 따는 사람이 세상에 또 있나."

김영훈이 헛기침을 하고 시작했다.

"Kubernetes는 **컨테이너 오케스트레이션 플랫폼**이야. Docker가 **하나의 컨테이너**를 관리한다면, K8s는 **수백, 수천 개의 컨테이너**를 관리해."

```
[Kubernetes 핵심 개념]

1. Pod
   - K8s의 최소 실행 단위
   - 하나 이상의 컨테이너를 포함
   - 같은 Pod 내 컨테이너는 네트워크/스토리지 공유
   - 예: genesis 컨테이너 + 로그 수집 사이드카

2. Service
   - Pod에 대한 안정적인 네트워크 엔드포인트
   - Pod가 죽고 다시 생겨도 Service IP는 유지
   - 종류: ClusterIP(내부), NodePort(외부), LoadBalancer

3. Deployment
   - Pod의 선언적 관리 — "이 Pod를 3개 유지하라"
   - 롤링 업데이트: 무중단 배포
   - 롤백: 이전 버전으로 즉시 복구

4. Namespace
   - 논리적 격리 — 같은 클러스터 내에서 환경 분리
   - 예: default, kube-system, soc-prod, soc-dev

5. RBAC (Role-Based Access Control)
   - 누가 어떤 리소스에 접근할 수 있는지 제어
   - Role → RoleBinding → ServiceAccount
   - 최소 권한 원칙 적용
```

"Docker Compose에서는 `docker-compose.yml` 하나로 관리했잖아." 김영훈이 말했다. "K8s에서는 **YAML 매니페스트**로 모든 것을 선언해."

```yaml
# genesis-deployment.yaml (예시)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: genesis
  namespace: soc-secure
  labels:
    app: genesis
    security-level: critical
spec:
  replicas: 1          # genesis는 1개만 — 데이터 정합성
  selector:
    matchLabels:
      app: genesis
  template:
    metadata:
      labels:
        app: genesis
    spec:
      containers:
      - name: genesis-core
        image: genesis:20.1.0
        volumeMounts:
        - name: genesis-data
          mountPath: /data
        securityContext:
          readOnlyRootFilesystem: true
          runAsNonRoot: true
          allowPrivilegeEscalation: false
      volumes:
      - name: genesis-data
        persistentVolumeClaim:
          claimName: genesis-pvc
```

"이렇게 정의하면 K8s가 **알아서 Pod를 생성하고, 죽으면 다시 만들어.**" 김영훈이 말했다. "genesis가 크래시해도 자동 복구되는 거야."

"좋은 것 같은데요?" 도윤이 말했다.

"좋은 것만은 아니야." 정규가 끼어들었다.

### K8s의 양날의 검

정규가 화이트보드의 왼쪽에 **장점**, 오른쪽에 **단점**을 적었다.

```
[K8s로 genesis를 운영하면?]

장점:
  ✅ 자동 복구 (Pod 크래시 → 즉시 재시작)
  ✅ 선언적 관리 (YAML로 인프라 상태 정의)
  ✅ 네임스페이스 격리 (soc-secure 네임스페이스에 분리)
  ✅ RBAC으로 접근 제어 (서문기 접근 차단 가능)
  ✅ 감사 로그 (K8s audit log로 모든 API 호출 기록)

단점:
  ⚠️ 공격 표면 증가 — K8s API 서버, etcd, kubelet 등 새로운 타겟
  ⚠️ etcd에 시크릿 노출 — 기본적으로 etcd는 암호화되지 않음
  ⚠️ 네트워크 복잡성 — CNI, Service Mesh, Ingress...
  ⚠️ 권한 관리 복잡 — RBAC 실수 하나로 전체 클러스터 접근 가능
  ⚠️ 운영 복잡도 — Docker 단독 대비 학습 곡선이 가파름
```

"genesis는 현재 Docker 하나로 **완벽하게 격리**되어 있어." 정규가 말했다. "포트 노출 없음, 전용 네트워크, 추팀장만 접근 가능. **단순하지만 견고해.**"

"K8s로 옮기면 **자동 복구**라는 장점이 생기지만, 동시에 **API 서버라는 새로운 공격 경로**가 생겨." 김영훈이 말했다. "K8s API 서버가 노출되면, `kubectl`로 모든 Pod를 제어할 수 있거든."

```bash
# K8s API 서버가 노출되면 이런 일이 가능
# (주의: 교육용 예시, 실행 금지)
kubectl get secrets --all-namespaces
kubectl exec -it genesis-pod -- /bin/sh
kubectl delete deployment genesis --namespace soc-secure
```

"**세 줄로 genesis를 삭제할 수 있다?**" 가람이 놀랐다.

"RBAC이 제대로 설정되지 않으면." 정규가 말했다. "그리고 etcd에는 K8s의 **모든 상태**가 저장돼. 시크릿, 설정, 인증서 — etcd를 빼내면 클러스터 전체를 장악할 수 있어."

김영훈이 고개를 끄덕였다. "동의합니다. genesis는 현재 Docker 환경을 **유지**합니다. OpsClaw 본체만 나중에 K8s 전환을 검토하죠."

"그래." 추박하가 만화책을 다시 들었다. "새 기술이 항상 더 안전한 건 아니야."

정규가 작게 중얼거렸다. "CKA 따서 써먹으려고 했는데."

도윤이 옆에서 속삭였다. "**심심해서** 딴 거라며."

### K8s 교육 마무리

김영훈이 마지막으로 정리했다.

"K8s 자체를 안 쓰겠다는 게 아니야. OpsClaw의 다른 서비스 — Manager API, SubAgent, 스케줄러 — 이런 것들은 K8s로 옮기면 장점이 많아."

```
[K8s 전환 전략]

OpsClaw 서비스:
  ✅ manager-api → K8s (Deployment, replicas=2)
  ✅ subagent-runtime → K8s (DaemonSet, 각 노드에 배포)
  ✅ scheduler-worker → K8s (CronJob)
  ✅ web-ui → K8s (Deployment + Ingress)
  ❌ genesis → Docker 유지 (증거 보전)

보안 요구사항:
  - NetworkPolicy로 Pod 간 통신 제어
  - PodSecurityPolicy (또는 Pod Security Standards)
  - etcd 암호화 (encryption at rest)
  - RBAC 최소 권한 원칙
  - 감사 로그 활성화
```

"K8s를 쓰면 **자동화와 복원력**이 올라가. 하지만 **공격 표면도 올라간다.** 보안 설정 없이 K8s를 도입하는 건, **방화벽 없이 서버를 인터넷에 노출하는** 것과 같아."
