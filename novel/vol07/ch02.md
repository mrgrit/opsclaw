# Chapter 2: 이미지 속의 비밀

---

월요일 오후. 이사회 보고까지 **2일.**

김영훈이 genesis Docker 이미지를 **레이어별로** 분석하기 시작했다.

"Docker 이미지는 **레이어 구조**야." 김영훈이 가람에게 설명했다. "각 레이어는 Dockerfile의 한 줄에 해당해. 파일을 추가하거나, 명령을 실행하거나, 환경 변수를 설정하는 각 단계가 하나의 레이어로 저장돼."

```bash
# genesis 이미지 히스토리 — 레이어 목록
docker history genesis:20.1.0 --no-trunc --format "{{.CreatedAt}} | {{.Size}} | {{.CreatedBy}}"
```

```
2026-01-15 | 42MB | COPY /app /app
2026-01-15 | 1.2KB | ENV GENESIS_DB_PATH=/data/genesis.db
2026-01-15 | 0B | VOLUME /data
2025-06-20 | 38MB | COPY /app /app
2024-12-01 | 35MB | COPY /app /app
2024-06-15 | 31MB | COPY /app /app
2023-12-01 | 28MB | COPY /app /app
...
2014-06-01 | 5MB | COPY /app /app  ← Docker 최초 이미지 (2006년 코드를 Docker화)
2014-06-01 | 245MB | FROM python:3.6-slim
```

"**이미지가 2014년 Docker 도입 이후 12년간 업데이트**되어 왔어." 가람이 놀랐다. "2014년 Docker 도입부터 2026년까지. 물론 genesis 자체는 2006년부터 존재했지만, Docker 이미지의 역사는 2014년부터야."

"각 `COPY /app /app`이 **코드 업데이트**야." 김영훈이 말했다. "추팀장이 genesis의 코드를 업데이트할 때마다 새 레이어가 추가된 거지."

"레이어가 **삭제되지 않고 쌓여** 있으니까, 과거 버전의 코드를 **복원**할 수 있어요?" 가람이 물었다.

"맞아. 이게 Docker 이미지의 **포렌식적 가치**야." 정규가 끼어들었다. "이미지를 삭제하지 않는 한, **모든 역사가 레이어에 남아.**"

### 이미지 보안 분석

김영훈이 보안 관점에서 이미지를 분석했다.

```bash
# 이미지 보안 스캐닝 — Trivy
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy:latest image genesis:20.1.0 --severity HIGH,CRITICAL 2>/dev/null | tail -20
```

```
genesis:20.1.0 (debian 12.5)
Total: 3 (HIGH: 2, CRITICAL: 1)

CRITICAL:
  CVE-2024-38475 - Apache HTTP Server (libapache2-mod-security2)
  → 하지만 genesis는 Apache를 사용하지 않음 → 실질적 영향 없음

HIGH:
  CVE-2024-6387 - OpenSSH (regreSSHion)
  → genesis 컨테이너에 SSH 없음 → 영향 없음

  CVE-2024-26581 - Linux Kernel
  → 컨테이너는 호스트 커널을 공유 → 호스트 패치 필요
```

"**Critical 1건, High 2건.** 하지만 실질적 영향은 **커널 취약점 1건**뿐이야." 김영훈이 말했다.

"호스트의 커널은 4권에서 패치를 **연기**했잖아요." 가람이 기억했다. "정태식 본부장이 Nexus-1 릴리즈 후로."

"아직 패치 안 됐을 수도 있어." 김영훈이 말했다. "확인하자."

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"uname -r","subagent_url":"http://localhost:8002"}'
```

```
6.8.0-106-generic
```

"**6.8.0-106.** 아직 구 버전이야. 115로 업데이트해야 해."

"genesis를 보호하려면 **호스트 커널도** 패치해야 한다." 정규가 말했다. "컨테이너의 격리는 커널에 의존하니까."

### Docker 이미지 레이어 포렌식

가람이 과거 레이어를 분석하는 방법을 배웠다.

```bash
# 특정 레이어의 내용 확인
# 2014년 최초 Docker 레이어 (2006년 코드를 Docker화)
docker save genesis:20.1.0 | tar -x -C /tmp/genesis_layers/
ls /tmp/genesis_layers/
```

"이미지를 **tar로 풀면** 각 레이어가 별도 디렉토리로 나와." 정규가 설명했다. "가장 오래된 레이어에 **2014년에 Docker화한 genesis 코드**가 있어. 2006~2014년의 원본 스크립트는 Docker 이전이라 이미지 안에는 없지만, genesis의 /data 볼륨에 원본 SQLite 백업이 보관되어 있어."

```bash
# 최초 Docker 레이어(2014)의 /app 디렉토리 — 코드 자체는 2006년 작성
ls /tmp/genesis_layers/<oldest_layer>/app/
```

```
daemon.sh
config.ini
genesis_db.py
evidence_writer.py
pow_chain.py
deadman_switch.py
README.md
```

"**2006년의 genesis 코드.** 6개 파일." 김영훈이 읽었다.

"README.md를 봐." 추박하가 말했다.

```bash
cat /tmp/genesis_layers/<oldest_layer>/app/README.md
```

```
# Genesis — Operational Record System
# Created: 2006-03-15
# Author: 박하 (추박하)
#
# 이 시스템은 넥서스AI(당시 NexusSec)의 운영 작업을
# 기록하고, 위변조 불가능하게 보존하기 위해 만들어졌다.
#
# 모든 기록은 SHA-256 해시 체인으로 연결된다.
# 기록은 삭제할 수 없다. 수정할 수 없다.
# 기록은 진실이다.
#
# "evidence 없는 완료 주장은 거짓말이다."
```

관제실이 고요해졌다.

"2006년의 추팀장이 쓴 README." 김영훈이 말했다.

"2006년에 쓴 README가 2014년 Docker 이미지에 그대로 복사된 거야." 정규가 말했다. "genesis의 **철학은 2006년부터**, 하지만 **Docker 컨테이너화는 2014년부터.**"

"**'evidence 없는 완료 주장은 거짓말이다.'**" 가람이 반복했다. "1권에서 추팀장님이 한 말과 **같아요.**"

"20년 전부터 같은 원칙이었어." 추박하가 말했다. "기록은 진실이다. 그때도, 지금도."

### 2006년 코드 vs 2026년 코드

정규가 최초 코드(2006)와 현재 코드(2026)를 비교했다.

```
[genesis 코드 진화 — 2006(스크립트) → 2014(Docker) → 2026(현재)]

2006년 (6개 파일 — 순수 Python, cron 기반):
  daemon.sh — 메인 데몬 (cron으로 주기 실행)
  config.ini — 설정 파일
  genesis_db.py — SQLite 기반 evidence 저장
  evidence_writer.py — evidence 생성
  pow_chain.py — SHA-256 해시 체인
  deadman_switch.py — 30일 미접근 시 자동 전송

2026년 (23개 파일):
  daemon.sh — 메인 데몬 (systemd 서비스)
  config.yaml — YAML 설정
  genesis_db.py — PostgreSQL 연동
  evidence_writer.py — evidence + PoW 블록 생성
  pow_chain.py — SHA-256 + nonce mining (difficulty=4)
  deadman_switch.py — 30일 + 외부 백업 + 암호화
  scheduler.py — 정기 수집 스케줄
  collector_email.py — 이메일 모니터링
  collector_network.py — 네트워크 트래픽 수집
  collector_file.py — 파일 변경 감지
  encryptor.py — 음성/문서 암호화 (AES-256)
  replicator.py — 분산 백업
  api_bridge.py — OpsClaw Manager 연동
  ...
```

"2006년의 **6개 파일**이 2026년에 **23개 파일**로 진화했어." 정규가 말했다.

"20년간 조금씩 기능을 추가한 거야." 추박하가 말했다. "이메일 모니터링, 네트워크 수집, 파일 변경 감지, 암호화, 분산 백업. 상황이 바뀔 때마다 필요한 기능을 추가했어."

"**collector_email.py — 이메일 모니터링?**" 가람이 물었다. "추팀장님이 다른 사람의 이메일을 **읽은** 건가요?"

"읽은 게 아니야." 추박하가 말했다. "넥서스AI의 메일 서버는 관리자가 **메타데이터**(발신/수신/시간/제목)를 볼 수 있어. 내용은 안 봐. 하지만 '서문기가 누구에게 언제 메일을 보냈는지'는 기록했어."

"메타데이터만으로도 많은 걸 알 수 있죠." 정규가 말했다.
