"언젠가는 K8s 환경에서 일하게 될 거야." 김영훈이 말했다. "공격을 모르면 방어도 못 해."

### 공격 경로 1: API 서버 노출

"K8s의 심장은 **API 서버**야." 정규가 말했다. "모든 명령이 API 서버를 거쳐. `kubectl`도 결국 API 서버에 REST 요청을 보내는 거야."

```
[공격 시나리오 1: K8s API 서버 노출]

상황: API 서버가 인터넷에 노출 (포트 6443)

공격 단계:
  1. 포트 스캐닝으로 6443 발견
     → nmap -sV -p 6443 target.com

  2. 인증 없이 접근 시도
     → curl -k https://target.com:6443/api/v1/namespaces
     → 응답이 오면: 인증이 비활성화된 상태 (치명적)

  3. 인증이 있더라도: 토큰 브루트포스, CVE 악용
     → CVE-2018-1002105: 권한 상승 취약점
     → 일반 사용자 → 관리자 권한 획득

  4. 클러스터 전체 장악
     → kubectl get secrets --all-namespaces
     → kubectl exec -it <pod> -- /bin/sh

방어:
  - API 서버를 인터넷에 노출하지 않음 (private subnet)
  - 인증 필수 (x509, OIDC, webhook)
  - 감사 로그 활성화
  - NetworkPolicy로 API 서버 접근 IP 제한
```

"API 서버가 인터넷에 노출된 K8s 클러스터가 **얼마나 많은 줄** 알아?" 정규가 물었다.

"설마 많겠어요?" 민재가 말했다.

"Shodan에서 검색하면 **수만 개**가 나와." 정규가 말했다. "2023년 Aqua Security 보고서에 따르면, 인터넷에 노출된 K8s API 서버의 **약 50%가 인증 없이 접근 가능**했어."

"**절반이요?**" 가람이 놀랐다.

"클라우드 설정 오류의 연장선이야." 김영훈이 말했다. "어제 S3 버킷도 그랬잖아. **기본값을 그대로 두면** 위험해."

### 공격 경로 2: etcd 미암호화

"etcd는 K8s의 **데이터베이스**야." 정규가 말했다. "모든 클러스터 상태 — Pod 정의, 설정, **시크릿** — 이 etcd에 저장돼."

```
[공격 시나리오 2: etcd 접근]

상황: etcd가 암호화되지 않은 상태

공격 단계:
  1. etcd 포트(2379) 접근
     → etcdctl --endpoints=http://target:2379 get / --prefix --keys-only

  2. K8s 시크릿 추출
     → etcdctl get /registry/secrets/default/my-secret
     → base64 디코딩하면 평문 비밀번호, API 키, 인증서 등

  3. ServiceAccount 토큰 추출
     → etcdctl get /registry/secrets/kube-system/default-token-xxxxx
     → 이 토큰으로 API 서버에 인증 → 클러스터 관리자 권한

위험:
  genesis를 K8s에 올리면, genesis의 DB 접근 정보가 K8s Secret에 저장됨
  etcd가 뚫리면 → genesis 접근 가능

방어:
  - etcd encryption at rest (EncryptionConfiguration)
  - etcd 접근을 API 서버에서만 허용 (mTLS)
  - etcd 포트(2379, 2380)를 네트워크에서 차단
  - External Secrets Operator로 시크릿을 외부(Vault)에서 주입
```

"K8s Secret은 기본적으로 **base64 인코딩**이야. 암호화가 아니라 **인코딩.**" 정규가 강조했다.

"base64는 `echo 'aGVsbG8=' | base64 -d`면 끝인데요." 가람이 말했다.

"맞아. **누구나 복호화**할 수 있어. 그래서 etcd를 반드시 암호화해야 하고, 가능하면 **HashiCorp Vault** 같은 외부 시크릿 관리를 쓰는 게 좋아."

### 공격 경로 3: 특권 컨테이너

"세 번째는 **특권 컨테이너(Privileged Container)**." 정규가 말했다.

```
[공격 시나리오 3: 컨테이너 탈출]

상황: 특권 모드(privileged: true)로 실행된 컨테이너

공격 단계:
  1. 컨테이너 내부에서 코드 실행 (웹 취약점 등)

  2. 특권 모드 확인
     → cat /proc/1/status | grep CapEff
     → CapEff: 0000003fffffffff  (모든 권한)

  3. 호스트 파일시스템 마운트
     → mkdir /tmp/host && mount /dev/sda1 /tmp/host
     → ls /tmp/host/  # 호스트의 전체 파일시스템

  4. 호스트에서 명령 실행
     → chroot /tmp/host /bin/bash
     → 컨테이너 → 호스트 탈출 완료

위험:
  컨테이너의 격리가 무의미해짐
  하나의 Pod를 통해 전체 노드 장악

방어:
  - privileged: false (절대적으로)
  - securityContext 설정:
      readOnlyRootFilesystem: true
      runAsNonRoot: true
      allowPrivilegeEscalation: false
      capabilities:
        drop: ["ALL"]
  - Pod Security Standards (Restricted 프로파일)
  - OPA/Gatekeeper로 정책 강제
```

"genesis Docker 컨테이너는요?" 가람이 물었다.

"genesis는 privileged가 아니야." 추박하가 만화책 뒤에서 말했다. "처음부터 **최소 권한**으로 만들었어."

"**20년 전부터요?**" 민재가 놀랐다.

"20년 전에는 Docker가 없었으니까, 처음에는 chroot jail이었어. 원리는 같아 — **격리와 최소 권한.**"

### 공격 경로 4: 서비스 어카운트 토큰

"마지막. K8s에서 **모든 Pod에는 서비스 어카운트**가 자동으로 연결돼." 정규가 말했다.

```
[공격 시나리오 4: ServiceAccount 토큰 탈취]

상황: Pod 내부에서 코드 실행 가능

공격 단계:
  1. Pod에 자동 마운트된 토큰 확인
     → cat /var/run/secrets/kubernetes.io/serviceaccount/token
     → eyJhbGciOiJSUzI1NiIsImt5cCI6...  (JWT 토큰)

  2. 이 토큰으로 API 서버에 요청
     → TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
     → curl -k -H "Authorization: Bearer $TOKEN" \
         https://kubernetes.default/api/v1/namespaces/default/secrets

  3. RBAC이 느슨하면: 다른 네임스페이스의 시크릿까지 접근 가능

  4. 횡적 이동(Lateral Movement)
     → 다른 Pod의 시크릿 탈취 → 다른 서비스에 접근

방어:
  - automountServiceAccountToken: false (불필요한 경우)
  - 서비스 어카운트별 최소 권한 RBAC
  - Bound Service Account Token (시간 제한)
  - Pod-to-Pod 통신에 mTLS (Istio, Linkerd)
```

"이 네 가지 공격을 조합하면—" 정규가 화이트보드에 화살표를 그렸다.

```
[K8s Kill Chain]

인터넷 → API 서버 노출(1) → 인증 우회 → etcd 접근(2)
  또는
웹 취약점 → Pod 내부 진입 → 특권 컨테이너 탈출(3) → 호스트 장악
  또는
웹 취약점 → Pod 내부 진입 → SA 토큰 탈취(4) → 횡적 이동 → 클러스터 장악
```

"K8s가 도입되면 **이 모든 공격을 막아야 해.**" 정규가 말했다. "보안팀의 일이 **두 배**로 늘어나는 거야."

"그래서 genesis를 Docker에 두자는 거야." 추박하가 만화책을 덮었다. "**단순한 시스템이 안전한 시스템**이야."

### 서문기의 사표

교육이 끝난 후, 점심시간.

진수현이 관제실로 들어왔다. 평소와 다른 표정이었다.

"진 부장님?" 김영훈이 물었다.

진수현이 잠시 망설였다. "... 서문기 상무가 **사표를 준비하고 있다**는 소식이 있어."

관제실이 멈췄다.

"**사표?**" 도윤이 놀랐다.

"확실한 건 아니야. 하지만 오전에 인사팀에서 서문기 상무의 **퇴직금 시뮬레이션 요청**이 있었다고."

"이사회 보고 **전에** 사표를 내겠다는 건가요?" 김영훈이 물었다.

"모르겠어. 하지만 **타이밍이 이상해.**" 진수현이 말했다.

추박하가 의자에 기대앉았다. 표정이 읽히지 않았다.

"추팀장님." 김영훈이 말했다. "서문기가 도망가려는 건 아닌가요?"

"도망?" 추박하가 중얼거렸다. "서문기가 도망가면 뭐가 달라지지?"

"이사회 보고에서 **책임자가 부재**하면, 보고의 임팩트가 줄어들 수 있어요."

"genesis의 evidence는 서문기가 있든 없든 **존재해.**" 추박하가 말했다. "하지만..."

추박하가 잠시 생각했다. "혜윤이에게 알려야 해. 이사회 보고 일정을 **앞당길 수 있는지** 확인해봐."

"앞당긴다고요?" 민재가 물었다.

"서문기가 **사표를 내기 전에** 이사회에 보고해야 해." 추박하가 말했다. "사표를 내고 나가버리면, 이사회가 **서문기에게 해명을 요구할 수 없어.**"

"체스네요." 가람이 말했다.

"항상 체스야." 정규가 말했다.

---

관제실의 분위기가 무거워졌다.

민재가 커피를 가져왔다. 여덟 잔. 말없이 한 잔씩 나눠주었다.

"고마워, 민재." 도윤이 말했다.

"서문기가 진짜 나가면..." 진호가 말을 꺼냈다. "우리한테는 좋은 거 아니에요? 귀찮은 사람이 없어지는 거잖아요."

"**아니야.**" 김영훈이 단호하게 말했다. "서문기가 나가면 **책임을 회피하는 것**이야. Nexus-1 API 인증 미조치, 보고서 수정 시도, 데이터 provenance 문제 — 이 모든 것의 **당사자가 사라지는** 거야."

"그리고 서문기가 나가면서 **어떤 것을 가져갈지** 모르잖아." 정규가 말했다. "USB? 클라우드 백업? **자기 보신용 증거?**"

"추팀장님." 가람이 조심스럽게 물었다. "서문기 상무가... genesis를 알고 있나요?"

추박하가 잠시 침묵했다.

"이름은 모를 거야. 하지만 **내가 뭔가를 기록하고 있다**는 걸 눈치챘을 수 있어." 추박하가 말했다. "그래서 사표를 내는 건지도 몰라. **배가 가라앉기 전에** 뛰어내리는 거지."

"쥐가 먼저 도망가는 거군요." 도윤이 말했다.

가람은 수첩에 적었다.

> `[Kubernetes 공격 — 4가지 주요 경로]`
>
> `1. API 서버 노출`
> `   인터넷에 6443 포트 노출 → 인증 우회 → 클러스터 장악`
> `   방어: private subnet, 인증 필수, 감사 로그`
>
> `2. etcd 미암호화`
> `   etcd 접근 → 시크릿(base64, 평문과 동일) 탈취`
> `   방어: encryption at rest, mTLS, 포트 차단`
>
> `3. 특권 컨테이너`
> `   privileged: true → 호스트 파일시스템 마운트 → 탈출`
> `   방어: privileged: false, Pod Security Standards`
>
> `4. ServiceAccount 토큰`
> `   자동 마운트된 JWT 토큰 → API 서버 인증 → 횡적 이동`
> `   방어: automountServiceAccountToken: false, RBAC 최소 권한`
>
> `K8s Kill Chain:`
> `  API 노출/웹 취약점 → 내부 진입 → 권한 상승 → 클러스터 장악`
>
> `[서문기 동향]`
> `  사표 준비 중 (미확인)`
> `  퇴직금 시뮬레이션 요청`
> `  추팀장: "이사회 보고를 앞당겨야 한다"`
> `  → 서문기가 나가기 전에 이사회에 사실을 보고해야 함`
>
> `교훈:`
> `  "K8s가 도입되면 이런 공격을 막아야 한다" — 미래 대비`
