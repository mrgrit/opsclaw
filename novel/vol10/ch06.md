# 10권 제6장: 코드로 다시 짓다

## 강의 연계: C6W14(심화) — IaC 보안

---

넥서스AI 인프라 재설계 프로젝트가 시작된 것은 사건 발생 3주 후였다.

이주원이 SOC 사무실에 와서 말했다.

"인프라를 처음부터 다시 짓는다."

김영훈이 물었다.

"전부요?"

"전부. 이번에는 제대로."

이주원이 화이트보드에 큰 글씨로 적었다.

**Infrastructure as Code — 모든 것을 코드로, 모든 코드를 투명하게.**

---

가람이 물었다.

"IaC가 뭔지는 아는데... 왜 지금 갑자기요?"

이주원이 답했다.

"이 사건에서 가장 큰 문제가 뭐였지?"

"데이터 은폐요."

"은폐가 가능했던 이유는?"

가람이 생각했다.

"... 인프라 변경 기록이 투명하지 않았으니까요. 누가, 언제, 뭘 바꿨는지 추적이 안 됐으니까."

"맞아. 그래서 이번에는 모든 인프라를 코드로 관리한다. 코드는 Git에 들어가고, Git은 모든 변경 이력을 기록해. 누가, 언제, 뭘, 왜 바꿨는지가 커밋 로그에 남아."

```
[IaC의 핵심 원칙]

1. 선언적 정의 (Declarative)
   - "서버를 만들어라" (X)
   - "이런 상태의 서버가 존재해야 한다" (O)

2. 버전 관리 (Version Control)
   - 모든 인프라 정의가 Git에 저장
   - 변경 이력 = 감사 추적(Audit Trail)

3. 재현 가능성 (Reproducibility)
   - 같은 코드 → 같은 인프라
   - "내 PC에서는 됐는데" 문제 제거

4. 자동화 (Automation)
   - 수동 콘솔 작업 금지
   - CI/CD 파이프라인을 통한 배포만 허용
```

이주원이 새 섹션을 적었다.

---

```
[IaC 보안 체크리스트]

1. 시크릿 관리
   - 절대 코드에 비밀번호/API 키를 하드코딩하지 않는다
   - AWS Secrets Manager 또는 HashiCorp Vault 사용
   - terraform.tfvars에 민감 정보 → .gitignore 필수

2. 코드 리뷰
   - 모든 인프라 변경은 PR(Pull Request)을 통해서만
   - 최소 2인 승인 필수
   - CISO 또는 보안팀 승인 필수 (보안 관련 변경)

3. 정적 분석
   - tfsec: Terraform 보안 스캐닝
   - checkov: IaC 정책 위반 탐지
   - CI 파이프라인에 자동 스캐닝 포함

4. 상태 파일 보호
   - terraform.tfstate는 S3 + DynamoDB로 원격 관리
   - 상태 파일 암호화 필수
   - 상태 파일 접근 로그 기록

5. 최소 권한 원칙
   - Terraform 실행 IAM 역할: 필요한 권한만
   - 프로덕션 apply: CI/CD 파이프라인만 (수동 금지)
```

가람이 적으며 말했다.

"2번이 핵심이네요. 코드 리뷰 없이는 인프라를 못 바꾸게 하는 거."

"맞아. 서문기가 한 것—혼자 몰래 설정을 바꾸고, 로그를 삭제하고—이런 게 불가능해지는 거야. 모든 변경이 PR로 가고, PR은 기록으로 남으니까."

가람이 Terraform 코드 예시를 작성했다.

```hcl
# terraform/modules/security-groups/main.tf

resource "aws_security_group" "soc_monitoring" {
  name_prefix = "soc-monitoring-"
  vpc_id      = var.vpc_id

  description = "SOC 모니터링 시스템 접근 제어"

  # Wazuh Agent 통신
  ingress {
    from_port   = 1514
    to_port     = 1515
    protocol    = "tcp"
    cidr_blocks = [var.wazuh_server_cidr]
    description = "Wazuh agent enrollment and communication"
  }

  # OpsClaw SubAgent
  ingress {
    from_port   = 8002
    to_port     = 8002
    protocol    = "tcp"
    cidr_blocks = [var.opsclaw_manager_cidr]
    description = "OpsClaw SubAgent runtime"
  }

  # 아웃바운드: 필요한 것만
  egress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTPS outbound for updates and API calls"
  }

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
    Team        = "SOC"
  }
}
```

"이 코드가 Git에 커밋되면, 누가 이 보안그룹을 만들었는지, 언제, 왜 만들었는지가 영원히 기록돼."

정규가 감탄했다.

"포트 하나 여는 것도 기록이 남는 거네요."

"그래야 해. 포트 하나가 침투 경로가 될 수 있으니까."

CI/CD 파이프라인도 설계했다.

```yaml
# .github/workflows/terraform-plan.yml

name: Terraform Plan

on:
  pull_request:
    paths:
      - 'terraform/**'

jobs:
  plan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: tfsec scan
        uses: aquasecurity/tfsec-action@v1.0.0

      - name: checkov scan
        uses: bridgecrewio/checkov-action@master

      - name: Terraform Init
        run: terraform init
        working-directory: terraform/environments/prod

      - name: Terraform Plan
        run: terraform plan -out=plan.tfplan
        working-directory: terraform/environments/prod

      - name: Post Plan to PR
        uses: robburger/terraform-pr-commenter@v1
        with:
          plan: plan.tfplan
```

"PR을 올리면 자동으로 보안 스캔이 돌고, Plan 결과가 PR에 코멘트로 달려. 리뷰어가 '이 변경이 어떤 영향을 미치는지' 코드로 확인할 수 있어."

"이번에는 제대로 짓자. 비밀이 없는 인프라를."

"비밀이 없는 인프라... 좋은 말이네요."

가람이 노트북에 적었다.

```
[가람의 노트]
IaC 보안 — 오늘 배운 것

핵심: "수동 변경 = 은폐의 가능성"

모든 인프라를 코드로 정의하면:
1. 변경 이력이 Git에 남는다 (누가, 언제, 왜)
2. 코드 리뷰가 필수가 된다 (혼자 못 바꾼다)
3. 정적 분석이 자동으로 돈다 (실수를 잡아준다)
4. 재현 가능하다 (같은 코드 → 같은 결과)

서문기가 IaC 환경에서 같은 일을 하려면?
- PR을 올려야 하고
- 2인 승인을 받아야 하고
- 보안 스캔을 통과해야 하고
- 모든 변경이 기록으로 남는다

... 불가능하진 않겠지만, 훨씬 어려워진다.
그리고 그 어려움의 차이가, 조직을 지키는 거다.
```

SOC 사무실의 모니터에는 새 Git 리포지토리가 열려 있었다. `nexus-infra`. 첫 커밋은 김영훈이 작성했다.
