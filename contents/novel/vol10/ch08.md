# 10권 제8장: 규칙을 만드는 사람

## 강의 연계: C2W10(심화) — Wazuh 룰

---

넥서스AI SOC 사무실. 인프라 재설계 프로젝트 2주차.

정규가 모니터 앞에서 머리를 쥐어뜯고 있었다.

"안 돼. 이 룰, 오탐이 너무 많아."

화면에는 Wazuh 알림 목록이 빼곡하게 떠 있었다. 지난 24시간 동안 3,742건. 대부분 오탐이었다.

김영훈이 다가왔다.

"뭐가 문제야?"

"추박하 선배가 작성한 기존 룰 세트가요... 일부가 너무 넓어요. 정상적인 DB 쿼리까지 다 잡고 있어요."

김영훈이 화면을 보았다.

```xml
<!-- 기존 룰: Rule ID 100210 (추박하 작성) -->
<rule id="100210" level="12">
  <if_sid>100200</if_sid>
  <match>DELETE FROM</match>
  <description>Database DELETE operation detected</description>
</rule>
```

"이건... 모든 DELETE 쿼리를 Level 12로 올리는 거네."

"네. 당시에는 서문기의 대량 삭제를 잡기 위해 넓게 잡은 것 같은데, 지금은 정상 운영 DELETE까지 전부 알림이 와요."

김영훈이 의자를 끌어와 앉았다.

"좋아. 이번 기회에 룰을 제대로 재설계하자. 기본부터."

---

김영훈이 화이트보드에 적기 시작했다.

"Wazuh 탐지 룰 설계 원칙. 이번에는 기본부터 가자."

```
[Wazuh 탐지 룰 설계 원칙]

1. 정밀도 우선 (Precision over Recall)
   - 오탐(False Positive)이 많으면 진짜 경보를 놓친다
   - "Alert Fatigue" — 알림 피로가 가장 큰 적

2. 계층적 구조 (Layered Rules)
   - 부모 룰 → 자식 룰로 점진적 세분화
   - <if_sid>로 상위 룰의 조건을 상속

3. 컨텍스트 추가 (Context Enrichment)
   - 단순 패턴 매칭이 아닌, 누가/언제/어디서를 함께 판단
   - 업무 시간 외 접근, 비인가 IP, 대량 작업 등

4. 레벨 기준 명확화
   - Level 0-4: 정보성 (로깅만)
   - Level 5-7: 주의 (대시보드 표시)
   - Level 8-10: 경고 (Slack 알림)
   - Level 11-13: 긴급 (즉시 대응)
   - Level 14-15: 치명 (자동 차단 + 전체 호출)

5. 테스트 필수
   - 프로덕션 적용 전 반드시 테스트 환경에서 검증
   - 오탐률/미탐률 측정 후 조정
```

정규가 열심히 적었다.

---

"자, 그러면 Rule 100210을 고쳐보자."

김영훈이 말했다.

"기존 룰의 문제가 뭐였지?"

정규가 답했다.

"모든 DELETE를 Level 12로 올린 것. 정상적인 배치 작업도, 유저 요청에 의한 삭제도, 전부 잡아요."

"그러면 어떻게 바꿔야 해?"

정규가 생각했다.

"... 컨텍스트를 추가해야 해요. '누가' 삭제했는지, '얼마나 많이' 삭제했는지를 조건에 넣어야 해요."

"좋아. 작성해봐."

```xml
<!-- 개선된 룰 세트: DB 대량 삭제 탐지 -->

<!-- 부모 룰: 모든 DB 쿼리 로그 (정보성, Level 3) -->
<rule id="100200" level="3">
  <decoded_as>postgresql</decoded_as>
  <description>Database query logged</description>
</rule>

<!-- 자식 룰 1: DELETE 쿼리 감지 (주의, Level 5) -->
<rule id="100210" level="5">
  <if_sid>100200</if_sid>
  <match>DELETE FROM</match>
  <description>Database DELETE query detected</description>
</rule>

<!-- 자식 룰 2: 대량 삭제 (1000건 이상, 경고 Level 10) -->
<rule id="100211" level="10">
  <if_sid>100210</if_sid>
  <regex>rows_affected:\s*(\d{4,})</regex>
  <description>Bulk DELETE: more than 1000 rows affected</description>
</rule>

<!-- 자식 룰 3: 업무 시간 외 대량 삭제 (긴급, Level 13) -->
<rule id="100212" level="13">
  <if_sid>100211</if_sid>
  <time>10pm - 6am</time>
  <description>Bulk DELETE during off-hours — immediate review required</description>
</rule>

<!-- 자식 룰 4: 관리자 계정의 민감 테이블 삭제 (치명, Level 15) -->
<rule id="100213" level="15">
  <if_sid>100210</if_sid>
  <match>customer_data|user_credentials|audit_log</match>
  <regex>srcuser:\s*svc_admin</regex>
  <description>Admin DELETE on sensitive table — CRITICAL</description>
</rule>
```

김영훈이 읽었다.

"좋아. 설명해봐."

정규가 설명했다.

"기존에는 모든 DELETE가 Level 12였어요. 이제는 계층적으로 나눴습니다. 일반 DELETE는 Level 5—기록만 하고 대시보드에 표시. 1000건 이상 대량 삭제는 Level 10—Slack 알림. 업무 시간 외 대량 삭제는 Level 13—즉시 대응. 그리고 관리자 계정이 민감 테이블을 건드리면 Level 15—자동 차단까지."

"오탐은?"

"Level 5에서는 오탐이 있어도 괜찮아요. 기록만 하니까. Level 10 이상은 조건이 구체적이라 오탐이 대폭 줄어요."

김영훈이 고개를 끄덕였다.

"이 서문기 사건에 이 룰이 적용되어 있었다면?"

정규가 타임라인을 대입해 보았다.

```
[시뮬레이션: 개선된 룰로 서문기 사건 탐지]

2024-06-14 02:17 — Rule 100213 (Level 15) 트리거
  조건: svc_admin_smg가 customer_data에서 47,293건 DELETE
  → 관리자 + 민감 테이블 → Level 15
  → 동시에 Rule 100212 (Level 13)도 트리거
  → 업무 시간 외(새벽 2시) + 대량 삭제

대응:
  - 즉시 Slack #soc-critical 알림
  - PagerDuty 에스컬레이션 → 팀장 호출
  - 해당 DB 계정 자동 잠금 (Level 15 정책)
  - SOC팀 15분 내 대응 시작
```

"기존 룰(Level 12)에서는 오탐에 묻혀서 놓쳤을 수 있어요. 개선된 룰에서는 Level 15가 뜨면서 자동 차단까지 가니까, 서문기가 삭제를 완료하기 전에 계정이 잠겼을 거예요."

---

가람이 옆에서 듣고 있다가 물었다.

"정규야, 추박하 선배가 작성한 원래 룰—그냥 삭제하는 거야?"

정규가 잠시 멈추었다.

"... 아뇨. 개선하는 거예요."

정규가 화면을 보며 말했다.

"추박하 선배의 룰이 없었으면 법정 증거도 없었어요. 그 룰이 서문기의 행동을 기록했으니까. 넓게 잡은 게 그때는 맞았어요. 지금은 상황이 다르니까 다듬는 거지, 버리는 게 아니에요."

김영훈이 미소 지었다.

"정규, 많이 컸다."

정규가 머리를 긁적였다.

---

저녁, 정규가 테스트 환경에서 새 룰을 검증했다.

```bash
# Wazuh 룰 테스트
/var/ossec/bin/wazuh-logtest

# 테스트 로그 입력
Type one log per line:
2024-06-14 02:17:33 postgresql: DELETE FROM customer_data WHERE leak_flag = true; srcuser: svc_admin_smg; rows_affected: 47293

# 결과
**Phase 3: Completed filtering (rules).
        id: '100213'
        level: '15'
        description: 'Admin DELETE on sensitive table — CRITICAL'
        groups: '['database','admin','sensitive']'
```

"Level 15. 정확해요."

정규가 다음 테스트를 했다.

```bash
# 정상 운영 DELETE 테스트
Type one log per line:
2024-06-14 14:30:00 postgresql: DELETE FROM session_cache WHERE expired = true; srcuser: app_cleaner; rows_affected: 150

# 결과
**Phase 3: Completed filtering (rules).
        id: '100210'
        level: '5'
        description: 'Database DELETE query detected'
```

"Level 5. 정상 운영은 낮은 레벨로 떨어져요. 오탐 없어요."

김영훈이 어깨를 두드렸다.

"프로덕션에 적용하자."

---

가람이 노트북에 적었다.

```
[가람의 노트]
탐지 룰 설계 — 정규에게 배운 것

1. 넓은 룰은 초기에는 유용하지만, 성숙하면 세분화해야 한다
2. 계층 구조: 부모 룰 → 자식 룰로 조건을 좁혀간다
3. 컨텍스트가 핵심: 같은 DELETE도 '누가, 언제, 얼마나'에 따라 다르다
4. 테스트 없이 프로덕션 적용은 금지
5. 기존 룰을 존중하되, 개선은 필수

"이번에는 기본부터" — 김영훈 팀장의 말.
기본이란 화려한 것이 아니라, 정확한 것이다.
```

정규가 새 룰 세트를 커밋했다.

```
commit: "Refactor Wazuh detection rules — layered structure, reduced false positives"
author: 정규
reviewed by: 김영훈, 강혜윤
```

추박하가 작성한 첫 번째 룰의 주석에는 이렇게 적혀 있었다.

```xml
<!-- Original rule by 추박하, 2023-07 -->
<!-- Refactored by 정규, 2026-04 -->
<!-- 기본부터 다시. -->
```
