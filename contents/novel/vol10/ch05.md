# 10권 제5장: 모니터링

## 강의 연계: C6W13(심화) — 클라우드 모니터링

---

사건이 터지고 열흘이 지났다. 검찰 수사는 계속되고, 언론은 매일 새로운 기사를 쏟아냈고, 주가는 반토막이 났다.

하지만 서버는 꺼지지 않았다.

새벽 3시 17분, SOC 사무실.

야간 근무 중인 가람의 모니터에 알림이 떴다.

```
[CloudWatch Alert]
Severity: Warning
Service: nexus-api-prod (ap-northeast-2)
Metric: CPUUtilization
Value: 87.3% (threshold: 80%)
Duration: 5 minutes
```

가람이 잠에서 깨듯 화면을 확인했다. 프로덕션 API 서버의 CPU 사용률이 임계치를 넘었다.

"누가 이 시간에..."

로그를 열어보았다.

```bash
# CloudWatch Logs Insights 쿼리
fields @timestamp, @message
| filter @message like /nexus-api/
| sort @timestamp desc
| limit 20
```

원인은 금방 찾았다. 언론 보도 이후 고객들이 자기 데이터 확인을 위해 "내 정보 조회" API를 폭발적으로 호출하고 있었다. 낮에는 오토스케일링으로 감당했지만, 새벽 스케일다운 이후에도 트래픽이 줄지 않은 것이다.

---

가람이 김영훈에게 메시지를 보냈다.

```
[가람 → 김영훈, 03:22]
API 서버 CPU 87%. 고객 정보 조회 트래픽 급증.
오토스케일링 최소 인스턴스 수 조정하겠습니다.
```

```
[김영훈 → 가람, 03:24]
확인. 진행해.
```

가람이 터미널을 열었다.

```bash
# Auto Scaling Group 최소 인스턴스 조정
aws autoscaling update-auto-scaling-group \
  --auto-scaling-group-name nexus-api-prod-asg \
  --min-size 4 \
  --max-size 12 \
  --desired-capacity 6
```

```
[결과]
Auto Scaling Group updated successfully.
Desired capacity changed from 3 to 6.
New instances launching...
```

5분 후 CPU 사용률이 52%로 떨어졌다.

가람이 한숨을 내쉬었다.

---

아침이 밝았다. 출근한 정규가 물었다.

"어젯밤에 무슨 일 있었어?"

"API 서버 스케일링 이슈. 처리했어."

가람이 덤덤하게 말했다.

조은별이 커피를 내려주며 말했다.

"회사가 난리여도 서버는 멈추면 안 되니까."

김영훈이 고개를 끄덕였다.

"맞아. 오늘 모니터링 체계 한 번 점검하자. 지금 같은 비상 상황에서는 평소보다 더 촘촘하게 봐야 해."

---

김영훈이 화이트보드에 모니터링 아키텍처를 그렸다.

"우리 모니터링 스택을 정리하자."

```
[넥서스AI 클라우드 모니터링 아키텍처]

┌─────────────────────────────────────────┐
│              Data Sources               │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌───────┐ │
│  │EC2   │ │RDS   │ │ELB   │ │Lambda │ │
│  └──┬───┘ └──┬───┘ └──┬───┘ └───┬───┘ │
└─────┼────────┼────────┼─────────┼──────┘
      │        │        │         │
      ▼        ▼        ▼         ▼
┌─────────────────────────────────────────┐
│           CloudWatch Metrics            │
│  CPU, Memory, Network, Disk, Latency   │
│  Custom Metrics (App-level)             │
└────────────────┬────────────────────────┘
                 │
    ┌────────────┼────────────┐
    ▼            ▼            ▼
┌────────┐ ┌─────────┐ ┌──────────┐
│Alarms  │ │Dashboard│ │Logs      │
│→ SNS   │ │Grafana  │ │Insights  │
│→ Slack │ │         │ │          │
└────────┘ └─────────┘ └──────────┘
```

"여기서 지금 문제가 될 수 있는 지점이 어디야?"

정규가 손을 들었다.

"오토스케일링 정책이 평시 기준이라, 지금처럼 비정상 트래픽이 오면 반응이 느려요."

"맞아. 그래서 임시로 스케일링 정책을 조정한다."

```
[비상시 모니터링 강화 체크리스트]

1. Auto Scaling 정책 조정
   - 최소 인스턴스: 3 → 6
   - Scale-out 임계치: CPU 70% → CPU 60%
   - Cooldown period: 300s → 120s

2. 알림 강화
   - Warning 임계치 하향 조정
   - 알림 채널 추가: Slack #soc-emergency
   - PagerDuty 에스컬레이션: 5분 무응답 시 팀장 직접 호출

3. 추가 메트릭 수집
   - API 엔드포인트별 요청 수 (특히 /my-data, /account)
   - 에러율 (4xx, 5xx) 실시간 추적
   - DB 커넥션 풀 사용률

4. 로그 보존 강화
   - CloudWatch Logs 보존 기간: 30일 → 365일
   - S3 아카이브: 일별 자동 백업
```

---

가람이 Grafana 대시보드를 열었다.

"대시보드에 패널 하나 추가할게요. 고객 정보 조회 API 요청 수를 실시간으로 보려고요."

```json
{
  "panel": {
    "title": "Customer Data Request Rate",
    "type": "timeseries",
    "datasource": "CloudWatch",
    "targets": [
      {
        "namespace": "NexusAI/API",
        "metricName": "RequestCount",
        "dimensions": {
          "Endpoint": "/api/v1/my-data"
        },
        "period": 60,
        "stat": "Sum"
      }
    ],
    "thresholds": [
      {"value": 1000, "color": "yellow"},
      {"value": 5000, "color": "red"}
    ]
  }
}
```

조은별이 옆에서 보며 말했다.

"언론 보도 나간 날 패턴 보여줘."

가람이 그래프를 확대했다.

"보도 당일 오전 7시부터 요청이 300% 증가했어요. 그 뒤로 매일 밤에도 평소의 2배 이상이에요. 고객들이 자기 데이터가 유출됐는지 확인하려는 거예요."

김영훈이 말했다.

"그건 당연한 거야. 우리가 해야 할 건, 그 요청에 안정적으로 응답하는 거지."

---

오후에 정규가 RDS 모니터링에서 이상을 발견했다.

"팀장님, DB 커넥션 풀이 90%까지 차고 있어요."

```
[RDS Monitoring]
DBInstanceIdentifier: nexus-prod-primary
DatabaseConnections: 180/200 (90%)
ReadIOPS: 12,450
WriteIOPS: 3,200
FreeableMemory: 1.2 GB (low)
```

김영훈이 즉시 지시했다.

"Read Replica 하나 더 올려. 읽기 트래픽 분산해야 해."

```bash
# Read Replica 추가
aws rds create-db-instance-read-replica \
  --db-instance-identifier nexus-prod-read-2 \
  --source-db-instance-identifier nexus-prod-primary \
  --db-instance-class db.r6g.xlarge
```

정규가 말했다.

"팀장님, 이런 건 원래 인프라팀 일 아닌가요?"

김영훈이 조용히 답했다.

"인프라팀 인원이 반으로 줄었어. 구조조정 1차로. 지금 우리가 안 하면 아무도 안 해."

SOC 사무실이 잠시 조용해졌다.

---

저녁, 가람이 노트북에 적었다.

```
[가람의 노트]
클라우드 모니터링은 '감시'가 아니라 '돌봄'이다.

서버가 아프면 알림이 온다.
트래픽이 몰리면 스케일링한다.
DB가 버거워하면 Replica를 올린다.

회사가 무너지고 있어도, 서비스가 살아있으면
고객은 자기 데이터를 확인할 수 있다.
그게 우리가 지킬 수 있는 마지막 신뢰다.

모니터링 핵심:
1. 메트릭 수집 → CloudWatch, Custom Metrics
2. 시각화 → Grafana 대시보드
3. 알림 → SNS, Slack, PagerDuty
4. 자동 대응 → Auto Scaling, Runbook
5. 로그 보존 → 장기 아카이브 (법적 증거 가능성)
```

새벽 2시, 교대 시간.

정규가 가람의 자리에 앉으며 물었다.

"형, 내일도 이래요?"

"내일도 이래."

"모레도?"

가람이 웃었다.

"모레도. 서버가 돌아가는 한."

가람은 코트를 입고 사무실을 나섰다. 복도의 형광등이 꺼진 사무실들을 비추고 있었다. 구조조정으로 비어가는 층들.

하지만 SOC 사무실의 불은 꺼지지 않았다.

---

*— 10권 5장 끝 —*
