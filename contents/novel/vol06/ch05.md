### nftables 실전 구성

정규가 secu 서버의 nftables 룰을 전면 재검토했다.

"현재 nftables 룰셋을 확인합니다."

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S nft list ruleset 2>/dev/null | wc -l && echo \"---\" && echo 1 | sudo -S nft list ruleset 2>/dev/null | grep -c drop && echo 1 | sudo -S nft list ruleset 2>/dev/null | grep -c accept","subagent_url":"http://10.20.30.1:8002"}'
```

```
187 (총 줄 수)
---
23 (drop 룰)
42 (accept 룰)
```

"**187줄, drop 23개, accept 42개.**" 정규가 말했다. "부족해. 어제 발견된 **SubAgent 무인증 접근**을 막으려면 더 세밀한 룰이 필요해."

정규가 새 룰셋을 설계했다.

```
[nftables 강화 방안]

1. 네트워크 세그먼테이션:
   - JuiceShop(:3000) 트래픽을 별도 체인으로 격리
   - JuiceShop → 내부 서비스(SubAgent, OpsClaw, Wazuh) 접근 차단
   - JuiceShop → 외부(인터넷) 접근 차단

2. SubAgent 보호:
   - 포트 8002에 대한 접근을 opsclaw(10.20.30.201)에서만 허용
   - 다른 모든 소스에서 8002 차단

3. 관리 포트 제한:
   - SSH(:22)는 opsclaw에서만 허용
   - Wazuh API(:55000)는 siem 로컬에서만 허용
   - OpsClaw API(:8000)는 관제실 네트워크에서만 허용

4. Rate Limiting:
   - 외부에서 들어오는 SYN 패킷: 초당 10개 제한
   - SSH 연결 시도: 분당 5회 제한
```

```bash
# nftables 룰 업데이트
curl -s -X POST "http://localhost:8000/projects/$PID/execute-plan" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "tasks": [
      {"order":1,"title":"secu: SubAgent 접근 제한","instruction_prompt":"echo 1 | sudo -S nft add rule inet filter input ip saddr != 10.20.30.201 tcp dport 8002 drop 2>/dev/null && echo \"SubAgent: only opsclaw allowed\"","risk_level":"high","subagent_url":"http://10.20.30.1:8002"},
      {"order":2,"title":"web: SubAgent 접근 제한","instruction_prompt":"echo 1 | sudo -S nft add rule inet filter input ip saddr != 10.20.30.201 tcp dport 8002 drop 2>/dev/null && echo \"SubAgent: only opsclaw allowed\"","risk_level":"high","subagent_url":"http://10.20.30.80:8002"},
      {"order":3,"title":"siem: SubAgent 접근 제한","instruction_prompt":"echo 1 | sudo -S nft add rule inet filter input ip saddr != 10.20.30.201 tcp dport 8002 drop 2>/dev/null && echo \"SubAgent: only opsclaw allowed\"","risk_level":"high","subagent_url":"http://10.20.30.100:8002"},
      {"order":4,"title":"secu: SSH rate limit","instruction_prompt":"echo 1 | sudo -S nft add rule inet filter input tcp dport 22 ct state new limit rate 5/minute accept 2>/dev/null && echo \"SSH rate limit: 5/min\"","risk_level":"high","subagent_url":"http://10.20.30.1:8002"}
    ],
    "subagent_url":"http://localhost:8002",
    "parallel": true
  }'
```

"**4대 서버의 SubAgent를 동시에 보호.** opsclaw(10.20.30.201)에서만 접근 가능하도록."

결과:

```
secu: SubAgent — only opsclaw allowed ✅
web:  SubAgent — only opsclaw allowed ✅
siem: SubAgent — only opsclaw allowed ✅
secu: SSH rate limit 5/min ✅
```

"이제 JuiceShop에서 SubAgent로 직접 접근하면 **drop**돼." 정규가 말했다.

### Suricata IPS 강화

동시에 도윤이 Suricata 룰을 업데이트했다.

"Red Team의 공격 패턴을 기반으로 **커스텀 룰**을 추가합니다."

```bash
# Suricata 커스텀 룰 추가
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S tee -a /etc/suricata/rules/local.rules << RULES\n# Red Team Exercise - Custom Rules\nalert http any any -> any any (msg:\"CUSTOM PHP Webshell Upload Attempt\"; flow:established,to_server; content:\".php\"; http_uri; content:\"<?php\"; http_client_body; sid:9000001; rev:1;)\nalert http any any -> any any (msg:\"CUSTOM SSRF Internal IP Access\"; flow:established,to_server; content:\"10.20.30.\"; http_uri; sid:9000002; rev:1;)\nalert tcp any any -> any 8002 (msg:\"CUSTOM Direct SubAgent Access Attempt\"; flow:established,to_server; sid:9000003; rev:1;)\nRULES\necho 1 | sudo -S systemctl reload suricata && echo \"Suricata rules updated\"","risk_level":"high","subagent_url":"http://10.20.30.1:8002"}'
```

"**3개 커스텀 룰** 추가:
1. PHP 웹셸 업로드 탐지
2. SSRF 내부 IP 접근 탐지
3. SubAgent 직접 접근 탐지"

"5권에서 만든 **LLM 룰 자동 생성**을 여기서도 활용할 수 있겠네요." 가람이 말했다.

"맞아. LLM에게 '어제 Red Team의 공격 패턴을 기반으로 Suricata 룰을 만들어줘'라고 하면 초안을 바로 받을 수 있어." 정규가 말했다. "내가 검토하고 배포하면 돼."

### 오후: 방어 테스트

방어를 강화한 후, 김영훈이 **셀프 테스트**를 수행했다.

"Red Team이 하기 전에 **우리가 먼저** 공격해보자."

```bash
# 셀프 테스트: JuiceShop → SubAgent 접근 시도
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"curl -s --connect-timeout 3 http://10.20.30.1:8002/a2a/health 2>/dev/null || echo \"CONNECTION BLOCKED\"","subagent_url":"http://10.20.30.80:8002"}'
```

```
CONNECTION BLOCKED
```

"**차단 확인.** web 서버에서 secu SubAgent에 접근 불가." 김영훈이 말했다.

"opsclaw에서는?"

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"curl -s http://10.20.30.1:8002/a2a/health","subagent_url":"http://localhost:8002"}'
```

```json
{"status":"ok","agent":"secu-subagent"}
```

"**opsclaw에서는 정상 접근.** nftables 룰이 정확히 동작해."

"좋아. **방패가 완성됐어.**" 김영훈이 말했다.

> `[방화벽+IPS 실전 구성 — 교육 정리]`

> `방어의 사이클:`
> `  공격 관찰 → 패턴 분석 → 룰 작성 → 배포 → 테스트 → 모니터링`
> `  → 이 사이클이 빠를수록 방어가 강해진다`
> `  → OpsClaw로 자동화하면 사이클이 **분 단위**로 줄어듦`
