### 인증 우회 공격

오전 11시. Red Team이 피싱으로 획득한 자격 증명을 사용하기 시작했다.

```
[11:05] SSH 로그인
  src: 198.51.100.11 → dst: 10.20.30.80 (web)
  user: jpark (박재영, 마케팅팀 — 피싱 피해자)
  auth: password
  result: SUCCESS ← ⚠️
```

"**web 서버에 SSH 로그인 성공.** 마케팅팀 박재영 계정으로." 정규가 말했다.

"잠깐. web 서버에 마케팅팀 직원의 SSH 계정이 **왜 있어?**" 김영훈의 눈이 가늘어졌다.

"... 서버 공유 문화요." 가람이 말했다. "3권에서 정태식 본부장이 '관리 편의'를 위해 **모든 직원에게 SSH 접근 권한**을 줬잖아요."

"그때 **SSH 키 인증으로 전환**했잖아." 민재가 말했다.

"전환했지만, **기존 비밀번호 인증도 아직 활성화**되어 있을 수 있어." 정규가 확인했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S grep PasswordAuthentication /etc/ssh/sshd_config","subagent_url":"http://10.20.30.80:8002"}'
```

```
PasswordAuthentication yes
```

"**PasswordAuthentication yes.** 비밀번호 인증이 아직 활성화되어 있어!" 정규가 말했다.

"4권에서 **SSH 키 인증 전환을 1주 내 단계적 적용**하기로 했었잖아요." 가람이 수첩을 넘겼다. "근데 web 서버는 전환이 안 된 거예요?"

"secu와 opsclaw는 전환했지만, **web과 siem**은 아직이야." 김영훈이 이를 악물었다. "정태식 본부장이 '웹 서비스 영향이 있을 수 있다'며 미뤘거든."

"**또 가용성 핑계.**" 도윤이 말했다.

---

Red Team은 web 서버에 로그인한 후, 내부 네트워크를 탐색하기 시작했다.

```
[11:15] web 서버에서 내부 스캔
  user: jpark (피싱 획득 계정)
  command: nmap -sn 10.20.30.0/24
  → secu(10.20.30.1), siem(10.20.30.100), opsclaw(10.20.30.201) 발견

[11:18] web → opsclaw 접근 시도
  command: curl http://10.20.30.201:8000/projects
  → 401 Unauthorized (API Key 필요)

[11:20] web → siem 접근 시도
  command: ssh jpark@10.20.30.100
  → SUCCESS ← ⚠️⚠️ (siem도 비밀번호 인증 활성화)
```

"**siem 서버에도 로그인 성공.** 같은 비밀번호로." 정규가 말했다.

"같은 비밀번호?" 김영훈이 물었다.

"피싱으로 획득한 비밀번호를 **다른 서버에도** 시도한 거야. **패스워드 리유즈** — 같은 비밀번호를 여러 서비스에 쓰는 것." 정규가 설명했다.

"**Credential Stuffing.** ATT&CK T1110.004." 진호가 매핑했다.

```
[인증 우회 공격 체인 — Red Team]

1. 피싱 메일 → 3명의 비밀번호 획득 (T1566.001)
2. 획득한 비밀번호로 web 서버 SSH 로그인 (T1078 - Valid Accounts)
3. web → 내부 네트워크 스캔 (T1046)
4. web → siem 서버 SSH 로그인 (T1021.004 - SSH)
5. siem 접근 = Wazuh 관리 콘솔 접근 가능
   → 탐지 룰 수정/삭제 가능 = SIEM 무력화 위험 (T1562.001)
```

"**5단계 만에 SIEM 서버에 도달.** Red Team이 Wazuh 룰을 수정하면, 자기 활동을 **탐지 불가능**하게 만들 수 있어." 김영훈이 말했다.

"3권의 APT 공격과 **같은 패턴**이에요." 가람이 말했다. "3권 5장 '눈먼 파수꾼'에서 공격자가 Wazuh agent를 정지시킨 것처럼."

"**이번에는 막아야 해.**" 김영훈이 말했다.

### Blue Team 대응

김영훈이 즉시 대응을 지시했다.

"**관찰 모드 종료.** 방어 모드 전환."

```bash
# 1. web/siem 서버: 비밀번호 인증 즉시 비활성화
curl -s -X POST "http://localhost:8000/projects/$PID/execute-plan" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "tasks": [
      {"order":1,"title":"web: SSH 비밀번호 인증 비활성화","instruction_prompt":"echo 1 | sudo -S sed -i \"s/PasswordAuthentication yes/PasswordAuthentication no/\" /etc/ssh/sshd_config && echo 1 | sudo -S systemctl restart sshd && echo \"web: PasswordAuth disabled\"","risk_level":"high","subagent_url":"http://10.20.30.80:8002"},
      {"order":2,"title":"siem: SSH 비밀번호 인증 비활성화","instruction_prompt":"echo 1 | sudo -S sed -i \"s/PasswordAuthentication yes/PasswordAuthentication no/\" /etc/ssh/sshd_config && echo 1 | sudo -S systemctl restart sshd && echo \"siem: PasswordAuth disabled\"","risk_level":"high","subagent_url":"http://10.20.30.100:8002"},
      {"order":3,"title":"피싱 피해 계정 비밀번호 강제 리셋","instruction_prompt":"echo 1 | sudo -S passwd -e jpark 2>/dev/null && echo \"jpark password expired - must change on next login\"","risk_level":"medium","subagent_url":"http://10.20.30.80:8002"}
    ],
    "subagent_url":"http://localhost:8002",
    "parallel": true
  }'
```

"**3개 조치를 동시 실행.** 비밀번호 인증 비활성화 + 피해 계정 리셋." 김영훈이 말했다.

결과:

```
web:  PasswordAuthentication → no, sshd restart 완료
siem: PasswordAuthentication → no, sshd restart 완료
jpark: password expired — 다음 로그인 시 변경 필요
```

"Red Team의 jpark 계정 세션은?" 정규가 확인했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S who | grep jpark","subagent_url":"http://10.20.30.80:8002"}'
```

```
jpark    pts/2        2026-05-13 11:05 (198.51.100.11)
```

"아직 로그인 중이야. **기존 세션은 비밀번호 변경으로 끊기지 않아.**" 정규가 말했다.

"강제 종료할까요?" 도윤이 물었다.

"아니." 김영훈이 말했다. "Red Team의 세션을 **관찰**하자. 다음에 뭘 할지 보고 나서 판단해."

20분 후. Red Team이 siem 서버에서 활동을 시작했다.

```
[11:42] siem에서 Wazuh 설정 파일 접근 시도
  user: jpark
  command: sudo cat /var/ossec/etc/ossec.conf
  result: jpark is not in the sudoers file. This incident will be reported.
```

"**sudo 권한 없음.** jpark은 일반 사용자니까 Wazuh 설정에 접근 못 해." 정규가 안도했다.

"**최소 권한 원칙**이 동작했어." 김영훈이 말했다. "모든 직원에게 SSH 접근을 줬지만, sudo는 안 줬으니까."

"3권에서 **NOPASSWD sudo를 제거**한 게 여기서 빛나네요." 가람이 말했다.

```
[11:45] Red Team이 jpark 세션에서 로그아웃
  → 비밀번호 변경 감지? 또는 sudo 차단으로 포기?
```

"Red Team이 후퇴했어." 진호가 말했다.

"임시 후퇴야." 김영훈이 말했다. "다른 경로를 찾을 거야."

### 오후: 기본 자격 증명 공격

오후 2시. Red Team이 새로운 접근을 시도했다.

```
[14:10] HTTP POST https://10.20.30.100:55000/security/user/authenticate
  src: 198.51.100.12
  body: {"user":"wazuh","password":"wazuh"}
  response: 401 Unauthorized

[14:11] body: {"user":"admin","password":"admin"}
  response: 401 Unauthorized

[14:11] body: {"user":"wazuh-wui","password":"wazuh-wui"}
  response: 401 Unauthorized

[14:12] body: {"user":"wazuh","password":"SecretPassword"}
  response: 200 OK ← ⚠️⚠️⚠️
```

"**Wazuh API에 기본 자격 증명으로 로그인 성공!**" 정규가 경악했다.

"'wazuh/SecretPassword'? 이건 **Wazuh 설치 시 기본 비밀번호**잖아!" 도윤이 말했다.

"**바꾸지 않은 거야?**" 김영훈이 정규를 바라보았다.

정규의 얼굴이 붉어졌다. "... Wazuh API 비밀번호는 제가 관리하는 게 아니라 인프라팀이..."

"**누가 관리하든 바뀌어야 했어.**" 김영훈이 단호하게 말했다.

기본 자격 증명(Default Credentials). 1권에서 SSH 비밀번호 '1', 2권에서 공유 계정 admin/admin123. 그리고 지금, Wazuh API의 기본 비밀번호.

*같은 실수가 반복되고 있다.*

```bash
# 즉시 대응: Wazuh API 비밀번호 변경
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S /var/ossec/bin/wazuh-security change-password admin \"$(openssl rand -base64 32)\" 2>/dev/null && echo \"Wazuh API password changed\"","risk_level":"high","subagent_url":"http://10.20.30.100:8002"}'
```
