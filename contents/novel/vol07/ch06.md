### AppArmor 프로필 적용

정규가 genesis_core 컨테이너의 AppArmor 상태를 확인했다.

```bash
# 현재 AppArmor 상태 확인
docker inspect genesis_core --format '{{.HostConfig.SecurityOpt}}'
```

```
[seccomp=genesis_seccomp.json]
```

"seccomp은 적용되어 있는데, **AppArmor 프로필이 없어.** docker-bench-security에서 WARN이 나온 항목 중 하나야." 정규가 말했다.

"AppArmor가 뭐예요?" 가람이 물었다.

"**Linux 보안 모듈(LSM)**이야. 프로세스가 접근할 수 있는 파일, 네트워크, capability를 **프로필 기반**으로 제한해." 정규가 설명했다. "seccomp이 **시스콜 레벨**에서 차단한다면, AppArmor는 **파일시스템과 네트워크 레벨**에서 차단해. 보완 관계야."

```bash
# genesis 전용 AppArmor 프로필 작성
cat > /etc/apparmor.d/docker-genesis << 'PROFILE'
#include <tunables/global>

profile docker-genesis flags=(attach_disconnected,mediate_deleted) {
  #include <abstractions/base>

  # 파일 접근: /app (읽기 전용), /data (읽기/쓰기)
  /app/** r,
  /app/daemon.sh rix,
  /app/*.py rix,
  /data/** rw,
  /data/genesis.db rw,
  /data/logs/** rw,

  # 임시 파일
  /tmp/** rw,

  # Python 런타임
  /usr/bin/python3* rix,
  /usr/lib/python3/** r,

  # 네트워크: 로컬만
  network inet stream,
  network inet dgram,

  # 명시적 거부
  deny /proc/** w,
  deny /sys/** w,
  deny /etc/shadow r,
  deny /etc/passwd w,
  deny /root/** rw,

  # capability 제한
  capability net_bind_service,
  deny capability sys_admin,
  deny capability sys_ptrace,
}
PROFILE

# 프로필 로드
sudo apparmor_parser -r /etc/apparmor.d/docker-genesis
```

"이 프로필은 genesis가 **/app과 /data에만** 접근할 수 있게 해. /proc, /sys, /etc/shadow, /root는 **명시적으로 거부.**" 정규가 설명했다.

```bash
# genesis 컨테이너에 AppArmor 프로필 적용
# 컨테이너 재생성 필요 (security-opt은 런타임에 변경 불가)
docker stop genesis_core
docker rm genesis_core
docker run -d \
  --name genesis_core \
  --network genesis_net \
  --security-opt apparmor=docker-genesis \
  --security-opt seccomp=/etc/docker/seccomp/genesis_seccomp.json \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --memory 512m \
  --cpus 0.5 \
  --user genesis \
  -v /dev/mapper/genesis_vault:/data:rw \
  genesis:20.1.0
```

"AppArmor + seccomp + Capabilities 최소화. **삼중 런타임 보호.**" 정규가 정리했다.

### Read-Only 파일시스템

김영훈이 한 가지를 더 제안했다.

"genesis의 **/app 디렉토리는 코드만 있잖아.** 런타임에 코드가 변경될 이유가 없어. **read-only 파일시스템**으로 만들자."

```bash
# read-only root filesystem + tmpfs
docker stop genesis_core
docker rm genesis_core
docker run -d \
  --name genesis_core \
  --network genesis_net \
  --security-opt apparmor=docker-genesis \
  --security-opt seccomp=/etc/docker/seccomp/genesis_seccomp.json \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --memory 512m \
  --cpus 0.5 \
  --user genesis \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  -v /dev/mapper/genesis_vault:/data:rw \
  genesis:20.1.0
```

"**--read-only**: 루트 파일시스템을 읽기 전용으로. 공격자가 컨테이너에 침투해도 **악성 코드를 파일시스템에 쓸 수 없어.**" 김영훈이 설명했다.

"**--tmpfs /tmp**: /tmp만 쓰기 가능한 임시 파일시스템. noexec 옵션으로 **실행 파일을 /tmp에서 실행할 수 없게** 했어." 정규가 덧붙였다.

"그러면 /data만 쓰기 가능한 거네요." 가람이 정리했다. "evidence를 쓰는 것만 허용하고, 나머지는 전부 읽기 전용."

### 리소스 제한 강화

추박하가 한 가지를 추가했다.

"**pids-limit**도 걸자. fork bomb 방지."

```bash
# 최종 genesis 컨테이너 실행 (모든 보안 옵션 적용)
docker stop genesis_core
docker rm genesis_core
docker run -d \
  --name genesis_core \
  --network genesis_net \
  --security-opt apparmor=docker-genesis \
  --security-opt seccomp=/etc/docker/seccomp/genesis_seccomp.json \
  --security-opt no-new-privileges:true \
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  --memory 512m \
  --memory-swap 512m \
  --cpus 0.5 \
  --pids-limit 64 \
  --user genesis \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  -v /dev/mapper/genesis_vault:/data:rw \
  --restart unless-stopped \
  genesis:20.1.0
```

"**no-new-privileges**: 프로세스가 실행 중에 **추가 권한을 획득할 수 없게**. setuid/setgid 바이너리 악용 방지." 추박하가 설명했다.

"**memory-swap = memory**: 스왑을 메모리와 같은 크기로 제한. 스왑 파티션을 과도하게 사용해서 호스트를 느리게 만드는 공격 방지." 정규가 말했다.

"**pids-limit 64**: 프로세스 수를 64개로 제한. fork bomb(무한 프로세스 생성 공격)을 차단." 추박하가 말했다.

"**restart unless-stopped**: 컨테이너가 비정상 종료되면 **자동 재시작**. 하지만 `docker stop`으로 수동 종료한 경우는 재시작하지 않아." 김영훈이 덧붙였다.

가람은 수첩에 적었다.

> `[Docker 보안 강화 — 계층별 정리]`
>
> `1. 런타임 보호 (컨테이너 내부 공격 차단):`
> `   AppArmor: 파일/네트워크 접근 프로필 기반 제한`
> `   seccomp: 시스콜 레벨 차단 (화이트리스트)`
> `   no-new-privileges: setuid/setgid 악용 방지`
> `   Capabilities: 최소 권한 원칙`
>
> `2. 파일시스템 보호 (데이터 변조 차단):`
> `   --read-only: 루트 파일시스템 읽기 전용`
> `   --tmpfs /tmp: noexec → 악성 코드 실행 불가`
> `   LUKS 암호화: 호스트에서도 키 없이 접근 불가`
>
> `3. 리소스 제한 (서비스 거부 차단):`
> `   memory + memory-swap: 메모리 폭주 방지`
> `   cpus: CPU 독점 방지`
> `   pids-limit: fork bomb 방지`
>
> `4. 네트워크 격리 (4장에서 정리)`
> `5. 물리적 보안 (법무팀 관할)`
>
> `서문기의 공격 벡터별 차단:`
> `   SSH 접근 → nftables (6권)`
> `   물리 접근 → 법무팀 출입 통제 (4장)`
> `   서버 재시작 → 법무팀 감사 규정 + restart 정책 (6장)`
> `   → "모든 공격 벡터에 대응하는 것이 보안 강화의 본질"`
