### 컨테이너 개론 — 가람의 학습

가람이 Docker를 처음 접하는 것은 아니었다 — 4권 3장에서 이다겸이 Docker Compose로 ETL 파이프라인을 운영한다는 것을 들었다. 하지만 **보안 관점**에서 Docker를 분석하는 것은 처음이었다.

정규가 가람에게 설명했다.

```
[Docker 컨테이너 기초 — 보안 관점]

컨테이너란?
  호스트 OS 위에서 격리된 환경을 제공하는 경량 가상화
  VM과 달리 커널을 공유 → 빠르고 가벼움, 하지만 격리 수준이 VM보다 약함

이미지 (Image):
  컨테이너의 "설계도". 읽기 전용.
  여러 레이어(Layer)로 구성 — 각 레이어는 변경 사항의 스냅샷

컨테이너 (Container):
  이미지를 실행한 인스턴스. 읽기/쓰기 가능.
  컨테이너가 삭제되면 데이터도 사라짐 → Volume으로 영구 저장

볼륨 (Volume):
  컨테이너의 데이터를 호스트에 영구 저장
  genesis는 evidence를 Volume에 저장 → 컨테이너 재시작해도 데이터 보존

네트워크:
  bridge: 기본 네트워크 (같은 bridge의 컨테이너끼리 통신)
  host: 호스트 네트워크 직접 사용 (격리 없음)
  none: 네트워크 없음 (완전 격리)
  custom: 사용자 정의 네트워크 (genesis_net처럼)
```

"genesis가 **custom 네트워크 + ICC 비활성화**를 쓰는 이유는, 다른 컨테이너에서 genesis로 **접근할 수 없게** 하기 위해서야." 정규가 말했다.

"추팀장님이 **genesis를 보호하기 위해** 이렇게 설계한 거네요." 가람이 말했다.

추박하가 고개를 끄덕였다. "genesis에 접근하려면 **이 서버에 직접 로그인**해야 해. 원격에서는 불가능하게 만들었어."

"하지만 서문기가 **admin@nexusai 계정**으로 이 서버에 접근하면?" 가람이 물었다.

"secu 서버의 nftables에서 opsclaw 서버의 SSH 접근을 **관제실 네트워크에서만** 허용했어. 6권에서 구성한 Zone 기반 정책." 정규가 말했다.

"서문기의 임원실 IP(192.168.20.x)에서는 opsclaw 서버에 **SSH 접근 불가**." 가람이 이해했다.

> `[Docker 컨테이너 보안 — 교육 정리]`
>
> `Docker 기본 구조: 이미지(설계도) → 컨테이너(실행 인스턴스)`
> `보안 핵심:`
> `  - 네트워크 격리: custom network + ICC 비활성화`
> `  - Volume: 데이터 영구 저장`
> `  - 비특권 사용자: root가 아닌 genesis 사용자로 실행`
> `  - 포트 미노출: 외부에서 네트워크 스캔으로 발견 불가`
>
> `genesis 아키텍처:`
> `  - Docker 컨테이너(genesis_core), 별도 네트워크(genesis_net)`
> `  - 347일간 무중단 운영`
> `  - 2.3GB 데이터 (텍스트 evidence + 암호화 음성)`
> `  - Dead Man's Switch: 30일 미접근 시 자동 발동`
> `  - 접근 경로: docker exec 또는 Manager 내부 API (로컬 전용)`
>
> `추팀장의 설계 철학:`
> `  "genesis는 기록하는 도구다."`
> `  "접근하려면 이 서버에 직접 와야 한다."`
> `  "원격에서는 불가능하게."`
> `  → 물리적 보안 + 네트워크 보안 + 컨테이너 격리 = 3중 보호`
