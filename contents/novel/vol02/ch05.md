# Chapter 5: 403

---

화요일. 박지후가 WAF 분석을 맡았다.

"공격 288건 중에서 WAF가 차단한 건 **0건**이야." 지후가 말했다.

"0?" 민재가 놀랐다. "ModSecurity가 동작하고 있지 않았어?"

"동작은 하고 있었어. 하지만 **포트가 달라**."

지후가 화이트보드에 그렸다.

```
외부 공격자
  │
  ├→ :80 (Apache 직접)     — ModSecurity 미적용 → SQLi 통과
  ├→ :3000 (JuiceShop 직접) — WAF 없음 → 모든 공격 통과
  ├→ :8082 (Apache 프록시)  — ModSecurity CRS 적용 → SQLi 403 차단
  └→ :5000 (Nexus-1 내부)  — 프록시 경유 시 WAF 미적용 → 통과
```

"공격자가 **:3000으로 직접** 접근했어. :8082가 아니라." 지후가 말했다. "WAF가 있는 :8082로 왔으면 403으로 차단됐을 텐데, :3000은 WAF를 **완전히 우회**해."

"그건 공격자가 우리 서버 구조를 **알고 있다**는 뜻이잖아." 정규가 말했다.

"맞아." 김영훈이 끼어들었다. "어떤 포트가 WAF가 적용되고 어떤 포트가 안 되는지 — 이건 내부 사람만 아는 정보야."

---

지후가 ModSecurity의 실제 동작을 확인했다.

```bash
# WAF(:8082)에 SQLi 테스트
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"curl -s -o /dev/null -w \"%{http_code}\" \"http://localhost:8082/?id=1+OR+1%3D1--\"","subagent_url":"http://10.20.30.80:8002"}'
```

```
403
```

"**403.** :8082에서는 차단돼." 지후가 말했다.

```bash
# JuiceShop(:3000)에 같은 SQLi
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"curl -s -o /dev/null -w \"%{http_code}\" \"http://localhost:3000/rest/products/search?q=test+OR+1%3D1--\"","subagent_url":"http://10.20.30.80:8002"}'
```

```
200
```

"**200.** :3000에서는 그냥 통과해."

"같은 서버, 같은 공격, **포트만 다른데** 하나는 차단하고 하나는 통과." 지후가 한숨을 쉬었다. "이건 **아키텍처 설계** 문제야. WAF를 배치했지만, 우회 경로가 열려 있으면 소용없어."

---

"ModSecurity가 차단할 때 로그는?" 김영훈이 물었다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"echo 1 | sudo -S tail -5 /var/log/apache2/modsec_audit.log 2>/dev/null","subagent_url":"http://10.20.30.80:8002"}'
```

```
[04/Apr/2026:10:15:23] ModSecurity: Access denied with code 403 (phase 2).
  Pattern match "(?i:\\bunion\\b.+?\\bselect\\b)" at ARGS:id.
  [file "/usr/share/modsecurity-crs/rules/REQUEST-942-APPLICATION-ATTACK-SQLI.conf"]
  [id "942100"] [msg "SQL Injection Attack Detected via libinjection"]
  [severity "CRITICAL"]
```

"CRS 룰 942100. SQL Injection 탐지." 지후가 읽었다. "UNION SELECT 패턴 매칭. **:8082로 오면 완벽하게 차단**하고 있어."

"그럼 해결책은?" 도윤이 물었다.

"**:3000과 :80을 직접 접근하지 못하게** 해야 해." 김영훈이 말했다. "nftables에서 외부→:3000, 외부→:80 접근을 차단하고, **:8082만** 외부에 열면 돼."

```bash
# nftables로 직접 접근 차단, WAF 경유만 허용
curl -s -X POST "http://localhost:8000/projects/$PID/execute-plan" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "tasks": [
      {"order":1,"title":"외부→:3000 차단","instruction_prompt":"echo 1 | sudo -S nft add rule inet filter input tcp dport 3000 ip saddr != 10.20.30.0/24 drop","risk_level":"medium","subagent_url":"http://10.20.30.1:8002"},
      {"order":2,"title":"외부→:80 차단","instruction_prompt":"echo 1 | sudo -S nft add rule inet filter input tcp dport 80 ip saddr != 10.20.30.0/24 drop","risk_level":"medium","subagent_url":"http://10.20.30.1:8002"}
    ],
    "subagent_url":"http://localhost:8002",
    "parallel":true
  }'
```

"이제 외부에서는 :8082(WAF 경유)로만 접근 가능해." 김영훈이 확인했다.

---

"하나 더." 지후가 말했다. "공격자가 **파일 업로드**도 시도했어."

```
POST /file-upload HTTP/1.1
Content-Type: multipart/form-data

file=@malicious.php
```

"JuiceShop의 파일 업로드 기능에 **PHP 웹셸**을 올리려 한 거야." 지후가 말했다. "JuiceShop은 Node.js라 PHP가 실행 안 되지만, 만약 Apache에 PHP 모듈이 있으면..."

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/dispatch" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{"command":"apache2ctl -M 2>/dev/null | grep php","subagent_url":"http://10.20.30.80:8002"}'
```

```
 php8.1_module (shared)
```

"**PHP 모듈이 설치되어 있어.**" 지후의 표정이 굳었다. "Apache(:80)에서 PHP가 실행 가능해. 만약 업로드된 파일이 Apache document root에 도달하면..."

"웹셸이 실행될 수 있어." 김영훈이 말을 마무리했다.

---

가람은 그날의 사건을 수첩에 정리했다.

```
[WAF 분석 결과]
- :8082 (WAF O): SQLi → 403 차단 ✓
- :3000 (WAF X): SQLi → 200 통과 ✗
- :80 (WAF X): 직접 접근 가능 ✗
- 대응: nftables로 외부→:3000/:80 차단, :8082만 허용

[추가 발견]
- PHP 모듈 설치 → 웹셸 위험
- 공격자가 포트 구조를 알고 있음 → 내부 정보 유출 재확인

[교훈]
- WAF가 있어도 우회 경로가 열려 있으면 무용지물
- "울타리가 높아도, 한 곳이 낮으면 그리로 넘어온다"
- 보안은 가장 약한 고리만큼만 강하다
```

가람은 수첩을 덮으며 생각했다.

*403. Forbidden. 접근 금지.*

*하지만 진짜 문제는 403이 아니라, 403을 **우회하는 경로**가 열려 있었다는 것이다.*

*이 회사에도 마찬가지. 보안 솔루션은 갖추고 있지만, 뒤쪽 문이 열려 있다. 그리고 누군가는 그 문을 알고 있다.*
