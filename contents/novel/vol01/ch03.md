# Chapter 3: 게으른 천재

---

오가람이 넥서스AI에 입사한 지 일주일이 지났다.

그 사이 가람은 두 가지를 배웠다. 첫째, SOC의 일상은 생각보다 반복적이다. 둘째, 추박하 팀장은 생각보다 **훨씬 더** 게으르다.

매일 10시를 넘겨 출근하고, 자리에 앉자마자 서랍에서 뭔가를 꺼낸다 — 만화책이거나 스마트폰이거나 가끔은 전자책 리더기. 업무 지시를 하는 모습은 본 적이 없다. 팀 미팅은 김영훈 과장이 주관하고, 기술 검토는 각 담당자가 자체적으로 처리한다. 팀장의 결재가 필요한 서류는 민재가 "팀장님 서명란"이라고 표시해서 책상 위에 두면, 언제인지 모르게 사인이 되어 돌아온다.

"팀장님이 정확히 뭘 하시는 거예요?" 가람이 점심 시간에 도윤에게 물었다.

도윤이 젓가락을 멈추고 생각했다.

"글쎄... 존재 자체가 역할인 건가?"

"그게 무슨 말이에요?"

"추팀장이 있다는 사실만으로 **위에서 SOC를 함부로 못 건드려**. 경력 20년에, 데프콘 세계 8위까지 한 사람이야. 회장님이 직접 데려온 거라는 소문도 있고."

"데프콘 8위요?" 가람이 놀라서 되물었다.

"옛날 일이라 확인은 안 되지만, 진호 선배가 과거 CTF 기록을 뒤져서 찾았대. 2010년인가 2011년인가." 도윤이 목소리를 낮추었다. "그 전에는 해커 그룹에서 활동했다는 얘기도 있어."

"블랙햇이요?"

"몰라. 그레이일 수도 있고. 아무도 안 물어봐. 물어보면 대답도 안 하시거든."

---

점심 후, 관제실로 돌아가자 추박하가 의자를 돌려 가람을 바라보고 있었다. 평소의 게으른 자세가 아니었다.

"가람아. 취약점(vulnerability)이 뭔지 알아?"

가람이 당황했다. 추팀장이 먼저 말을 건 건 처음이었다.

"시스템의... 보안 약점 아닌가요? 해커가 침입할 수 있는 구멍 같은."

추박하가 만화책을 내려놓았다. "반만 맞아. 취약점은 **시스템의 약점(weakness)**이야. 하지만 구멍이라고 하면, '구멍을 막으면 된다'고 생각하게 돼. 실제로는 막을 수 없는 취약점도 있어."

"막을 수 없는 취약점이요?"

"사람." 추박하가 말했다. "소셜 엔지니어링. 아무리 시스템을 완벽하게 만들어도, 사람이 비밀번호를 알려주면 끝이야."

추박하가 자리에서 일어나 화이트보드로 걸어갔다. 팀원들이 놀란 얼굴로 바라보았다 — 추팀장이 화이트보드를 쓰는 건 일 년에 두세 번이었다.

"웹 애플리케이션에서 가장 흔한 취약점 목록이 있어. **OWASP Top 10**이라고."

그가 목록을 적었다.

```
OWASP Top 10 (간략)
1. Broken Access Control — 권한 없는 접근
2. Cryptographic Failures — 암호화 실패
3. Injection — SQL, OS, LDAP 인젝션
4. Insecure Design — 설계 단계의 결함
5. Security Misconfiguration — 보안 설정 오류
6. Vulnerable Components — 취약한 구성요소 사용
7. Authentication Failures — 인증 실패
8. Software Integrity Failures — 무결성 검증 실패
9. Logging & Monitoring Failures — 로깅/모니터링 부재
10. SSRF — Server-Side Request Forgery
```

"이 중에서 가장 오래된 취약점이 뭔지 알아?" 추박하가 물었다.

"인젝션?" 가람이 조심스럽게 답했다.

"맞아. SQL 인젝션은 1998년에 처음 보고됐어. 거의 30년 전이야. 그런데 **지금도** 매일 발견돼." 추박하가 웃었다. "XSS(크로스사이트 스크립팅)도 마찬가지야. 사용자 입력을 제대로 검증하지 않으면 발생하는데, 이론적으로는 간단한 문제지만 현실에서는 끊이지 않아."

추박하가 화이트보드에 문 그림을 하나 그렸다.

"취약점은 **문**이야. 열쇠가 없어도 약한 문은 열려."

가람이 고개를 끄덕였다. "그럼 저희가 하는 일은 문을 찾는 건가요?"

"아니. 우리 일은 문이 **열렸는지** 감시하는 거야. 문을 찾아서 고치는 건 개발팀 일이고, 열쇠를 만드는 건 해커의 일이고, **문이 열린 순간을 포착하는** 게 SOC의 일이야."

추박하가 다시 자리로 돌아가 만화책을 집어들었다. 강의 끝.

가람은 수첩에 적었다.

> `취약점(Vulnerability) = 시스템의 약점. 구멍이 아니라 "문".`
> `OWASP Top 10: 웹 애플리케이션 취약점 상위 10가지.`
> `  - Injection: 사용자 입력이 명령어로 실행되는 것 (SQL, OS 등)`
> `  - XSS: 악성 스크립트가 다른 사용자 브라우저에서 실행`
> `  - Broken Access Control: 권한 없는 자가 접근`
> `추팀장: "취약점은 문이야. 열쇠가 없어도 약한 문은 열려."`
> `SOC의 역할: 문을 고치는 게 아니라 문이 열리는 순간을 포착하는 것.`

---

그날 오후, 가람은 증적 정리를 하면서 OpsClaw의 프로젝트 목록을 둘러보고 있었다.

```bash
curl -s http://localhost:8000/projects \
  -H "X-API-Key: opsclaw-api-key-2026" | python3 -c "
import sys, json
projects = json.load(sys.stdin).get('projects', [])
print(f'총 프로젝트: {len(projects)}개')
for p in projects[:10]:
    print(f'  [{p[\"id\"][:12]}] {p[\"name\"]:40s} stage={p[\"current_stage\"]}')
"
```

화면에 나타난 프로젝트 목록:

```
총 프로젝트: 52개
  [prj_c9d49578] compare-opsclaw-r1                       stage=execute
  [prj_e8bb1b98] vwr-agent-benchmark                      stage=execute
  [prj_4cc08792] red-tier1-webapp-attack                   stage=execute
  [prj_6709620c] blue-tier1-defense                        stage=execute
  ...
```

가람은 스크롤을 내렸다. 프로젝트 이름들을 하나씩 읽었다. `exp-A-chain-integrity`, `exp-B-rl-convergence`, `exp-F-par-N5`... 대부분은 실험이나 테스트 프로젝트였다.

그런데 목록의 맨 아래에 이상한 프로젝트가 하나 있었다.

```
  [prj_00000000] genesis                                   stage=completed
```

`prj_00000000`. 모든 프로젝트 ID는 `prj_` 다음에 랜덤한 8자리 해시가 오는데, 이건 전부 0이었다. 생성 날짜를 확인하려 했지만, `created_at` 필드가 없었다.

가람은 해당 프로젝트의 상세를 조회했다.

```bash
curl -s http://localhost:8000/projects/prj_00000000 \
  -H "X-API-Key: opsclaw-api-key-2026" | python3 -m json.tool
```

```json
{
    "status": "error",
    "detail": "Project not found or access denied"
}
```

접근이 거부되었다. 다른 프로젝트는 모두 조회 가능한데, 이것만 안 된다.

가람은 잠시 화면을 바라보다가 수첩을 꺼내 적었다.

> `prj_00000000 "genesis" — 접근 거부. 왜?`

---

"뭐 하고 있어?"

가람이 고개를 들자 김영훈이 서 있었다.

"아, 프로젝트 목록 정리 중이었는데요, 이상한 프로젝트가 하나—"

"genesis?" 김영훈이 화면을 흘끗 보더니 말했다. "그건 시스템 초기화 때 만들어진 테스트 프로젝트야. 신경 쓸 필요 없어."

"그런데 접근이 안 되는데요."

"권한 문제겠지. 관리자 전용일 거야." 김영훈의 톤은 담담했지만, 가람은 그가 0.5초 정도 화면을 **더 오래** 바라본 것을 느꼈다.

"네, 알겠습니다."

김영훈이 돌아간 후, 가람은 다시 수첩을 보았다.

> `김영훈 과장도 genesis를 알고 있다. "신경 쓸 필요 없다"고 했지만, 표정이 살짝 변했다.`

---

오후 4시. 가람은 관제실 한쪽의 교육 공간에서 도윤과 함께 OpsClaw 교육을 받고 있었다. 강사는 최민재.

"자, 오늘은 **dispatch**와 **execute-plan**의 차이를 배울 거야."

민재가 화이트보드에 그림을 그렸다.

```
dispatch: 단일 명령 즉시 실행
  POST /projects/{id}/dispatch
  → 한 서버에 한 명령

execute-plan: 여러 명령을 묶어서 실행
  POST /projects/{id}/execute-plan
  → 여러 서버에 여러 명령, 병렬 가능
```

"dispatch는 '지금 이 서버에서 이 명령 실행해'야. execute-plan은 '이 목록의 태스크들을 순서대로(또는 병렬로) 실행해'야."

민재가 터미널을 열었다.

"실습 해보자. 4대 서버의 상태를 동시에 점검하는 execute-plan을 만들어봐."

도윤이 먼저 시도했다.

```bash
curl -s -X POST "http://localhost:8000/projects/$PID/execute-plan" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: opsclaw-api-key-2026" \
  -d '{
    "tasks": [
      {"order":1,"title":"opsclaw 점검","instruction_prompt":"hostname && uptime","risk_level":"low","subagent_url":"http://localhost:8002"},
      {"order":2,"title":"secu 점검","instruction_prompt":"hostname && uptime","risk_level":"low","subagent_url":"http://10.20.30.1:8002"},
      {"order":3,"title":"web 점검","instruction_prompt":"hostname && uptime","risk_level":"low","subagent_url":"http://10.20.30.80:8002"},
      {"order":4,"title":"siem 점검","instruction_prompt":"hostname && uptime","risk_level":"low","subagent_url":"http://10.20.30.100:8002"}
    ],
    "subagent_url":"http://localhost:8002",
    "parallel":true
  }'
```

0.6초 후 결과가 돌아왔다.

```
overall: success
tasks_ok: 4
tasks_failed: 0
```

"4대 서버를 **동시에** 0.6초 만에 점검했어." 민재가 웃었다. "하나씩 SSH로 접속해서 했으면 최소 2초는 걸렸을 거야."

가람이 질문했다. "이걸 직접 SSH로 하면 안 돼요?"

민재의 표정이 진지해졌다.

"할 수는 있어. 근데 **기록이 안 남아**." 그가 OpsClaw의 evidence summary를 열었다. "봐. 방금 execute-plan으로 4개 태스크를 실행했더니, evidence 4건이 자동으로 생겼어. PoW 블록 4개. 보상(reward) 4건. 타임라인도."

```bash
curl -s "http://localhost:8000/projects/$PID/replay" \
  -H "X-API-Key: opsclaw-api-key-2026" | python3 -c "
import sys,json
d = json.load(sys.stdin)
print(f'총 보상: {d[\"total_reward\"]}')
for s in d['timeline']:
    print(f'  [{s[\"task_order\"]}] {s[\"task_title\"]:20s} exit={s[\"exit_code\"]} reward={s[\"total_reward\"]} dur={s[\"duration_s\"]}s')
"
```

```
총 보상: 5.2
  [1] opsclaw 점검         exit=0 reward=1.3 dur=0.171s
  [2] secu 점검            exit=0 reward=1.3 dur=0.150s
  [3] web 점검             exit=0 reward=1.3 dur=0.185s
  [4] siem 점검            exit=0 reward=1.3 dur=0.153s
```

"직접 SSH로 하면 이 **replay가 불가능**해. 나중에 '네가 그때 뭘 했어?'라고 물으면, 터미널 히스토리를 뒤져야 해. 그것도 세션이 끊기면 사라지고."

민재가 화이트보드에 큰 글씨로 적었다.

```
직접 SSH:  명령 → 결과 → (증적 없음)
OpsClaw:   명령 → 결과 → evidence + PoW + reward + replay
```

도윤이 중얼거렸다. "그래서 팀장님이 '모든 작업을 OpsClaw를 통해서 하라'고 하신 거군요."

"맞아." 민재가 고개를 끄덕였다. "팀장님이 딱 한 가지 엄격한 게 있다면, **기록**이야. '증거 없이 일했다고 주장하지 마라.' 그게 팀장님의 유일한 원칙이야."

---

교육이 끝난 후, 가람은 자리로 돌아가다가 추팀장의 자리를 지나쳤다. 추박하는 의자를 뒤로 젖히고 천장을 바라보고 있었다. 만화책은 펼쳐진 채 배 위에 올려져 있었다.

가람이 조심스럽게 물었다.

"팀장님, 질문 하나 해도 될까요?"

"음?"

"OpsClaw의 PoW 체인에 있는 블록... 47,000개가 넘잖아요. 그게 다 실제 작업 기록인 건가요?"

추박하가 천천히 고개를 돌렸다. 처음으로 가람을 **제대로** 바라보는 것 같았다.

"왜 물어?"

"그냥... 회사 역사가 20년이고 블록이 47,000개면, 하루 평균 6~7개 정도인데, 그게 실제 보안 작업량에 비하면 적은 것 같기도 하고, 많은 것 같기도 해서요."

추박하의 눈이 미묘하게 움직였다. 가람은 그 표정을 읽지 못했다.

"OpsClaw가 처음부터 있었던 건 아니야." 추박하가 말했다. "정식 가동은 2년 전이야. 그 전 블록은..."

그가 잠시 멈추었다.

"프로토타입 시절 테스트 데이터야."

*47,832 블록은 OpsClaw **전체**의 PoW 체인이다. 모든 프로젝트의 작업 기록을 합친 것. 하지만 그 중 genesis 프로젝트만의 블록이 40,000개 가까이 된다면... "테스트"라는 설명이 맞을까?*

"아, 그렇군요."

추박하가 다시 천장을 바라보았다. 대화 끝.

하지만 가람은 무언가 걸렸다. **프로토타입 테스트 데이터가 47,000블록?** OpsClaw 정식 가동 2년이면, 하루 10개씩 실행해도 7,300블록이다. 나머지 40,000개가 테스트?

가람은 수첩에 적었다.

> `47,832 블록. 정식 가동 2년 = ~7,300. 나머지 ~40,000 = "테스트"?`
> `추팀장이 대답할 때 0.5초 멈춤. "테스트 데이터야"라고 말하기 전에.`
> `genesis 프로젝트와 관련?`

가람은 수첩을 덮었다. 아직 신입이었고, 아직 아는 것이 너무 적었다.

하지만 한 가지는 느꼈다. 이 관제실에는 **보이는 것보다 많은 것**이 숨겨져 있다.

---

그날 밤, 가람은 퇴근길에 도윤에게 문자를 보냈다.

> "선배, 추팀장님이 OpsClaw를 직접 설계하셨다고 했죠?"

잠시 후 답이 왔다.

> "ㅇㅇ. 아키텍처는 팀장님이 설계하고, 실제 개발은 김영훈 과장이 팀 이끌고 했대."

> "팀장님이 설계할 때 특별히 강조하신 게 뭐였어요?"

긴 침묵 후.

> "'모든 것을 기록한다.' 그 한 마디였다고."

가람은 지하철 창밖을 바라보았다.

*모든 것을 기록한다.*

그 말이 단순히 "보안 작업의 증적을 남긴다"는 뜻일까? 아니면...

지하철이 터널로 들어갔다. 창밖이 깜깜해졌다. 가람의 얼굴이 유리에 반사되었다.

*아니면, 다른 무언가를 기록하는 것일까.*
