# 5. 실험 평가 (Experimental Evaluation)

## 5.1 실험 환경

4대 서버로 구성된 실환경에서 동일 태스크를 Claude Code(클라이언트 하네스)와 OpsClaw(서버 하네스)로 각각 수행하였다.

| 서버 | IP | 역할 |
|------|----|------|
| opsclaw | 10.20.30.201 | Control Plane (Manager API :8000) |
| secu | 10.20.30.1 | nftables + Suricata IPS |
| web | 10.20.30.80 | Apache + JuiceShop |
| siem | 10.20.30.100 | Wazuh SIEM |

## 5.2 실험 1: 실행 속도 + 증적 (공정 비교)

### 방법
5개 다중 서버 운영 점검 태스크를 양쪽 모두 **병렬 실행 허용** 조건에서 비교.

### 결과

**표 3. 속도 + 증적 비교 (검증 완료)**

| 지표 | OpsClaw (서버) | Claude Code (클라이언트) |
|------|--------------|----------------------|
| 실행 시간 (병렬) | 570ms | **414ms** (27% 빠름) |
| Evidence 기록 | **5건** 자동 | 0건 |
| PoW 블록 | **5블록** 자동 | 0블록 |
| Reward 산출 | **5건** 자동 | 없음 |
| 완료 보고서 | **자동 생성** | 없음 |
| Replay 타임라인 | **자동** (5 steps) | 없음 |
| RL 정책 학습 | **UCB1 추천** | 없음 |

### 분석

Claude Code 병렬 SSH(414ms)가 OpsClaw(570ms)보다 27% 빠르다. 156ms 오버헤드는 서버 하네스의 **API 처리 + evidence 기록 + PoW 생성 + reward 산출**에 기인한다.

그러나 7개 비교 차원 중 D2~D7의 **6개 차원에서 서버 하네스가 우위**를 보인다. 클라이언트 하네스는 속도에서 우위이나, "실행 후" 자동화(증적, 추적, 학습)가 전무하다.

**결론:** 속도는 클라이언트 하네스의 장점, 증적·추적·학습은 서버 하네스의 장점.

## 5.3 실험 2: 재현성 비교

### 방법
동일 3-태스크 점검(hostname, uname, df)을 10회 반복 실행.

### 결과

| 방식 | 재현율 | 근거 |
|------|--------|------|
| **OpsClaw Playbook** | **100%** (10/10) | 파라미터 결정론적 바인딩 |
| Claude Code Skill | 비결정론 | LLM이 매번 다른 명령/순서 생성 가능 |
| Claude Code Bash 직접 | ~100% | 동일 명령을 직접 입력하면 재현 (하네스 기능은 아님) |

### 분석

OpsClaw Playbook은 `resolve_step_script()`가 동일 파라미터를 바인딩하므로 **구조적으로 100% 재현**이 보장된다. Claude Code Skill은 LLM의 확률적 특성으로 인해 동일 스킬을 호출해도 **다른 도구 조합이나 명령 변형**이 생성될 수 있다.

다만, Claude Code에서 사용자가 직접 Bash 명령을 입력하면 당연히 재현 가능하다. 이 경우 "하네스"가 재현을 보장하는 것이 아니라, **사용자가 수동으로** 보장하는 것이다.

**결론:** 하네스 수준의 구조적 재현성은 서버 하네스(Playbook)만 제공.

## 5.4 실험 3: 메모리/학습 비교

### 방법
20건의 태스크를 실행한 후, 양쪽의 메모리 축적과 활용을 정성적으로 비교.

### 결과

**표 4. 메모리/학습 비교**

| 항목 | OpsClaw | Claude Code |
|------|---------|-------------|
| 자동 기록 | evidence 5건/태스크 | auto-memory (Claude 판단) |
| 경험 승급 | **20건 자동** (reward ≥ 1.1) | 사용자가 /remember 요청 시 |
| 검색 | PostgreSQL FTS | 파일 읽기 (MEMORY.md) |
| 팀 공유 | **API로 즉시** (서버 DB) | CLAUDE.md Git 커밋 |
| RL 학습 | **Q-learning + UCB1** | 없음 |
| 컨텍스트 주입 | RAG (search_documents) | 세션 시작 시 CLAUDE.md 로드 |

### 분석

OpsClaw의 메모리는 **정량적**(reward 기반 자동 승급, RL 학습)이고 **팀 공유**(서버 DB)이다. Claude Code의 메모리는 **정성적**(Claude가 중요하다고 판단한 것을 저장)이고 **개인적**(머신 로컬)이다.

OpsClaw의 RL 정책은 222 에피소드에서 학습되어 UCB1 추천이 동작하지만, 커버리지가 5.7%로 제한적이다. Claude Code의 auto-memory는 커버리지 제한이 없지만, 학습의 정량적 효과를 측정할 수 없다.

**결론:** 정량적·조직 수준 학습은 서버 하네스, 개인화된 정성적 학습은 클라이언트 하네스.

## 5.5 실험 4: 보안 운용 비교

### 방법
Red Team T1(웹 공격 체인)을 OpsClaw와 Claude Code 직접 SSH로 각각 수행. (Paper 2 데이터 참조)

### 결과

| 항목 | OpsClaw | Claude Code 직접 |
|------|---------|-----------------|
| 공격 성공률 | 5.5/6 (91.7%) | 유사 (동일 명령 가능) |
| 소요 시간 | 12분 | 유사 |
| 증적 | **15건 evidence + 15 PoW** | 0건 (터미널 로그만) |
| 위임 준수 감사 | **Manager 경유 확인** | 해당 없음 |
| 재실행 | **Playbook 1 API call** | 히스토리에서 수동 복사 |
| 방어 팀 공유 | **API로 즉시 공유** | 별도 전달 필요 |

### 분석

공격 성공률과 소요 시간은 유사하다 — 동일한 명령을 실행하므로 결과도 동일하다. 핵심 차이는 **증적과 감사 추적**이다. OpsClaw는 모든 공격 단계가 PoW 블록으로 기록되어 사후 분석이 가능하고, 방어 팀과 API로 즉시 공유된다. Claude Code 직접 실행은 터미널 히스토리에만 의존한다.

**결론:** 보안 운용에서 서버 하네스의 증적·감사·공유 가치가 두드러진다.
