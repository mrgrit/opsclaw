# 1. 서론 (Introduction)

## 1.1 연구 배경

LLM 에이전트를 IT 운영에 적용하는 프레임워크가 다수 제안되고 있으나, 대부분 아키텍처 설계와 벤치마크 결과만 보고할 뿐, 실제 구축·운용 과정에서의 설계 결정, 발견된 문제, 축적된 교훈을 공유하는 사례 연구는 부족하다. 연구 프로토타입과 실운영 시스템 사이의 간극(gap)은 크며, 이를 좁히기 위한 실증적 경험 보고가 필요하다.

본 논문은 IT 운영·보안 자동화 플랫폼 **OpsClaw**를 설계 기준선(M0)부터 보안 실험 운용(M25)까지 25개 마일스톤에 걸쳐 점진적으로 구축·운용한 사례를 보고한다. "이렇게 만들었다"가 아니라 **"이렇게 만들면서 이런 것을 배웠다"**에 초점을 맞추어, 유사 플랫폼을 구축하려는 실무자와 연구자에게 재현 가능한 교훈을 제공한다.

## 1.2 주요 기여

- **C1.** 25 마일스톤 개발 여정의 핵심 설계 결정과 그 결과를 체계적으로 기술한다.
- **C2.** 실운영 중 발견된 5건의 버그와 2건의 보안 취약점의 원인·해결·교훈을 분석한다.
- **C3.** 하네스(OpsClaw) vs 단독 에이전트(Claude Code)의 공정 비교 실험을 통해, 하네스의 가치가 실행 속도가 아닌 "실행 후" 자동화(증적·추적·학습)에 있음을 실증한다.
- **C4.** 검증 가능한 작업 기록(Verifiable Work Record)과 보상 체계의 활용 시나리오(성과 평가, 비용 정산, 컴플라이언스, 에이전트 마켓플레이스)를 제시한다.

## 1.3 논문 구성

2장에서 OpsClaw 플랫폼을 개요하고, 3장에서 25 마일스톤 개발 여정을 기술한다. 4장에서 하네스 vs 단독 에이전트 공정 비교 실험 결과를 제시하고, 5장에서 검증 가능한 작업 기록의 활용 시나리오를 논의한다. 6장에서 축적된 교훈을 정리하고, 7장에서 결론을 맺는다.
