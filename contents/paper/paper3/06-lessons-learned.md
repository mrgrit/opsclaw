# 6. 축적된 교훈 (Lessons Learned)

25 마일스톤 개발 여정에서 축적된 교훈을 아키텍처, 운영, 보안 세 범주로 정리한다.

## 6.1 아키텍처 교훈

**표 1. 아키텍처 교훈 요약**

| # | 교훈 | 근거 | 적용 시점 |
|---|------|------|---------|
| L1 | 설계 기준 문서를 코드보다 먼저, 코드보다 상위에 고정하라 | M0 문서가 25 마일스톤 동안 방향 유지 | 프로젝트 시작 |
| L2 | 외부 런타임 의존은 어댑터 경계를 반드시 설정하라 | pi_adapter가 pi runtime 변경을 격리 | 외부 연동 시 |
| L3 | 상태 머신은 무효 전이를 데이터로 차단하라 | VALID_TRANSITIONS 딕셔너리 (실험 E: 100% 차단) | 라이프사이클 설계 |
| L4 | Master 교체 가능성이 아키텍처의 핵심 분기점이다 | Mode A/B 분리로 LLM 교체 가능 | 아키텍처 결정 |

**위임 분리의 가치와 비용.** 3계층 위임(Master→Manager→SubAgent)은 감사 추적과 Master 교체 가능성을 보장하지만, 단일 태스크에서 API 호출 오버헤드(~156ms)를 수반한다. 비교 실험에서 Claude Code 직접 SSH(414ms)가 OpsClaw(570ms)보다 빠른 이유이다. 이 트레이드오프는 **반복 운영 환경에서 증적·추적·학습의 누적 가치가 1회당 오버헤드를 상쇄**하는 시점에서 정당화된다.

**Playbook 결정론 vs LLM 유연성.** Playbook은 100% 재현성을 보장하지만(실험 G), 새로운 상황에 대한 적응력이 없다. LLM adhoc 실행은 적응적이지만 재현율이 ~30-50%이다. 실무에서는 **탐색 단계에서 LLM adhoc, 확정 후 Playbook화**하는 이중 경로가 효과적이었다.

## 6.2 운영 교훈

**표 2. 운영 교훈 요약**

| # | 교훈 | 근거 |
|---|------|------|
| L5 | 타임스탬프에 절대 의존하지 마라 | B-02: TZ 차이로 해시 불일치 |
| L6 | 동시성 문제는 초기 설계에서 처리하라 | 병렬 PoW 생성 시 prev_hash 충돌 |
| L8 | 실험 전 인프라 호환성 검증을 체크리스트화하라 | Wazuh Agent/Manager 버전 불일치 |

**VWR의 실용적 가치.** 검증 가능한 작업 기록(VWR)의 가장 즉각적인 가치는 "위변조 방지"가 아니라 **디버깅과 운영 가시성**이었다. 태스크가 실패했을 때 replay API로 정확한 실행 순서, 각 단계의 exit_code와 stdout을 즉시 확인할 수 있었다. 이는 터미널 로그를 뒤지는 것보다 훨씬 효율적이었다.

**한국어 FTS 미지원의 실무 영향.** RAG 기반 경험 검색(실험 D)에서 한국어 쿼리 적중률이 0%였다. 한국어가 주 언어인 운영 환경에서 이는 경험 재활용 기능을 사실상 무력화한다. `pg_trgm` 확장 또는 별도 한국어 형태소 분석기 연동이 시급하다.

**SubAgent 배포·업데이트의 운영 부담.** 4대 서버에 SubAgent를 배포하고 업데이트하는 작업은 `deploy_subagent.sh` 스크립트로 자동화하였으나, SSH 키 관리, 서비스 재시작, 버전 동기화 등의 운영 부담이 존재한다. 컨테이너화(Docker)를 통한 배포 표준화가 향후 과제이다.

## 6.3 보안 운용 교훈

**표 3. 보안 운용 교훈 요약**

| # | 교훈 | 근거 |
|---|------|------|
| L7 | Control-plane 보안은 보호 대상만큼 중요하다 | M28: API 무인증 RCE 발견 |
| — | sudo NOPASSWD ALL은 SIEM 무력화의 근본 원인 | Red T3~T4: agent 정지, syslog 차단 |
| — | 탐지 없는 차단은 근본적 해결이 아니다 | Purple: nftables로 차단했으나 탐지 체계는 미흡 |

**LLM 에이전트의 sudo 발견 능력.** Red Team T3에서 LLM 에이전트는 `sudo -l` 한 줄로 `NOPASSWD: ALL` 취약점을 즉시 발견하였다. 이는 숙련된 침투 테스터의 첫 번째 행동과 동일하며, LLM이 보안 점검의 기본적 판단을 수행할 수 있음을 보여준다.

**네트워크 차단 vs 탐지의 상호보완.** Purple Team에서 최종 방어는 nftables 화이트리스트로 달성되었으나, "왜 차단되었는지"를 SIEM에서 확인할 수 없었다(탐지율 29%). 차단과 탐지는 상호보완적이며, 하나만으로는 완전한 방어가 아니다.

**교훈 L9: 경량 모델의 CLI 환각은 로컬 지식으로 해결하라.** 8B~12B 경량 모델은 도구의 실제 CLI를 정확히 모른다(예: `wazuh-api create-rule` 존재하지 않는 명령). 각 서버의 실제 경로·명령어·성공 경험을 local_knowledge.json으로 보유하면, 시스템 프롬프트에 주입되어 환각이 감소한다. Blue의 SSH 접속 성공률이 0%→100%로 개선된 것이 대표적 사례이다.

**교훈 L10: 지식은 중앙 집중과 분산 캐시를 병행하라.** 전체 지식은 Manager DB에서 통합 관리하되, 각 SubAgent가 자기 서버의 환경을 로컬에 캐시하면 (1) 미션 시작 속도 향상, (2) 경량 모델의 정확도 향상, (3) 네트워크 단절 시에도 기본 운영 가능이라는 이점이 있다.

## 6.4 유사 플랫폼 구축 시 권장사항

본 경험을 바탕으로 LLM 에이전트 기반 IT 자동화 플랫폼 구축 시 다음을 권장한다:

1. **설계 기준 문서를 먼저 작성하라.** 역할 분리, 서비스 경계, 의존 방향, 완료 판정 기준을 코드 작성 전에 고정한다.
2. **Master 교체 가능성을 초기에 설계하라.** 특정 LLM에 종속되지 않는 API 경계를 설계한다.
3. **증적 기록을 선택이 아닌 기본으로 만들어라.** "evidence 없는 완료 주장 금지"를 시스템 불변식으로 강제한다.
4. **해시 계산에 타임스탬프 객체를 사용하지 마라.** 원본 문자열을 보존하여 사용한다.
5. **Control-plane 보안을 1순위로 처리하라.** "내부용"이라는 가정은 위험하다.
6. **실험 전 인프라 호환성을 체크리스트로 검증하라.** 버전 불일치는 실험 전체를 무효화할 수 있다.
7. **경량 모델에는 로컬 지식을 제공하라.** 서버별 도구 경로·성공 명령어를 local_knowledge.json으로 유지하면 CLI 환각이 대폭 감소한다.
8. **지식 축적을 자동화하라.** 미션 성공 시 사용한 명령을 자동으로 로컬 지식에 추가하면, 반복할수록 정확도가 향상된다.
