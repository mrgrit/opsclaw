# 2. 배경 연구 (Background and Related Work)

본 장에서는 4계층 통합 교육 파이프라인의 설계 동기를 뒷받침하는 기존 연구를 분석한다. 보안 교육의 도전 과제, CTF 플랫폼의 특성과 한계, LLM 에이전트의 교육 활용, 블록체인 기반 교육 증적, OpsClaw 아키텍처의 순서로 살펴본다.

## 2.1 보안 교육의 도전 과제

### 2.1.1 역량 프레임워크

사이버보안 교육의 체계화를 위해 다양한 역량 프레임워크가 제안되어 왔다. NICE(National Initiative for Cybersecurity Education) Cybersecurity Workforce Framework [6]은 7개 범주(Securely Provision, Operate and Maintain, Oversee and Govern, Protect and Defend, Analyze, Collect and Operate, Investigate)와 33개 전문 영역으로 사이버보안 직무를 분류한다. CyBOK(Cyber Security Body of Knowledge) [7]은 21개 지식 영역(Knowledge Area)으로 보안 교육 콘텐츠를 구조화한다.

이들 프레임워크는 "무엇을 가르쳐야 하는가"에 대한 체계를 제공하지만, "어떻게 통합적으로 가르칠 것인가"에 대한 구현 방법론은 제시하지 않는다. 특히 이론 학습, 실습 활동, 평가가 단일 플랫폼에서 연속적으로 이루어지는 파이프라인에 대한 구체적 제안은 부재하다.

### 2.1.2 실습 중심 교육의 부상

SANS Institute [10], Offensive Security(OSCP) [11], TryHackMe [12] 등은 실습 중심 보안 교육의 대표적 사례이다. 이들은 가상 머신(VM) 기반 실습 환경을 제공하여 학습자가 실제 명령을 실행하도록 한다. 그러나:

- 실습 환경이 교육 목적으로 설계된 격리 환경이므로, 실제 운영 환경의 복잡성(다중 서버, 방화벽, IPS, SIEM 연동)을 반영하지 못한다.
- 학습 활동의 추적이 세션 로그에 의존하며, 구조적 증적 관리가 부재하다.
- 이론 강의와 실습이 별도 시스템에서 별도 시점에 이루어진다.

Raman et al. [13]은 보안 교육에서 "이론-실습 격차(theory-practice gap)"가 학습 효과를 저해하는 핵심 요인임을 지적하였다.

### 2.1.3 학습 이론적 기반

보안 교육 파이프라인의 설계는 학습 이론에 기반해야 한다. Kolb [44]의 경험학습 이론(Experiential Learning Theory)에 따르면, 효과적인 학습은 구체적 경험(Concrete Experience) → 반성적 관찰(Reflective Observation) → 추상적 개념화(Abstract Conceptualization) → 능동적 실험(Active Experimentation)의 4단계 순환으로 이루어진다.

본 논문의 4계층은 Kolb의 순환과 다음과 같이 대응한다:

| Kolb 단계 | 파이프라인 계층 | 활동 예시 |
|-----------|----------------|-----------|
| 구체적 경험 | Novel (소설) | 등장인물이 보안 사고를 경험하는 장면 읽기 |
| 반성적 관찰 | Education (교안) | 해당 기술의 이론과 원리 학습 |
| 추상적 개념화 | System (실행) | 실제 인프라에서 명령 실행 및 결과 관찰 |
| 능동적 실험 | CTF (평가) | CTF 챌린지를 통해 기술을 응용·검증 |

Vygotsky [45]의 근접발달영역(Zone of Proximal Development) 개념은 적응형 난이도 조정의 이론적 근거를 제공한다. 학습자의 현재 수준에서 "약간 어려운" 과제를 제시할 때 학습 효과가 극대화되며, OpsClaw의 RL 정책 엔진은 이러한 최적 난이도를 자동으로 산출할 수 있다.

Gagne [46]의 학습 조건 이론은 효과적인 교수 설계를 위한 9가지 교수 사건(Events of Instruction)을 제안한다. 본 파이프라인에서:

1. 주의 획득 → 소설의 서사적 도입
2. 학습 목표 제시 → 교안의 학습 목표 섹션
3. 선수 학습 회상 → 이전 교안/소설 장과의 연결
4. 자극 자료 제시 → 교안의 이론 섹션
5. 학습 안내 제공 → 교안의 실습 단계
6. 수행 유도 → OpsClaw dispatch를 통한 명령 실행
7. 피드백 제공 → evidence의 exit_code와 stdout
8. 수행 평가 → CTF flag 검증
9. 파지와 전이 촉진 → PoW 기반 학습 이력 보존

## 2.2 CTF 플랫폼

### 2.2.1 주요 플랫폼

CTF(Capture The Flag)는 보안 교육에서 가장 널리 사용되는 게임화 평가 방식이다 [2]. 주요 플랫폼의 특성을 비교하면 다음과 같다:

| 플랫폼 | 유형 | 이론 연계 | 증적 관리 | 서사 맥락 | 실환경 |
|--------|------|-----------|-----------|-----------|--------|
| CTFd [3] | Jeopardy | × | Flag만 기록 | × | × |
| PicoCTF [4] | Jeopardy | 부분 | × | × | × |
| HackTheBox [5] | Attack-Defense | × | × | × | △ |
| TryHackMe [12] | Guided | 부분 | × | × | △ |
| SEED Labs [14] | Lab | ○ | × | × | △ |
| **제안 파이프라인** | **통합** | **○** | **PoW 체인** | **○** | **○** |

표에서 보듯이, 기존 플랫폼은 이론 연계, 증적 관리, 서사 맥락, 실환경 연동 중 하나 이상의 영역에서 한계를 노출한다. 본 논문이 제안하는 파이프라인은 이 네 가지 요소를 모두 통합한다.

### 2.2.2 CTF의 교육적 효과와 한계

Karagiannis와 Magkos [15]는 CTF가 학습자의 문제 해결 능력과 팀워크를 촉진하지만, 체계적 지식 구축에는 한계가 있음을 보고하였다. Svabensky et al. [16]은 CTF를 교육 커리큘럼에 통합할 때 이론적 프레임워크와의 매핑이 필수적임을 제안하였다. 이는 본 논문이 120개 교안과 CTF 챌린지를 1:1로 매핑하는 설계의 동기가 된다.

## 2.3 LLM 에이전트의 교육 활용

### 2.3.1 대화형 튜터

GPT-4, Claude 등 대규모 언어 모델을 교육 튜터로 활용하는 연구가 활발하다. Kasneci et al. [8]은 ChatGPT의 교육적 기회와 도전을 종합적으로 분석하였으며, Becker et al. [9]은 프로그래밍 교육에서 LLM의 영향을 조사하였다. 이들 연구는 LLM이 **정보 전달**과 **질의응답**에서 효과적임을 보이지만, LLM이 실제 인프라에서 명령을 실행하고 그 결과를 구조적으로 관리하는 **오케스트레이션 기반 교육**은 다루지 않는다.

### 2.3.2 자율 에이전트

AutoGPT [17], Claude Code [18], Devin [19] 등 자율 코딩/운영 에이전트는 자연어 지시를 실행 가능한 명령으로 변환한다. 보안 분야에서는 PentestGPT [20]가 LLM 기반 자동 침투 테스트를 시연하였다. 그러나 이들 에이전트는 **교육 목적의 구조적 증적 관리**를 제공하지 않으며, 학습 파이프라인과의 통합은 보고되지 않았다.

### 2.3.3 AI 에이전트 오케스트레이션

OpsClaw는 Master→Manager→SubAgent 3계층 위임 구조로 에이전트를 오케스트레이션한다 [21]. 이 아키텍처는 본래 IT 운영·보안 자동화를 위해 설계되었으나, 본 논문에서는 이를 교육 파이프라인의 오케스트레이터로 재활용(repurpose)한다. 핵심 기능인 evidence 자동 기록, PoW 체인, RL(강화학습) 기반 정책 최적화는 교육 맥락에서 각각 학습 증적 관리, 학업 무결성 보장, 적응형 난이도 조정으로 재해석된다.

## 2.4 블록체인과 교육 증적

### 2.4.1 학력·자격 검증

블록체인을 학위·자격증 검증에 활용하는 연구가 활발하다. MIT Media Lab의 BlockCerts [22]는 블록체인 기반 학위 증명을 제안하였으며, Sony Global Education [23]은 분산 원장 기반 학습 기록 관리를 시도하였다. Grech와 Camilleri [24]는 유럽위원회 보고서에서 블록체인이 교육 증적의 이식성(portability)과 검증 가능성(verifiability)을 높일 수 있음을 제안하였다.

### 2.4.2 학습 활동 추적

Chen et al. [25]은 블록체인 기반 학습 분석(learning analytics)이 학습자의 활동 기록을 위변조 불가능하게 보존할 수 있음을 보였다. Turkanovic et al. [26]은 EduCTX 플랫폼을 통해 ECTS(European Credit Transfer System) 학점을 블록체인으로 관리하는 프로토타입을 제시하였다.

그러나 기존 연구는 **최종 결과물(학위, 학점)**의 블록체인 기록에 초점을 맞추며, **학습 과정(process)**의 세부 실행 기록을 해시 체인으로 보존하는 접근은 보고되지 않았다. OpsClaw의 PoW 체인은 학습자가 실행한 모든 명령과 그 결과를 개별 블록으로 기록하여, 학습 **과정** 자체의 무결성을 보장한다.

### 2.4.3 해시 체인의 기술적 원리

Nakamoto [43]가 Bitcoin에서 제안한 해시 체인(hash chain)은 각 블록이 이전 블록의 해시를 포함하여, 임의의 블록 변조가 후속 블록의 해시 불일치를 유발하는 구조이다. OpsClaw는 이 원리를 교육 증적에 적용한다:

```
Block 1: hash₁ = SHA-256(genesis_hash + evidence₁ + nonce₁)
Block 2: hash₂ = SHA-256(hash₁ + evidence₂ + nonce₂)
Block 3: hash₃ = SHA-256(hash₂ + evidence₃ + nonce₃)
...
```

이 체인에서 Block 1의 evidence를 변조하면 hash₁이 변경되고, 이는 Block 2의 prev_hash와 불일치하여 체인 전체의 무결성 검증 실패를 유발한다. 이 메커니즘은 학습자의 실습 기록이 사후에 위변조될 수 없음을 암호학적으로 보장한다.

## 2.5 OpsClaw 아키텍처 개요

OpsClaw는 IT 운영·보안 자동화를 위한 자기 개선형 에이전트 하네스로, 다음의 핵심 구성 요소를 가진다 [21]:

```
┌──────────────────────────────────────┐
│  Master (LLM / External Claude Code) │  ← 계획 수립
├──────────────────────────────────────┤
│  Manager API (:8000)                 │  ← 프로젝트 관리, 증적, PoW, RL
├──────────────┬───────────┬───────────┤
│ SubAgent     │ SubAgent  │ SubAgent  │  ← 명령 실행
│ (secu)       │ (web)     │ (siem)    │
└──────────────┴───────────┴───────────┘
```

**핵심 기능:**

1. **Evidence 자동 기록**: 모든 태스크 실행 결과(명령, stdout, stderr, exit_code, 소요 시간)가 evidence 테이블에 자동 저장된다.
2. **PoW(Proof of Work) 체인**: 각 evidence에 대해 SHA-256 해시와 nonce를 포함하는 블록이 생성되며, 이전 블록의 해시로 연쇄 연결된다.
3. **RL(강화학습) 정책 엔진**: Q-learning + UCB1 기반으로 태스크의 위험도별 최적 실행 전략을 학습한다.
4. **4-Layer 경험 메모리**: evidence→task_memory→experience→retrieval의 4계층 메모리 구조로 과거 경험을 축적·재활용한다.
5. **Replay API**: 프로젝트의 전체 실행 이력을 시간순으로 재생하여 감사 추적(audit trail)을 지원한다.

이러한 기능은 본래 IT 운영 자동화를 위해 설계되었으나, 본 논문에서는 이를 교육 파이프라인에 재활용하여 학습 증적 관리, 학업 무결성 보장, 적응형 난이도 조정을 구현한다.

## 2.6 연구 격차 요약

기존 연구를 종합하면, 다음의 세 가지 연구 격차(research gap)가 식별된다:

1. **통합 파이프라인의 부재**: 이론(강의), 실습(CTF/Lab), 서사(동기 부여), 평가를 단일 파이프라인으로 연결하는 플랫폼이 보고되지 않았다.
2. **과정 수준 증적 관리의 부재**: 학습 **결과물**이 아닌 학습 **과정**의 세부 실행 기록을 해시 체인으로 보존·검증하는 시스템이 없다.
3. **실행 기반 교육 오케스트레이션의 부재**: LLM 에이전트가 실제 인프라에서 교육용 태스크를 실행하고 그 결과를 구조적으로 관리하는 통합 프레임워크가 부재하다.

본 논문은 OpsClaw를 오케스트레이터로 활용하는 4계층 파이프라인을 통해 이 세 가지 격차를 동시에 해결한다.

## 2.7 요약

아래 표는 본 장에서 분석한 기존 연구의 한계와 본 논문이 제안하는 해결 방안을 요약한다:

| 기존 연구 유형 | 핵심 한계 | 본 논문의 해결 방안 |
|---------------|-----------|-------------------|
| CTF 플랫폼 [3][4][5] | 이론 맥락 부재, Flag만 기록 | Education+Novel 계층 연결, evidence 전과정 기록 |
| 강의 시스템 [6][7] | 실습 분리, 수동 평가 | System 계층 통합, 자동 evidence 기반 평가 |
| 실습 플랫폼 [12][14] | 격리 환경, 증적 없음 | 실환경 SubAgent, PoW 해시 체인 |
| LLM 튜터 [8][9] | 대화형만, 실행 없음 | execute-plan 기반 실행 오케스트레이션 |
| 블록체인 교육 [22][26] | 결과만 기록 | 과정(process) 전체 블록 기록 |

본 장의 분석을 종합하면, 보안 교육 분야에서 이론·실습·서사·평가의 통합, 학습 과정의 불변 증적 관리, 실행 기반 교육 오케스트레이션이라는 세 가지 핵심 요소가 모두 미해결 과제로 남아 있다. 기존 CTF 플랫폼은 평가에, 강의 시스템은 이론에, 실습 플랫폼은 실행에 각각 특화되어 있으나, 이들을 단일 파이프라인으로 연결하는 통합 접근은 보고되지 않았다. 다음 장에서는 이 세 가지 격차를 동시에 해결하는 4계층 파이프라인의 아키텍처를 제시한다.
