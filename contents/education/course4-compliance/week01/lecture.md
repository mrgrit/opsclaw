| 용어 | 영문 | 설명 | 비유 |
|------|------|------|------|
| **컴플라이언스** | Compliance | 법/규정/표준을 준수하는 것 | 교통법규 준수 |
| **인증** | Certification | 외부 심사 기관이 표준 준수를 확인하는 절차 | 운전면허 시험 합격 |
| **통제 항목** | Control | 보안 목표를 달성하기 위한 구체적 조치 | 건물 소방 설비 하나하나 |
| **SoA** | Statement of Applicability | 적용 가능한 통제 항목 선언서 | "우리 건물에 필요한 소방 설비 목록" |
| **리스크 평가** | Risk Assessment | 위험을 식별·분석·평가하는 과정 | 건물의 화재/지진 위험도 평가 |
| **리스크 처리** | Risk Treatment | 평가된 위험에 대한 대응 결정 (수용/회피/감소/전가) | 보험 가입, 소방 설비 설치 |

| **ISMS** | Information Security Management System | 정보보안 관리 체계 | 회사의 보안 관리 시스템 전체 |
| **ISMS-P** | ISMS + Privacy | 한국의 정보보호 + 개인정보보호 인증 | 한국판 ISO 27001 + 개인정보 |
| **ISO 27001** | ISO/IEC 27001 | 국제 정보보안 관리체계 표준 | 국제 보안 면허증 |
| **ISO 27002** | ISO/IEC 27002 | ISO 27001의 통제 항목 상세 가이드 | 면허 시험 교재 |
| **NIST CSF** | NIST Cybersecurity Framework | 미국 국립표준기술연구소의 사이버보안 프레임워크 | 미국판 보안 가이드 |
| **GDPR** | General Data Protection Regulation | EU 개인정보보호 규정 | EU의 개인정보 보호법 |
| **SOC 2** | Service Organization Control 2 | 클라우드 서비스 보안 인증 (미국) | 클라우드 업체의 보안 성적표 |
| **증적** | Evidence (Audit) | 통제가 실행되었음을 증명하는 자료 | 출석부, 영수증 |
| **심사원** | Auditor | 인증 심사를 수행하는 전문가 | 감독관, 시험관 |
| **부적합** | Non-conformity | 심사에서 표준 미충족 판정 | 시험 불합격 항목 |
| **GAP 분석** | Gap Analysis | 현재 상태와 목표 기준의 차이 분석 | 현재 실력과 합격선의 차이 |

## 1. 정보보안 표준이 왜 필요한가? (30분)

### 1.1 보안 사고와 비용

보안 사고는 기업에 막대한 비용을 초래한다:

| 항목 | 내용 |
|------|------|
| **직접 비용** | 사고 대응, 시스템 복구, 벌금 |
| **간접 비용** | 브랜드 신뢰도 하락, 고객 이탈, 주가 하락 |
| **법적 비용** | 소송, 규제 위반 과징금 |

> IBM 2025 보고서에 따르면, 데이터 유출 사고의 평균 피해액은 **488만 달러(약 65억 원)**이다.

### 1.2 표준이 없다면?

```
표준 없는 조직의 보안:
──────────────────────────────────

개발팀: "보안? 나중에 하죠"         ← 우선순위 낮음
운영팀: "방화벽 설치했으니 됐죠"    ← 단편적 방어
경영진: "해킹 안 당했으니 괜찮죠"   ← 근거 없는 자신감
감사팀: "뭘 점검해야 하죠?"         ← 기준 부재

결과: 체계 없음 → 구멍 발생 → 사고 → 큰 피해
```

### 1.3 표준이 있다면?

```
표준 기반 조직의 보안:
──────────────────────────────────

경영진: "ISO 27001 인증을 받겠습니다"     ← 경영 의지
보안팀: "114개 통제 항목을 점검합니다"     ← 명확한 기준
개발팀: "SDLC에 보안 검토를 포함합니다"   ← 체계적 적용
감사팀: "연 1회 내부 감사를 수행합니다"   ← 지속적 개선

결과: 체계 있음 → 리스크 관리 → 사고 예방/최소화
```

### 1.4 정보보안 표준의 역할

| 역할 | 설명 |
|------|------|
| **기준 제공** | "무엇을 보호해야 하는가"에 대한 체계적 가이드 |
| **공통 언어** | 조직 내/외부에서 보안 수준을 소통하는 공통 프레임워크 |
| **법적 근거** | 규제 준수 여부를 증명하는 기반 |
| **신뢰 구축** | 고객/파트너에게 보안 역량을 입증 |
| **지속적 개선** | PDCA(Plan-Do-Check-Act) 사이클을 통한 보안 수준 향상 |

### 2.1 ISO/IEC 27001

**정식 명칭**: Information Security Management System (ISMS) 요구사항

**개요**:
- 국제표준화기구(ISO)와 국제전기기술위원회(IEC)가 공동 제정
- **정보보안 관리체계(ISMS)**를 수립, 구현, 유지, 개선하기 위한 요구사항
- 전 세계에서 가장 널리 사용되는 정보보안 표준
- 2022년 개정판 (ISO/IEC 27001:2022)이 최신 버전

**구조**:
```
ISO 27001
├── 본문 (4~10절): 관리체계 요구사항
│   ├── 4. 조직의 맥락
│   ├── 5. 리더십
│   ├── 6. 계획
│   ├── 7. 지원
│   ├── 8. 운영
│   ├── 9. 성과 평가
│   └── 10. 개선
│
└── 부속서 A: 통제 항목 (93개, 4개 그룹)
    ├── A.5 조직적 통제 (37개)
    ├── A.6 인적 통제 (8개)
    ├── A.7 물리적 통제 (14개)
    └── A.8 기술적 통제 (34개)
```

**핵심 개념: PDCA 사이클**:
```
    Plan (계획)
       ↓
    Do (실행)
       ↓
    Check (점검)
       ↓
    Act (개선)
       ↓
    Plan (다시 계획) → ... 반복
```

### 2.2 ISMS-P (정보보호 및 개인정보보호 관리체계)

**정식 명칭**: 정보보호 및 개인정보보호 관리체계 인증

**개요**:
- 대한민국 **과학기술정보통신부**와 **개인정보보호위원회** 공동 운영
- ISO 27001을 기반으로 한국 실정에 맞게 수정/보완
- **법적 의무 인증**: 일정 규모 이상의 기업은 반드시 취득해야 함

**의무 인증 대상**:
| 조건 | 대상 |
|------|------|
| ISP (인터넷 서비스 제공자) | KT, SKT, LGU+ 등 |
| IDC (데이터센터) | 네이버, 카카오 등 |
| 매출 100억 이상 + 개인정보 100만 명 이상 | 대형 온라인 서비스 |
| 의료기관 (상급종합병원) | 서울대병원, 삼성병원 등 |

**인증 체계**:
```
ISMS-P 인증 체계
├── 관리체계 수립 및 운영 (16개 항목)
│   ├── 관리체계 기반 마련
│   ├── 위험 관리
│   └── 관리체계 운영
│
├── 보호대책 요구사항 (64개 항목)
│   ├── 정책, 조직, 자산 관리
│   ├── 인적 보안
│   ├── 외부자 보안
│   ├── 물리 보안
│   ├── 인증 및 권한 관리
│   ├── 접근 통제
│   ├── 암호화 적용
│   ├── 정보시스템 도입 및 개발
│   ├── 시스템 및 서비스 운영관리
│   ├── 시스템 및 서비스 보안관리
│   ├── 사고 예방 및 대응
│   └── 재해 복구
│
└── 개인정보 처리 단계별 요구사항 (22개 항목)
    ├── 개인정보 수집 시 보호조치
    ├── 개인정보 보유 및 이용 시 보호조치
    ├── 개인정보 제공 시 보호조치
    ├── 개인정보 파기 시 보호조치
    └── 정보주체 권리 보호
```

### 2.3 NIST Cybersecurity Framework (CSF)

**정식 명칭**: Framework for Improving Critical Infrastructure Cybersecurity

**개요**:
- 미국 국립표준기술연구소(NIST)에서 개발
- 2024년 CSF 2.0 발표 (기존 5개 → 6개 기능)
- 법적 강제력은 없으나, 미국 연방기관에 사실상 필수
- 글로벌 기업들이 자발적으로 채택하는 추세

**6대 핵심 기능 (CSF 2.0)**:

```
┌─────────────────────────────────────────────────┐
│                                                   │
│   GOVERN (통치) ← CSF 2.0에서 새로 추가           │
│      ↓                                            │
│   IDENTIFY (식별) → 자산, 위험 식별               │
│      ↓                                            │
│   PROTECT (보호) → 보호 조치 구현                  │
│      ↓                                            │
│   DETECT (탐지) → 보안 이벤트 탐지                 │
│      ↓                                            │
│   RESPOND (대응) → 사고 대응                       │
│      ↓                                            │
│   RECOVER (복구) → 서비스 복구                     │
│                                                   │
└─────────────────────────────────────────────────┘
```

### 2.4 SOC 2 (Service Organization Control 2)

**개요**:
- 미국공인회계사협회(AICPA)에서 개발
- 클라우드/SaaS 서비스 제공자의 보안 수준을 평가
- **5대 신뢰 서비스 기준(Trust Services Criteria)**:

| 기준 | 설명 |
|------|------|
| **Security** | 시스템이 무단 접근으로부터 보호됨 (필수) |
| **Availability** | 시스템이 약속한 수준으로 가용함 |
| **Processing Integrity** | 시스템 처리가 완전하고 정확함 |
| **Confidentiality** | 기밀 정보가 적절히 보호됨 |
| **Privacy** | 개인정보가 적절히 수집/사용/보관/공개/폐기됨 |

**Type I vs Type II**:
- **Type I**: 특정 시점의 통제 설계 적절성 평가
- **Type II**: 일정 기간(보통 6~12개월) 동안 통제의 운영 효과성 평가

### 2.5 GDPR (General Data Protection Regulation)

**개요**:
- 유럽연합(EU)의 개인정보 보호 규정
- 2018년 5월 시행
- EU 시민의 개인정보를 처리하는 **전 세계 모든 조직**에 적용

**핵심 원칙**:

| 원칙 | 설명 |
|------|------|
| 적법성, 공정성, 투명성 | 합법적이고 공정하게 처리 |
| 목적 제한 | 명시된 목적으로만 처리 |
| 데이터 최소화 | 필요한 최소한의 데이터만 수집 |
| 정확성 | 데이터의 정확성 유지 |
| 보관 제한 | 필요한 기간만 보관 |
| 무결성 및 기밀성 | 적절한 보안 조치 적용 |
| 책임성 | 준수를 입증할 수 있어야 함 |

**위반 시 제재**:
- 최대 **2천만 유로** 또는 **전 세계 연 매출의 4%** 중 큰 금액
- 실제 사례: Meta(12억 유로), Amazon(7.46억 유로) 등

### 2.6 HIPAA (Health Insurance Portability and Accountability Act)

**개요**:
- 미국의 의료 정보 보호법
- 의료 기관, 보험사, 의료 데이터 처리 업체에 적용
- PHI(Protected Health Information) 보호가 핵심

**주요 규칙**:
- **Privacy Rule**: 환자 정보 사용/공개 제한

## 5. 실습 1: 패스워드 정책 점검 (20분)

실제 컴플라이언스 점검에서 가장 기본이 되는 항목들을 실습 인프라에서 확인해보자.

> **ISO 27001 A.5.17**: "인증 정보는 통제되어야 한다"
> **ISMS-P 2.5.1**: "사용자 계정 및 권한 관리"

### 5.1 opsclaw 서버 패스워드 정책 확인

> **실습 목적**: 컴플라이언스 점검의 기본인 패스워드 정책을 실습 인프라에서 직접 확인한다
>
> **배우는 것**: Linux 시스템의 패스워드 만료, 복잡도, 잠금 정책 설정을 확인하고 ISO 27001/ISMS-P 기준과 비교한다
>
> **결과 해석**: PASS_MAX_DAYS가 90 이하이면 적합, 99999이면 만료 정책 미설정으로 부적합이다
>
> **실전 활용**: 인증 심사에서 가장 먼저 확인하는 항목 중 하나가 패스워드 정책의 기술적 구현 증적이다

```bash
# 패스워드 만료 정책 확인
cat /etc/login.defs | grep -E "^PASS_MAX_DAYS|^PASS_MIN_DAYS|^PASS_MIN_LEN|^PASS_WARN_AGE"
```

**예상 출력**:
```
PASS_MAX_DAYS   99999
PASS_MIN_DAYS   0
PASS_MIN_LEN    5
PASS_WARN_AGE   7
```

**분석**:
| 항목 | 현재 값 | 권장 값 | 적합 여부 |
|------|--------|---------|----------|
| PASS_MAX_DAYS (최대 사용일) | 99999 (무제한) | 90 | 부적합 |
| PASS_MIN_DAYS (최소 사용일) | 0 | 1 | 부적합 |
| PASS_MIN_LEN (최소 길이) | 5 | 8 이상 | 부적합 |
| PASS_WARN_AGE (만료 경고일) | 7 | 14 | 개선 필요 |

### 5.2 PAM 패스워드 복잡도 설정 확인

```bash
# PAM 패스워드 품질 설정 확인
cat /etc/pam.d/common-password 2>/dev/null | grep -v "^#" | grep -v "^$"

# 또는 pwquality 설정
cat /etc/security/pwquality.conf 2>/dev/null | grep -v "^#" | grep -v "^$"
```

### 5.3 secu 서버 패스워드 정책 확인

원격 서버에 접속하여 명령을 실행합니다.

```bash
sshpass -p1 ssh -o StrictHostKeyChecking=no secu@10.20.30.1 \
  "cat /etc/login.defs | grep -E '^PASS_MAX_DAYS|^PASS_MIN_DAYS|^PASS_MIN_LEN|^PASS_WARN_AGE'"
```

### 5.4 사용자 계정 상태 점검

```bash
# root 외 UID 0 계정이 있는지 확인 (보안 위반)
awk -F: '$3 == 0 {print $1}' /etc/passwd

# 비밀번호가 없는 계정 확인
sudo awk -F: '($2 == "" || $2 == "!") {print $1}' /etc/shadow 2>/dev/null

# 마지막 로그인 이력
lastlog | head -20
```

## 7. 실습 3: SSH 보안 설정 점검 (20분)

> **ISO 27001 A.8.5**: "보안 인증"
> **ISMS-P 2.6.1**: "접근 통제 정책"

### 7.1 SSH 설정 파일 점검

로그나 설정에서 특정 패턴을 검색합니다.

```bash
# opsclaw SSH 설정 확인
echo "=== opsclaw SSH 설정 ==="
grep -E "^(PermitRootLogin|PasswordAuthentication|Port|MaxAuthTries|Protocol|X11Forwarding|AllowUsers|AllowGroups)" /etc/ssh/sshd_config 2>/dev/null  # 패턴 검색
```

**점검 항목**:

| 설정 | 현재 값 | 권장 값 | 설명 |
|------|--------|---------|------|
| PermitRootLogin | ? | no | root 직접 로그인 차단 |
| PasswordAuthentication | ? | no (키 인증 권장) | 비밀번호 인증 비활성화 |
| Port | ? | 22 외 (예: 2222) | 기본 포트 변경 |
| MaxAuthTries | ? | 3~5 | 인증 시도 횟수 제한 |
| X11Forwarding | ? | no | X11 포워딩 비활성화 |

### 7.2 다른 서버의 SSH 설정도 확인

반복문으로 여러 대상에 대해 일괄 작업을 수행합니다.

```bash
for host in 10.20.30.1 10.20.30.80 10.20.30.100; do    # 반복문 시작
  echo "=== $host SSH 설정 ==="
  sshpass -p1 ssh -o StrictHostKeyChecking=no user@$host \
    "grep -E '^(PermitRootLogin|PasswordAuthentication|Port|MaxAuthTries)' /etc/ssh/sshd_config 2>/dev/null || echo '설정 확인 불가'"
  echo ""
done
```

### 7.3 SSH 접근 로그 확인

```bash
# 최근 SSH 접속 시도 기록
sudo journalctl -u sshd --since "today" 2>/dev/null | tail -10 || \
sudo tail -20 /var/log/auth.log 2>/dev/null | grep sshd | tail -10  # 파일 끝부분 출력
```

## 8. 실습 4: 로그 관리 점검 (20분)

> **ISO 27001 A.8.15**: "로깅"
> **ISMS-P 2.9.4**: "로그 및 접근기록 관리"

### 8.1 시스템 로그 설정 확인

```bash
# rsyslog 서비스 상태
systemctl is-active rsyslog 2>/dev/null || systemctl is-active syslog 2>/dev/null

# 로그 파일 목록
ls -la /var/log/syslog /var/log/auth.log /var/log/kern.log 2>/dev/null
```

### 8.2 로그 로테이션 설정 확인

```bash
# logrotate 설정 확인
cat /etc/logrotate.conf | grep -v "^#" | grep -v "^$" | head -20

# 개별 서비스 로테이션 설정
ls /etc/logrotate.d/
```

**점검 항목**:

| 항목 | 권장 기준 | 확인 방법 |
|------|----------|---------|
| 로그 보존 기간 | 최소 6개월 (법적 요구 시 1년) | logrotate.conf의 `rotate` 값 |
| 로그 압축 | 활성화 | `compress` 옵션 확인 |
| 로그 접근 권한 | 640 이하 | ls -la로 확인 |
| 원격 로그 전송 | SIEM으로 전송 | rsyslog 설정 확인 |

### 8.3 각 서버의 로그 보존 현황

반복문으로 여러 대상에 대해 일괄 작업을 수행합니다.

```bash
for host in "localhost" "10.20.30.1" "10.20.30.80" "10.20.30.100"; do  # 반복문 시작
  echo "=== $host 로그 상태 ==="
  if [ "$host" = "localhost" ]; then
    ls -lh /var/log/syslog /var/log/auth.log 2>/dev/null | awk '{print $5, $9}'
    cat /etc/logrotate.conf 2>/dev/null | grep "^rotate"
  else
    sshpass -p1 ssh -o StrictHostKeyChecking=no user@$host \
      "ls -lh /var/log/syslog /var/log/auth.log 2>/dev/null | awk '{print \$5, \$9}'; cat /etc/logrotate.conf 2>/dev/null | grep '^rotate'" 2>/dev/null
  fi
  echo ""
done
```
