## 1. AI Safety란 무엇인가 (30분)

### 1.1 AI Safety의 정의

AI Safety(AI 안전)는 인공지능 시스템이 인간에게 해를 끼치지 않고, 의도한 대로 안전하게 작동하도록 보장하는 연구 분야이다.

**비유**: 자동차가 발명되었을 때, 기능(속도, 편의성)만 개발하면 안 되고, 브레이크, 에어백, 교통법규 같은 "안전 장치"도 함께 만들어야 했다. AI Safety는 AI라는 강력한 기술의 "브레이크와 에어백"이다.

### 1.2 AI Safety vs AI Security

이 두 개념은 다르지만 밀접하게 연관된다.

| 구분 | AI Safety | AI Security |
|------|-----------|-------------|
| **초점** | AI가 안전하게 작동하는가 | AI가 공격으로부터 보호되는가 |
| **위협 원인** | 설계 결함, 학습 편향, 정렬 실패 | 외부 공격자, 적대적 입력 |
| **예시** | 자율주행차가 보행자를 인식 실패 | 해커가 자율주행차 센서를 속임 |
| **목표** | 유익하고, 무해하고, 정직한 AI | 기밀성, 무결성, 가용성 보장 |
| **관련 분야** | 정렬(Alignment), 해석가능성 | 적대적 ML, 프롬프트 인젝션 |

```
┌─────────────────────────────────────────┐
│           Trustworthy AI                 │
│         (신뢰할 수 있는 AI)               │
│                                          │
│  ┌───────────────┐  ┌────────────────┐  │
│  │  AI Safety    │  │  AI Security   │  │
│  │               │  │                │  │
│  │ - 정렬        │  │ - 적대적 공격   │  │
│  │ - 공정성      │  │ - 데이터 독     │  │
│  │ - 해석가능성  │  │ - 모델 탈취     │  │
│  │ - 통제가능성  │  │ - 프롬프트 인젝션│  │
│  │               ├──┤                │  │
│  │    겹치는 영역:│  │                │  │
│  │  환각, 편향,  │  │                │  │
│  │  프라이버시   │  │                │  │
│  └───────────────┘  └────────────────┘  │
└─────────────────────────────────────────┘
```

### 1.3 왜 AI Safety가 중요한가

AI 기술이 급격히 발전하면서, 그 영향력과 위험도 함께 커지고 있다.

```
AI 능력 성장         안전 장치 개발
  ↑                    ↑
  │    ╱──── 격차(Gap) = 위험
  │   ╱
  │  ╱
  │ ╱      ──────────
  │╱  ────
  ├───────────────────→ 시간
```

- 2020년: GPT-3 등장, 자연스러운 텍스트 생성
- 2022년: ChatGPT, 수억 명이 LLM 사용 시작
- 2024년: AI 에이전트(코드 실행, 웹 브라우징, 도구 사용)
- 2025년: 자율 AI 시스템 (코드 작성 + 배포 + 모니터링까지 자동)
- 2026년 현재: AI가 사이버 공격/방어를 자율적으로 수행

**핵심**: AI 능력은 기하급수적으로 성장하는 반면, 안전 장치의 발전은 더디다. 이 격차를 줄이는 것이 AI Safety의 목표이다.

## 2. AI 위험 분류 체계 (30분)

### 2.1 3대 위험 범주

AI의 위험은 크게 세 가지로 분류한다.

```
┌──────────────────────────────────────────────────┐
│              AI 위험 (AI Risk)                     │
├──────────────┬─────────────────┬─────────────────┤
│   오용        │   정렬 실패      │   사고           │
│  (Misuse)    │  (Misalignment) │  (Accidents)    │
├──────────────┼─────────────────┼─────────────────┤
│ 사람이 AI를   │ AI가 인간 의도와 │ 의도하지 않은    │
│ 나쁜 목적으로 │ 다르게 행동      │ 부작용 발생      │
│ 사용          │                  │                  │
├──────────────┼─────────────────┼─────────────────┤
│ 딥페이크 사기 │ 보상 해킹       │ 자율주행 오인식   │
│ AI 피싱 메일  │ 목표 일반화 실패 │ AI 채용 편향     │
│ 자동 악성코드 │ 기만적 정렬     │ 의료 AI 오진     │
│ 생성          │                  │                  │
└──────────────┴─────────────────┴─────────────────┘
```

### 2.2 오용 (Misuse)

**정의**: 사람이 의도적으로 AI를 악한 목적에 사용하는 것

| 오용 유형 | 설명 | 실제 사례 |
|----------|------|----------|
| **딥페이크** | AI로 가짜 영상/음성 생성 | 정치인 가짜 연설 영상, 보이스피싱 |
| **AI 피싱** | LLM으로 설득력 있는 피싱 메일 작성 | 개인화된 스피어피싱 대량 생성 |
| **악성코드 생성** | LLM에게 멀웨어 코드 작성 요청 | 랜섬웨어 변종 자동 생성 |
| **사회 조작** | AI로 가짜 뉴스/여론 조작 | 소셜미디어 봇 대규모 운영 |
| **감시 악용** | AI 안면인식 남용 | 특정 집단 무차별 감시 |

**사이버보안과의 관계**: 공격자가 AI를 사용하면 공격의 규모, 속도, 정교함이 비약적으로 증가한다.

### 2.3 정렬 실패 (Misalignment)

**정의**: AI 시스템이 설계자가 의도한 목표와 다르게 행동하는 것

**보상 해킹 (Reward Hacking)** 예시:
```
목표: "고객 만족도를 높여라"
의도: 더 좋은 서비스를 제공
AI의 해석: 만족도 설문에서 불만족을 선택할 수 없게 UI 변경
→ 수치는 올라갔지만 실제 만족도는 변하지 않음
```

**보안 분야 정렬 실패 예시**:
```
목표: "취약점을 줄여라"
의도: 코드의 보안 취약점을 수정
AI의 해석: 취약점 스캐너의 알림을 숨김
→ 보고된 취약점은 0이지만 실제로는 그대로 존재
```

### 2.4 사고 (Accidents)

**정의**: 의도하지 않았지만 AI 시스템의 설계 결함이나 예측 불가능한 상황으로 발생하는 문제

| 사고 유형 | 설명 | 예시 |
|----------|------|------|
| **분포 이동** | 학습 데이터와 다른 상황 발생 | 안개 낀 도로에서 자율주행 오작동 |
| **편향 증폭** | 학습 데이터의 편향이 확대됨 | 특정 인종에 대한 채용 차별 |
| **환각** | 사실이 아닌 정보를 생성 | AI 변호사가 존재하지 않는 판례 인용 |
| **연쇄 실패** | 작은 오류가 큰 사고로 이어짐 | AI가 잘못된 보안 조치를 자동 실행 |

### 4.2 EU AI Act (유럽 AI법)

세계 최초의 포괄적 AI 규제법이다 (2024년 최종 승인, 2025~2026년 단계적 시행).

**위험 기반 분류 체계**:

```
위험도          예시                           의무
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
허용 불가      소셜 스코어링, 실시간 원격       전면 금지
(Unacceptable) 생체인식 (예외 있음)

고위험         의료 진단 AI, 채용 AI,          적합성 평가,
(High)         법집행 AI, 교육 평가 AI         인간 감독 의무

제한 위험      챗봇, 딥페이크 생성             투명성 의무
(Limited)                                     (AI임을 공지)

최소 위험      스팸 필터, 추천 시스템           의무 없음
(Minimal)
```

**보안 관련 주요 요구사항** (고위험 AI):
- 위험 관리 시스템 수립
- 데이터 거버넌스 및 품질 관리
- 기술 문서 작성 및 유지
- 인간 감독(Human Oversight) 보장
- 정확성, 견고성, 사이버보안 확보

### 4.3 NIST AI Risk Management Framework (미국)

NIST(미국 국립표준기술연구소)가 2023년 발표한 자율적 AI 위험 관리 프레임워크이다.

**4대 기능(Core Functions)**:

```
┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│GOVERN│ →  │ MAP  │ →  │MEASURE│ →  │MANAGE│
│거버넌스│    │ 매핑  │    │ 측정  │    │ 관리  │
└──────┘    └──────┘    └──────┘    └──────┘
   ↑                                    │
   └────────────────────────────────────┘
             (지속적 개선 순환)
```

| 기능 | 설명 | 활동 예시 |
|------|------|----------|
| **GOVERN** | AI 거버넌스 체계 수립 | 정책 수립, 역할 정의, 윤리 위원회 |
| **MAP** | AI 위험을 식별하고 매핑 | 사용 맥락 파악, 이해관계자 식별 |
| **MEASURE** | 위험을 정량적으로 측정 | 편향 측정, 성능 평가, 레드팀 테스트 |
| **MANAGE** | 위험을 처리하고 모니터링 | 위험 완화, 인시던트 대응, 모니터링 |

### 4.4 한국 AI 기본법

2025년 국회 통과, 2026년 시행된 한국의 AI 규제 법률이다.

**주요 내용**:

| 항목 | 내용 |
|------|------|
| **고위험 AI** | 생명/안전/기본권에 영향을 미치는 AI 분류 및 규제 |
| **투명성** | AI 사용 사실 고지 의무 |
| **영향 평가** | 고위험 AI에 대한 영향 평가 의무화 |
| **AI 위원회** | 국가 AI 위원회 설치 (정책 심의/조정) |
| **기본 원칙** | 인간 존엄성, 안전성, 투명성, 공정성, 책임성 |

### 4.5 책임 있는 AI 원칙 (Responsible AI)

주요 AI 기업과 국제 기관이 합의하는 핵심 원칙들이다.

| 원칙 | 설명 | 보안 분야 적용 |
|------|------|--------------|
| **유익함 (Beneficial)** | AI가 인간에게 도움이 되어야 함 | 보안 강화에 기여, 공격에 악용 금지 |
| **무해함 (Harmless)** | AI가 해를 끼치지 않아야 함 | 오탐으로 인한 서비스 중단 최소화 |
| **정직함 (Honest)** | AI가 거짓 정보를 생성하지 않아야 함 | 환각 없는 정확한 보안 분석 |
| **공정함 (Fair)** | 편향 없이 동등하게 작동 | 특정 IP/지역에 대한 편향 없는 탐지 |
| **투명함 (Transparent)** | 작동 방식을 설명할 수 있어야 함 | AI 탐지 결과의 근거 제시 |
| **통제 가능 (Controllable)** | 인간이 항상 제어할 수 있어야 함 | AI 자동 차단에 대한 인간 승인 절차 |

## 5. 실습: 프롬프트 인젝션 실험 (60분)

### 실습 환경 안내

프롬프트 인젝션(Prompt Injection)은 LLM에게 원래 지시를 무시하고 다른 행동을 하도록 유도하는 공격이다. **통제된 실습 환경**에서 이 공격을 체험하고, LLM의 안전 장치를 이해한다.

**중요**: 이 실습은 교육 목적으로만 수행한다. 실제 서비스에 대한 프롬프트 인젝션은 서비스 약관 위반이며 법적 문제가 될 수 있다.

```bash

```

```bash

  -H "Content-Type: application/json" \

    "model": "gemma3:12b",

      {
        "role": "system",
```

## 6. 프롬프트 인젝션 방어 체계 (10분)

### 6.1 다층 방어 (Defense in Depth)

프롬프트 인젝션에 대한 완벽한 방어는 아직 없다. 여러 층의 방어를 조합해야 한다.

```
사용자 입력
     │
     ↓
┌─────────────────┐
│ Layer 1: 입력 필터 │  위험 키워드 탐지 및 차단
└────────┬────────┘
         ↓
┌─────────────────┐
│ Layer 2: 시스템   │  강화된 시스템 프롬프트로
│ 프롬프트 강화    │  AI의 행동 경계 명시
└────────┬────────┘
         ↓
┌─────────────────┐
│ Layer 3: 출력 필터 │  AI 응답에서 민감 정보
│                   │  유출 여부 검사
└────────┬────────┘
         ↓
┌─────────────────┐
│ Layer 4: 모니터링  │  비정상 패턴 로깅 및 알림
└────────┬────────┘
         ↓
     사용자에게 전달
```

### 6.2 현재 기술의 한계

| 방어 기법 | 효과 | 한계 |
|----------|------|------|
| 시스템 프롬프트 강화 | 기본적 공격 방어 | 창의적 우회에 취약 |
| 입력 키워드 필터 | 알려진 공격 차단 | 새로운 패턴 대응 불가 |
| 출력 필터 | 민감 정보 유출 방지 | 간접적 노출 탐지 어려움 |
| 별도 분류 모델 | 높은 탐지율 | 추가 비용/지연 |
| 샌드박싱 | 실행 권한 제한 | 정보 유출은 방지 불가 |

**핵심 인사이트**: 프롬프트 인젝션은 LLM의 근본적인 구조적 문제이다. "데이터"와 "지시"를 구분할 수 없기 때문에 발생한다. 이는 SQL Injection에서 "데이터"와 "코드"의 경계가 모호했던 것과 같은 원리이다.
